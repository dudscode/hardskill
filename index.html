Segurança — CORS, CSP e CSRF

CORS Cross-Origin Resource Sharing

  • Política Same-Origin (SOP): por padrão o browser bloqueia requisições de uma origem para outra diferente (protocolo+host+porta).
  • CORS relaxa essa política via cabeçalhos HTTP que o servidor envia.
  • Preflight (OPTIONS): para métodos não-simples (PUT, DELETE, cabeçalhos customizados), o browser envia uma request OPTIONS antes da real.
  • Cabeçalhos chave: Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Headers, Access-Control-Allow-Credentials.
  • Wildcard * e credenciais não podem coexistir — ao usar cookies/tokens, a origem deve ser explícita.
  • Erro comum: configurar CORS no frontend — ele só existe no servidor.

Se não está, você precisa bater, perguntar se pode entrar (preflight), e só aí ele diz "pode" ou "não pode". O browser é o guarda que lê essa placa antes de você atravessar.

  5. Explique o comportamento de CORS em Service Workers — eles herdam as políticas da página ou operam de forma independente?

CSP Content Security Policy

  • Defesa contra XSS: informa ao browser quais fontes de conteúdo são confiáveis.
  • Diretivas principais: default-src, script-src, style-src, img-src, connect-src, frame-ancestors.
  • Nonces e hashes permitem scripts inline sem abrir 'unsafe-inline' — cada request gera um nonce único.
  • Report-Only: modo de auditoria — viola mas não bloqueia; envia reports para um endpoint via report-uri / report-to.
  • Strict CSP: usa nonce + 'strict-dynamic' para eliminar listas de hosts (que podem ser bypassadas via CDN comprometido).
  • Entregue via header HTTP ou meta tag (limitado — sem frame-ancestors).

  3. Como um atacante pode bypassar um CSP que permite script-src https://cdn.example.com se o CDN hospedar conteúdo enviado por usuários (ex.: JSONP endpoints, Angular/React CDN)?

  5. Descreva uma estratégia de rollout de CSP em produção sem causar regressão — incluindo fases de auditoria, tratamento de violações legítimas e gestão de nonces em SSR.

CSRF Cross-Site Request Forgery

  • Ataque: força o browser autenticado de uma vítima a enviar requisições involuntárias para outro site.
  • Explora cookies automáticos — o browser os envia em toda request sem que o JS precise tocá-los.
  • Defesas principais: CSRF tokens (synchronizer token pattern), Double Submit Cookie, SameSite cookies.
  • SameSite=Strict/Lax é a mitigação mais moderna — impede o browser de enviar cookies em requisições cross-site.
  • APIs que usam apenas Bearer tokens (Authorization header) não são vulneráveis a CSRF pois o JS precisa setar explicitamente o header.
  • Verificar Origin/Referer como defesa complementar mas não suficiente isolada.

Arquitetura

Clean Architecture

  • Regra da dependência: camadas internas não conhecem camadas externas — dependências só apontam para dentro.
  • Camadas (de dentro pra fora): Entities → Use Cases → Interface Adapters → Frameworks/Drivers.
  • Entities: regras de negócio enterprise — independentes de tudo.
  • Use Cases: orquestram entities; definem Ports (interfaces) que serão implementados por adapters.
  • Testabilidade: core testável em isolamento total sem banco, framework ou HTTP.
  • Inversão de dependência (DIP): o use case define a interface do repositório; o adapter implementa.
🧅 Uma cebola: o miolo (regras do negócio) não sabe que existe a casca de fora (banco de dados, tela, framework). As camadas de fora dependem das de dentro, nunca o contrário. Trocar o banco de dados é como trocar a casca — o coração da cebola não muda nada.

MVC Model-View-Controller

  • Model: dados e regras de negócio; notifica a View quando muda.
  • View: apresentação — renderiza o estado do Model.
  • Controller: recebe input do usuário, manipula Model, seleciona View.
  • Fluxo clássico: User → Controller → Model → View → User.
  • Problemas comuns: Massive View Controller (iOS), lógica de negócio vazando para Controller.
  • Variações: MVP (passive view), MVVM (data binding), MVT (Django), VIPER (iOS).
🍽️ Num restaurante: o garçom (Controller) recebe o pedido do cliente, fala com a cozinha (Model) que prepara o prato, e o prato vai para a mesa (View) para o cliente ver e comer. O cliente não vai até a cozinha, e a cozinha não precisa saber como a mesa está decorada.

MVVM Model-View-ViewModel

  • ViewModel: expõe estado observável e comandos; não referencia a View.
  • Data Binding: a View se sincroniza automaticamente com o ViewModel — bidirecional ou unidirecional.
  • Separação de responsabilidades: lógica de apresentação fica no ViewModel, testável sem UI.
  • Popularizado pelo WPF/Silverlight, amplamente adotado em iOS (Combine), Android (Jetpack), Angular, SwiftUI.
  • Reatividade: ViewModel usa Observables/Streams; View "reage" a mudanças sem polling.

SOLID

  • S — Single Responsibility: uma classe tem apenas um motivo para mudar.
  • O — Open/Closed: aberto para extensão, fechado para modificação.
  • L — Liskov Substitution: subtipos devem poder substituir seus tipos base sem quebrar o programa.
  • I — Interface Segregation: clientes não devem depender de métodos que não usam — interfaces menores e coesas.
  • D — Dependency Inversion: dependa de abstrações, não de implementações concretas.
  • Objetivo: código manutenível, extensível e testável.
🔧 S: Um canivete suíço é útil, mas um chef prefere facas especializadas. O: Adicione uma nova lâmina sem destruir as outras. L: Qualquer pássaro que herda de "Pássaro" deve voar — não coloque Avestruz como pássaro voador se ela não voa. I: Não obrigue o Avestruz a implementar voar(). D: A tomada não depende da marca do carregador — ela usa um padrão (interface).

Padrões Criacionais GoF — Instanciação de Objetos

Padrões criacionais abstraem o processo de instanciação, tornando o sistema independente de como objetos são criados, compostos e representados. O objetivo é reduzir o acoplamento entre o código cliente e as classes concretas que ele precisa instanciar.

  • Singleton — instância única por processo; acesso global controlado. Custo: dificulta testes, mascara acoplamento implícito, não funciona como "global" em multi-processo.
  • Factory Method — a superclasse define a interface de criação; subclasses decidem qual classe concreta instanciar. Desacopla criador de produto sem romper a hierarquia.
  • Abstract Factory — escala Factory Method para famílias de produtos relacionados. Garante compatibilidade entre os produtos criados (ex: todos do mesmo tema visual).
  • Builder — constrói objetos complexos passo a passo via uma interface fluente ou por um Diretor. Elimina "construtores telescópicos" com dezenas de parâmetros opcionais.
  • Prototype — cria novos objetos clonando um existente. O próprio objeto é responsável pela cópia, respeitando campos privados. Alternativa à subclassificação só para configurações diferentes.

Guia de escolha rápido: 1 instância → Singleton · subclasse decide o tipo → Factory Method · família de produtos compatíveis → Abstract Factory · muitos parâmetros opcionais → Builder · clonar configurações existentes → Prototype.

  • Intenção: Garantir que uma classe tenha exatamente uma instância e prover um ponto de acesso global a ela.
  • Problema: Recursos compartilhados que não podem ter múltiplas instâncias (pool de conexões, logger, configuração global, registro de serviços) precisam de controle centralizado de instanciação — o construtor público não é suficiente.
  • Solução / Estrutura:
    ① Construtor privado — bloqueia new externo.
    ② Campo estático privado instance armazena a instância.
    ③ Método estático público getInstance(): cria na primeira chamada (lazy init) e retorna sempre a mesma instância.
  • Thread Safety (implementações):
      • Synchronized method — seguro, mas gargalo em alta concorrência.
      • Double-Checked Locking + volatile — lazy + performático (Java/C++).
      • Initialization-on-demand holder — usa garantia do class loader; sem lock explícito (Java).
      • Enum Singleton — proof against serialization e reflection attacks (Java).
      • Módulo Python — por design, módulos são singletons no mesmo processo.
  • Quando usar: logger centralizado, pool de conexões, objeto de configuração, event bus, cache em memória compartilhado.
  • Prós: instância única garantida; lazy initialization; controle sobre variáveis globais.
  • Contras: viola SRP (gerencia instância + lógica própria); dificulta testes unitários — estado persiste entre testes, exige reset manual ou injeção de dependência; em serverless/workers, cada processo tem seu próprio heap → a "unicidade" é por processo, não por sistema; cria dependências implícitas invisíveis nas assinaturas dos métodos.
  • Relaciona-se com: Facade (frequentemente implementada como Singleton); Abstract Factory, Builder, Prototype podem ser Singletons.
  • Intenção: Definir uma interface para criar um objeto em uma superclasse, mas deixar subclasses decidirem qual classe concreta instanciar.
  • Problema: Um framework ou biblioteca precisa criar objetos mas não pode (ou não deve) conhecer os tipos concretos — eles serão definidos pelo usuário. Usar new ProdutoConcreto() cria acoplamento rígido entre criador e produto.
  • Solução / Estrutura:
      • Produto — interface comum a todos os objetos produzidos.
      • Produtos Concretos — implementações específicas da interface.
      • Criador — declara o factory method abstrato ou com impl. padrão; opera sobre o produto via interface; a criação não é a responsabilidade principal do Criador.
      • Criadores Concretos — sobrescrevem o factory method para retornar o produto adequado.
  • Factory Method vs. Simple Factory: Simple Factory é apenas um método estático com if/switch — não é padrão GoF, viola OCP (novo tipo = modificar o método). Factory Method usa herança: novo tipo = nova subclasse do Criador, sem modificar o código existente.
  • Quando usar: tipo exato do objeto a criar é desconhecido em design time; permitir que usuários de framework estendam componentes internos; economizar recursos reutilizando objetos existentes em um pool.
  • Prós: desacopla Criador de Produto concreto (SRP); adicionar novos produtos sem modificar o Criador (OCP); código extensível por herança.
  • Contras: proliferação de subclasses para cada novo tipo de produto; pode ser excessivo onde um simples mapa de registro (Map<String, Supplier>) resolveria.
  • Relaciona-se com: Abstract Factory (usa Factory Methods internamente), Template Method (factory method pode ser uma etapa), Prototype (alternativa sem herança quando clonar basta).
  • Intenção: Produzir famílias de objetos relacionados ou dependentes sem especificar suas classes concretas.
  • Problema: O código precisa criar múltiplos objetos relacionados que devem ser compatíveis entre si (UI cross-platform, temas, drivers de banco). Instanciar produtos concretos diretamente impede trocar a família inteira com uma única decisão.
  • Solução / Estrutura:
      • Produtos Abstratos — interface para cada tipo de produto distinto (Button, Checkbox).
      • Produtos Concretos — implementações agrupadas por família (WinButton, MacButton).
      • Fábrica Abstrata — interface com um factory method por tipo de produto.
      • Fábricas Concretas — uma por família; garantem compatibilidade interna.
      • Cliente — recebe a fábrica por injeção de dependência; opera apenas sobre interfaces abstratas.
  • Abstract Factory vs. Factory Method: FM usa herança (um Criador por produto); AF usa composição (uma fábrica por família de produtos). Use FM para 1 tipo de produto variável; AF quando vários tipos precisam variar juntos e de forma consistente.
  • Quando usar: código independente de como produtos são criados/compostos; garantia de compatibilidade entre produtos da mesma família; troca de família de produtos em runtime via configuração.
  • Prós: compatibilidade garantida entre produtos da mesma família; SRP e OCP para novas famílias; cliente desacoplado de implementações concretas.
  • Contras: adicionar um novo tipo de produto (não uma família) exige modificar a interface abstrata e todas as fábricas concretas — viola OCP nessa dimensão.
  • Relaciona-se com: Factory Method (AF implementada com FMs), Singleton (fábricas são frequentemente singletons), Bridge (AF pode criar implementações para Bridge), Prototype (alternativa — registro de protótipos por família).
  • Intenção: Separar a construção de um objeto complexo de sua representação, permitindo que o mesmo processo de construção crie representações diferentes.
  • Problema: Dois cenários clássicos:
    ① Construtor telescópico — new Casa(2, 3, true, false, null, "vermelho", 120, ...): ilegível, frágil, qualquer novo campo quebra todos os call sites.
    ② Explosão de subclasses — criar CasaComPiscina, CasaComGaragem, CasaComPiscinaEGaragem... combinações crescem exponencialmente.
  • Solução / Estrutura:
      • Builder — interface com um método por etapa de construção.
      • Builders Concretos — implementam as etapas produzindo representações diferentes; têm método getResult().
      • Diretor (opcional) — encapsula sequências de etapas reutilizáveis; desacopla o cliente da ordem de construção.
      • Produto — o objeto resultante; Builders diferentes podem produzir tipos sem interface comum.
  • Builder Fluente: cada método de etapa retorna this, permitindo encadeamento: builder.setParedes(2).setTeto("plano").addJanelas(4).build(). Muito usado em ORMs (QueryBuilder), HTTP clients, SDKs e DSLs. O método build() pode validar invariantes antes de instanciar.
  • Quando usar: construtores com ≥ 4 parâmetros, especialmente opcionais; múltiplas representações do mesmo objeto com processo de construção comum; construção recursiva/incremental de estruturas (árvores Composite).
  • Prós: construção passo a passo e configurável; reutilização do código de construção via Diretor; SRP — lógica de construção isolada; permite criar objetos imutáveis (build() valida antes de criar).
  • Contras: complexidade desnecessária para objetos simples; requer coordenação entre o cliente e a sequência correta de chamadas (mitigada pelo Diretor).
  • Relaciona-se com: Composite (Builder excelente para construir árvores recursivas), Bridge (Diretor como abstração, Builders como implementações).
  • Intenção: Criar novos objetos copiando protótipos existentes, sem fazer o código depender das classes concretas dos objetos copiados.
  • Problema: Três obstáculos ao copiar diretamente:
    ① Campos privados são inacessíveis externamente.
    ② Requer conhecer a classe concreta do objeto → acoplamento.
    ③ Frequentemente você conhece apenas a interface, não a implementação.
  • Solução / Estrutura:
      • Interface Protótipo — declara método clone().
      • Protótipos Concretos — implementam clone() internamente (acesso a campos privados via construtor de cópia: new ConcretePrototype(this)).
      • Registro de Protótipos (opcional) — catálogo de objetos pré-configurados com nomes/chaves; o cliente solicita uma cópia pelo nome sem conhecer a classe concreta.
  • Deep vs. Shallow Copy: cópia rasa copia referências (objetos aninhados são compartilhados — mutações afetam o original). Cópia profunda clone recursivamente os objetos referenciados — necessária quando o protótipo contém coleções ou objetos mutáveis. Referências circulares exigem tratamento especial (mapa de objetos já copiados).
  • Quando usar: independência de classes concretas é necessária; criar um objeto do zero é mais custoso que clonar um existente (objetos com estado calculado, resultados de I/O); reduzir subclasses criadas apenas para configurações diferentes.
  • Prós: independência de classes concretas; elimina código repetitivo de inicialização; alternativa eficiente à subclassificação para configurações predefinidas.
  • Contras: deep copy de objetos com referências circulares é complexo e propenso a erros sutis.
  • Relaciona-se com: Abstract Factory (pode usar Prototype para criar produtos), Command (Prototype para salvar cópias no histórico de undo/redo), Composite + Decorator (Prototype facilita clonar estruturas compostas complexas).
  1. Singleton em ambiente serverless (AWS Lambda, Vercel Functions): por que a garantia de "instância única" não se aplica entre invocações? Como você projeta um pool de conexões de banco de dados nesse modelo sem desperdício nem race conditions?
  2. Factory Method vs. Simple Factory: construa um exemplo de sistema de pagamentos onde a diferença entre os dois é arquiteturalmente significativa ao adicionar um novo método de pagamento — explique como cada abordagem escala e qual viola OCP.
  3. Abstract Factory: ao adicionar um novo tipo de produto (ex: Tooltip) a uma AF já existente com 10 fábricas concretas, qual é o custo de mudança? Como o padrão Prototype com Registro pode ser uma alternativa mais extensível nesse caso?
  4. Como você implementa validação de invariantes de negócio no Builder Fluente — por exemplo, garantir que dataFim > dataInicio ou que campos obrigatórios estejam preenchidos — sem lançar exceções no meio da cadeia de chamadas?
  5. Qual é a diferença entre cópia rasa e cópia profunda no contexto de Prototype? Descreva um bug sutil que ocorre quando um objeto clonado via shallow copy compartilha uma coleção mutável com o original — e como detectá-lo em testes.

Padrões Estruturais GoF — Composição de Classes e Objetos

Padrões estruturais tratam da composição de classes e objetos para formar estruturas maiores. Eles usam herança e composição para montar interfaces e implementações flexíveis.

  • Adapter — converte a interface de uma classe em outra interface esperada pelo cliente. Permite colaboração entre classes incompatíveis sem modificar nenhuma das duas.
  • Bridge — separa uma abstração de sua implementação em hierarquias independentes. Preferível à herança quando tanto a abstração quanto a implementação precisam variar. (GoF — menos comum em entrevistas)
  • Composite — trata objetos individuais e composições uniformemente via interface comum. Ideal para estruturas em árvore (sistemas de arquivos, UI hierárquica, documentos).
  • Decorator — adiciona comportamentos a objetos envolvendo-os em wrappers com a mesma interface. Alternativa à herança para extensão dinâmica em runtime.
  • Facade — interface unificada e simplificada para um subsistema complexo. Reduz acoplamento entre cliente e subsistema; não elimina o subsistema.
  • Flyweight — compartilha estado intrínseco (imutável) entre muitos objetos para economizar memória. (GoF — foco em otimização de memória)
  • Proxy — substituto que controla o acesso ao objeto real. Viabiliza lazy init, cache, controle de acesso, logging, execução remota sem modificar o objeto original.

Confusões frequentes: Adapter muda a interface · Decorator mantém a interface e adiciona comportamento · Proxy mantém a interface e controla acesso · Facade cria uma nova interface simplificada para um subsistema.

  • Intenção: Converter a interface de uma classe em outra interface esperada pelo cliente. Permite que classes com interfaces incompatíveis colaborem.
  • Problema: Você quer usar uma biblioteca ou classe legada, mas sua interface é incompatível com o que o restante do sistema espera. Modificar qualquer uma das partes não é viável (código de terceiro, múltiplos consumidores).
  • Solução / Estrutura (duas variantes):
      • Adaptador de Objeto (composição): o Adapter implementa a interface do Cliente e mantém uma referência ao Serviço incompatível. As chamadas ao Adapter são traduzidas e delegadas ao Serviço. Funciona com subclasses do Serviço; flexível.
      • Adaptador de Classe (herança múltipla): o Adapter herda de ambas as classes simultaneamente. Não precisa de wrapping — mais direto, mas requer herança múltipla (disponível em C++, não em Java/C#).
  • Two-Way Adapter: implementa as interfaces de ambas as partes simultaneamente — permite que qualquer uma use o Adapter, tornando-as intercambiáveis sem mudanças.
  • Quando usar: integrar código legado ou de terceiros com interface diferente da esperada; reutilizar subclasses sem duplicar código de conversão em cada uma.
  • Prós: SRP — conversão separada da lógica de negócio; OCP — novos adapters sem modificar cliente ou serviço.
  • Contras: complexidade extra; se você controla ambas as interfaces, adaptá-las diretamente é mais limpo.
  • Vs. outros padrões:
      • Facade — define uma nova interface simplificada para todo um subsistema.
      • Decorator — mantém a interface original e adiciona comportamento.
      • Proxy — mantém a interface original e controla acesso.
  • Intenção: Compor objetos em estruturas de árvore para representar hierarquias parte-todo. Permite tratar objetos individuais e composições de forma uniforme.
  • Problema: Um sistema de encomendas tem caixas que contêm produtos e outras caixas. Calcular o preço total exige lógica condicional para distinguir caixas de produtos — e vai ficando mais complexo com mais níveis. O cliente não deveria precisar saber se está lidando com um produto simples ou uma cesta inteira.
  • Solução / Estrutura:
      • Componente — interface comum com operações para folhas e contêineres (ex: getPreco(), render()).
      • Folha — elemento sem filhos; executa a operação diretamente.
      • Composite (Contêiner) — armazena uma coleção de Componentes (folhas ou outros Composites); delega a operação recursivamente e agrega resultados.
      • Cliente — opera exclusivamente via interface Componente; ignora se está falando com folha ou contêiner.
  • Decisão de design: a interface Componente deve incluir métodos de gerenciamento de filhos (add/remove/getChildren) ou apenas operações de negócio? Incluir na interface viola o princípio de segregação de interface para folhas; excluir exige downcasting. Escolha baseada no contexto — transparência vs. segurança de tipos.
  • Quando usar: hierarquias parte-todo (árvores); código que deve tratar elementos simples e compostos uniformemente; sistemas de arquivos, DOMs, UIs hierárquicas, ASTs, menus aninhados.
  • Prós: polimorfismo e recursão simplificam trabalho com árvores; OCP para novos tipos de nós.
  • Contras: interface comum pode ser muito genérica quando folhas e contêineres têm capacidades muito distintas; dificultar restrições de tipos (ex: proibir certos tipos como filhos).
  • Relaciona-se com: Builder (ideal para construir árvores Composite), Iterator/Visitor (para percorrer a estrutura), Decorator (similar estruturalmente, mas adiciona responsabilidades em vez de agregar).
  • Intenção: Adicionar responsabilidades a objetos dinamicamente envolvendo-os em objetos decoradores com a mesma interface. Alternativa flexível à subclassificação.
  • Problema: Uma classe de notificação precisa suportar combinações de canais (email + SMS + Slack). Criar uma subclasse por combinação gera explosão exponencial. Herança é estática — não permite trocar ou combinar comportamentos em runtime.
  • Solução / Estrutura (5 participantes):
      • Componente — interface comum (ex: Notifier).
      • Componente Concreto — implementação base (ex: EmailNotifier).
      • Decorador Base — implementa a interface e mantém referência ao Componente envolvido; delega todas as chamadas.
      • Decoradores Concretos — sobrescrevem métodos, executando comportamento extra antes/após delegar ao componente envolvido.
      • Cliente — responsável por montar a pilha de decoradores.
  • Empilhamento: new SlackDecorator(new SMSDecorator(new EmailNotifier())) — cada camada delega para a próxima e adiciona comportamento. A ordem importa — verifique se há dependências entre decoradores.
  • Quando usar: adicionar responsabilidades em runtime sem subclassificar; extensão impossível ou impraticável via herança (classes final); lógica estruturada em camadas opcionais e combináveis (compressão + criptografia + log em streams).
  • Prós: extensão sem herança; add/remove em runtime; combinações infinitas de comportamentos; SRP — cada decorador faz uma coisa.
  • Contras: difícil remover um decorador específico do meio da pilha; comportamento pode depender da ordem de empilhamento; muitos pequenos objetos podem dificultar debug.
  • Vs. outros padrões:
      • Proxy — mesma estrutura, mas controla acesso em vez de adicionar comportamento; ciclo de vida do serviço geralmente gerenciado pelo Proxy.
      • Composite — agrega filhos e agrega resultados; Decorator envolve um único componente e transforma o resultado.
  • Intenção: Fornecer uma interface simplificada para um subsistema complexo — uma "porta de entrada" que expõe apenas o que o cliente precisa.
  • Problema: O código cliente precisa de apenas uma fração das funcionalidades de um subsistema complexo (biblioteca de vídeo, framework de ORM, sistema de autenticação). Trabalhar diretamente com todas as classes do subsistema cria acoplamento forte e dificulta substituição futura.
  • Solução / Estrutura:
      • Facade — conhece o subsistema; redireciona chamadas do cliente para os objetos corretos; orquestra o fluxo sem expor detalhes.
      • Fachadas Adicionais — se a Facade crescer demais, extrai-se comportamento em fachadas secundárias especializadas.
      • Subsistema — conjunto de classes que realizam o trabalho real; não conhecem a Facade.
      • Cliente — se comunica exclusivamente com a Facade.
  • Atenção — Facade ≠ elimina o subsistema: a Facade é uma camada de conveniência; clientes avançados ainda podem acessar o subsistema diretamente se precisarem de controle fino. Isso a diferencia de uma API REST que substitui completamente o subsistema.
  • Quando usar: interface simples para subsistema complexo; estruturar o sistema em camadas com pontos de entrada bem definidos (Facade entre camadas); reduzir acoplamento entre subsistemas que precisam colaborar.
  • Prós: isola o cliente da complexidade interna; facilita substituição do subsistema sem afetar clientes; ponto único de integração.
  • Contras: risco de se tornar "God Object" acoplado a muitas classes; pode esconder problemas de design no subsistema em vez de corrigi-los.
  • Vs. outros padrões:
      • Adapter — converte interface existente; Facade cria uma nova.
      • Mediator — também centraliza comunicação, mas entre objetos que se conhecem mutuamente (coordenação bidirecional); Facade é unidirecional (cliente → subsistema).
      • Proxy — mesma interface do serviço; Facade cria interface nova simplificada.
  • Intenção: Fornecer um substituto ou representante de outro objeto, controlando o acesso a ele.
  • Problema: Você precisa adicionar comportamento ao acesso de um objeto (lazy init, cache, controle de acesso, logging, execução remota) sem modificar sua classe — seja porque é código de terceiros, seja para não violar SRP.
  • Solução / Estrutura:
      • Interface do Serviço — interface que tanto o Serviço Real quanto o Proxy implementam (transparência para o cliente).
      • Serviço Real — contém a lógica de negócio útil.
      • Proxy — referência ao Serviço Real; processa antes/depois e delega.
  • Tipos de Proxy:
      • Virtual (Lazy) — adia criação de objetos pesados até a primeira necessidade real.
      • Protection — controla acesso baseado em permissões/papéis.
      • Remote — representa objeto em outro processo/máquina (ex: stub RPC/gRPC).
      • Caching — armazena resultados de operações custosas; retorna cópia em cache nas demais chamadas.
      • Logging — registra chamadas sem alterar o serviço.
      • Smart Reference — conta referências ativas; libera recurso quando zerado.
  • Quando usar: lazy initialization de objetos pesados; controle de acesso sem modificar o serviço; cache transparente; logging/auditoria; comunicação com objetos remotos.
  • Prós: controle transparente (cliente não sabe que há proxy); gerenciamento de ciclo de vida do serviço; OCP — comportamento extra sem modificar o serviço.
  • Contras: latência adicional; complexidade extra com novas classes; pode dar falsa sensação de que o serviço é barato quando na verdade é caro.
  • Proxy vs. Decorator: estrutura idêntica. Diferença semântica: Proxy gerencia o ciclo de vida do serviço real e controla quando/se ele é acessado; Decorator adiciona comportamentos ao que já é acessado, com o cliente controlando a composição.
  1. Adapter, Decorator e Proxy têm estruturas de implementação muito semelhantes — todos envolvem um objeto e delegam chamadas. Como você diferencia os três em termos de intenção, ciclo de vida do objeto envolvido e quem controla a composição? Dê um exemplo onde trocar um pelo outro quebraria o design.
  2. Você está projetando um sistema de autorização usando Proxy de proteção. Como você garante que o Proxy intercepta todas as formas de acesso ao serviço — incluindo chamadas internas da mesma classe — sem que o código de segurança vaze para o serviço real?
  3. No padrão Composite, discuta a tensão entre transparência (incluir métodos de gerenciamento de filhos na interface Componente) e segurança de tipos (deixar esses métodos apenas no Composite). Qual abordagem você escolheria para um sistema de renderização de UI e por quê?
  4. Em que cenário Facade pode se tornar um anti-padrão? Descreva como uma Facade bem-intencionada pode mascarar problemas de coesão no subsistema e dificultar testes de integração.
  5. Como o padrão Decorator se comporta quando dois decoradores têm dependências entre si (ex: o decorador de criptografia deve vir antes do de compressão)? Como você codifica e documenta essa restrição de ordem sem criar acoplamento entre os decoradores?

Padrões Comportamentais GoF — Comunicação e Responsabilidade

Padrões comportamentais tratam de algoritmos e distribuição de responsabilidades entre objetos. Focam em como objetos se comunicam e colaboram sem criar acoplamento excessivo.

  • Observer — mecanismo de assinatura pub/sub. Publicador notifica assinantes sobre eventos sem conhecê-los diretamente. Base de event buses, reactive programming e data binding.
  • Strategy — família de algoritmos encapsulados em classes intercambiáveis. O contexto delega o algoritmo para a estratégia injetada. Substitui condicionais volumosos por polimorfismo.
  • Command — encapsula um pedido como objeto com tudo necessário para executá-lo. Viabiliza: parametrização, enfileiramento, logging, undo/redo e transações.
  • Chain of Responsibility — passa um request por uma cadeia de handlers até que um o processe. Desacopla remetente de receptores; permite composição dinâmica da cadeia.
  • Template Method — define o esqueleto de um algoritmo na superclasse; subclasses preenchem etapas específicas sem alterar a estrutura geral. Usa herança.
  • State — permite que um objeto altere comportamento quando seu estado muda. Parece que o objeto muda de classe. Substitui máquinas de estado baseadas em switch.
  • Iterator — percorre elementos de uma coleção sem expor sua representação interna. Separa o algoritmo de travessia da estrutura de dados. (amplamente nativo em linguagens modernas)
  • Mediator — centraliza comunicações entre objetos num objeto mediador. Reduz dependências caóticas entre componentes, mas concentra lógica. (relação com Event Bus e CQRS)
  • Memento — captura e externaliza o estado interno de um objeto sem violar encapsulamento. Permite restaurar estados anteriores — base de undo/redo.
  • Visitor — separa algoritmos dos objetos sobre os quais operam. Adiciona novas operações sem modificar as classes dos elementos. Implementa double dispatch.
  • Intenção: Definir um mecanismo de assinatura para notificar múltiplos objetos sobre eventos que ocorrem no objeto observado.
  • Problema: Um objeto precisa notificar outros quando muda de estado, mas não deve conhecer quem são esses outros (nem quantos são). Polling é ineficiente; notificação direta cria acoplamento. O conjunto de dependentes é dinâmico.
  • Solução / Estrutura:
      • Publicador (Subject) — mantém lista de assinantes; expõe subscribe(), unsubscribe() e notify(); notifica ao mudar de estado.
      • Interface Assinante — declara o método de atualização (update(event)).
      • Assinantes Concretos — implementam a reação; podem se inscrever/desinscrever dinamicamente.
      • Cliente — instancia publicadores e assinantes; registra os assinantes.
  • Push vs. Pull:
      • Push — o publicador envia os dados no evento. Assinante não precisa consultar o sujeito; pode receber mais do que precisa.
      • Pull — o publicador notifica sem dados; assinante consulta o sujeito quando necessário. Assinante decide o que ler; mais flexível mas adiciona acoplamento ao sujeito.
  • Observer em sistemas modernos: o padrão é a base de Reactive Streams (RxJS, Project Reactor), Event Bus, data binding em frameworks (Vue reactivity, MobX), DOM events e WebSockets. Em arquiteturas distribuídas, equivale ao pub/sub de brokers (Kafka, RabbitMQ) — com garantias adicionais de durabilidade.
  • Quando usar: mudança em um objeto requer atualização de outros cujo número é desconhecido; observação temporária ou específica a contexto; desacoplamento entre publicador e assinantes.
  • Prós: OCP — novos assinantes sem modificar publicador; relações estabelecidas em runtime; suporte a broadcast.
  • Contras: ordem de notificação é indefinida por padrão; assinantes não removidos criam memory leaks; cascatas de notificação podem ser difíceis de depurar.
  • Relaciona-se com: Mediator (alternativa quando a lógica de coordenação é complexa), Command (comandos como eventos), Event Sourcing (eventos persistidos, não apenas notificados).
  • Intenção: Definir uma família de algoritmos, encapsular cada um em uma classe e torná-los intercambiáveis. O cliente pode trocar o algoritmo independentemente dos clientes que o usam.
  • Problema: Uma classe começa com um algoritmo e, ao longo do tempo, acumula variantes via condicionais (if tipoRota == "carro"... else if "bicicleta"...). Cada nova variante infla a classe, cria conflitos de merge e dificulta testes isolados.
  • Solução / Estrutura:
      • Contexto — mantém referência a uma Estratégia via interface; delega a execução do algoritmo; expõe setter para troca em runtime.
      • Interface Strategy — declara o método de execução do algoritmo.
      • Estratégias Concretas — implementam variantes do algoritmo.
      • Cliente — instancia a estratégia e a injeta no contexto.
  • Strategy vs. condicionais: quando há 2–3 variantes simples e estáticas, condicionais são mais legíveis. Strategy compensa quando: as variantes são muitas; variam independentemente do contexto; precisam ser injetadas externamente (teste, configuração); ou serão adicionadas por terceiros sem modificar o código existente.
  • Strategy em linguagens funcionais: em JS/Python/Go, passar funções diretamente (funções de primeira classe) serve o mesmo propósito sem a cerimônia das classes. Strategy com classes brilha em OO estrito onde comportamento + estado são necessários.
  • Quando usar: trocar algoritmos em runtime; múltiplas classes que diferem apenas em comportamento; isolar lógica de negócio de detalhes de algoritmo; eliminar condicionais de seleção de algoritmo.
  • Prós: troca em runtime; SRP — cada estratégia em sua classe; OCP — novas estratégias sem modificar o contexto; substitui herança por composição.
  • Contras: cliente precisa conhecer as diferenças entre estratégias para escolher; overhead de classes para algoritmos simples.
  • Vs. Template Method: Strategy usa composição — o algoritmo inteiro está na estratégia, trocável em runtime. Template Method usa herança — o esqueleto está na superclasse, partes nas subclasses, estático após compilação.
  • Intenção: Encapsular um pedido como objeto independente contendo toda a informação necessária para executá-lo. Permite parametrizar métodos com pedidos, enfileirar/logar pedidos e implementar operações reversíveis.
  • Problema: Múltiplos pontos de entrada na UI (botão, menu, atalho, API) precisam disparar a mesma operação — duplicação de código e acoplamento direto entre camada de UI e lógica de negócio. Adicionar undo/redo ou enfileiramento de operações fica ainda mais complexo.
  • Solução / Estrutura (5 participantes):
      • Remetente (Invoker) — armazena referência ao comando; o aciona (não conhece o destinatário).
      • Interface Comando — declara método único execute(); opcionalmente undo().
      • Comandos Concretos — implementam operações específicas; armazenam referência ao Destinatário + parâmetros.
      • Destinatário (Receiver) — contém a lógica de negócio real; não conhece o Remetente.
      • Cliente — cria Comandos Concretos com seus Destinatários e os associa a Remetentes.
  • Undo/Redo: armazene os comandos executados em uma pilha (histórico). Para desfazer, chame undo() no topo da pilha. Para refazer, mantenha uma pilha de desfeitos. Comandos devem armazenar o estado anterior do destinatário (ou usar Memento para capturá-lo).
  • Macro Commands: um comando que contém uma lista de outros comandos. execute() executa todos em sequência. Permite transações compostas e operações em lote.
  • Quando usar: parametrizar objetos com operações; enfileirar, agendar ou executar operações remotamente; implementar undo/redo; transações com rollback; sistemas de plugin onde operações são injetadas.
  • Prós: SRP — desacopla remetente de destinatário; OCP — novos comandos sem modificar remetentes; undo/redo, enfileiramento, logging nativos; composição em macro commands.
  • Contras: camada extra entre UI e lógica; proliferação de classes de comando.
  • Relaciona-se com: Memento (captura estado para undo), Chain of Responsibility (handlers como comandos), Prototype (clona comandos para histórico), Strategy (ambos parametrizam objetos — Command converte operações em objetos; Strategy troca algoritmos).
  • Intenção: Passar um pedido por uma cadeia de handlers. Cada handler decide processar o pedido ou passá-lo adiante. Desacopla remetente de receptores.
  • Problema: Múltiplas verificações/transformações sequenciais (autenticação → autorização → validação → rate limiting → cache → handler) concentradas em um único ponto criam um monólito frágil — difícil de reordenar, reutilizar ou testar isoladamente.
  • Solução / Estrutura:
      • Interface Handler — declara método para processar pedido e setter para o próximo handler.
      • Handler Base (opcional) — abstrato; armazena referência ao próximo; implementa método padrão que delega se não processar.
      • Handlers Concretos — implementam lógica específica; podem: processar e parar, processar e continuar, ou apenas passar adiante.
      • Cliente — monta a cadeia e envia o pedido ao primeiro handler (ou a qualquer handler).
  • Variantes de comportamento:
      • Exclusivo — apenas um handler processa (clássico GoF); os demais passam adiante.
      • Inclusivo — todos os handlers processam em sequência (pipeline/middleware). Variante comum em frameworks web (Express middleware, ASP.NET pipeline, AWS Lambda layers).
  • CoR em frameworks: é o padrão por trás de middlewares (Express/Koa), interceptors (Spring), filtros de servlet, pipeline de validação, e tratadores de eventos DOM (bubbling/capturing).
  • Quando usar: múltiplos objetos podem tratar um pedido e o tratador é determinado em runtime; a cadeia de handlers deve ser configurável dinamicamente; evitar acoplamento direto entre remetente e receptor.
  • Prós: controle da ordem de tratamento; desacouplamento; OCP — novos handlers sem modificar a cadeia existente; handlers podem ser compartilhados entre cadeias.
  • Contras: alguns pedidos podem percorrer toda a cadeia sem ser tratados (sem handler padrão); dificulta depuração em cadeias longas.
  • Vs. Decorator: estrutura similar, mas Decorator sempre delega e não pode parar o fluxo; CoR pode interromper a cadeia ao processar o pedido.
  • Intenção: Definir o esqueleto de um algoritmo na superclasse, deixando que subclasses preencham etapas específicas sem modificar a estrutura geral.
  • Problema: Múltiplas classes realizam operações muito similares (parsing de dados em formatos diferentes, exportação para múltiplos destinos) mas duplicam a estrutura do algoritmo. Qualquer mudança na estrutura geral exige atualizar todas as classes.
  • Solução / Estrutura:
      • Classe Abstrata — declara o método template (geralmente final) que define a sequência fixa; declara métodos abstratos (etapas obrigatórias) e métodos concretos opcionais (com impl. padrão).
      • Ganchos (Hooks) — métodos com corpo vazio que subclasses podem (mas não precisam) sobrescrever para injetar comportamento em pontos específicos do algoritmo.
      • Classes Concretas — implementam as etapas abstratas; podem sobrescrever opcionais e ganchos.
  • Hollywood Principle: "Não nos chame, nós chamamos você." A superclasse controla o fluxo e chama as etapas das subclasses — inversão de controle na herança. A subclasse não decide quando suas etapas são chamadas.
  • Template Method vs. Strategy: Template Method usa herança — o esqueleto está na superclasse, a relação é estática, estabelecida em compile time. Strategy usa composição — o algoritmo inteiro está na estratégia, trocável em runtime. Use TM quando a estrutura do algoritmo é fixa e apenas detalhes variam; use Strategy quando o algoritmo inteiro deve ser intercambiável.
  • Quando usar: permitir que clientes estendam apenas partes específicas de um algoritmo; eliminar código duplicado em múltiplas classes com estruturas algorítmicas similares; algoritmo com estrutura fixa mas etapas variáveis.
  • Prós: elimina duplicação da estrutura; OCP para novas etapas; controle centralizado do fluxo.
  • Contras: restrição ao cliente — estrutura do algoritmo é imutável; pode violar LSP se subclasses sobrescrevem mais do que deveriam; hierarquia de herança pode ser difícil de manter com muitas variações.
  • Relaciona-se com: Strategy (alternativa baseada em composição), Factory Method (etapa de TM que cria objetos).
  • Intenção: Permitir que um objeto altere seu comportamento quando seu estado interno muda — como se mudasse de classe em runtime.
  • Problema: Máquinas de estado implementadas com switch/if no contexto crescem exponencialmente com novos estados. Lógica de transição espalhada por múltiplos métodos; adicionar um novo estado exige modificar todos os métodos condicionais.
  • Solução / Estrutura:
      • Contexto — armazena referência ao objeto de estado atual; delega chamadas de comportamento ao estado; expõe método para troca de estado (usado pelos próprios estados).
      • Interface de Estado — declara os métodos de comportamento.
      • Estados Concretos — implementam comportamentos específicos; podem iniciar transições de estado substituindo o estado do Contexto.
  • Quem controla as transições?
      • Contexto — centralizado; mais fácil de ver o quadro geral das transições, mas acopla o Contexto a todos os estados.
      • Estados — descentralizado; cada estado conhece os estados para os quais pode transicionar, mas cria acoplamento entre estados.
  • State vs. Strategy: estrutura idêntica, intenção diferente. State — os estados podem conhecer-se e iniciar transições; a troca é automática e baseada em lógica interna. Strategy — estratégias são independentes; a troca é manual, controlada pelo cliente.
  • Quando usar: comportamento varia significativamente com o estado; há muitos condicionais baseados em estado; código duplicado entre estados similares; novos estados serão adicionados frequentemente.
  • Prós: SRP — cada estado em sua classe; OCP — novos estados sem modificar contexto ou estados existentes; elimina condicionais complexos de máquina de estado.
  • Contras: overhead para máquinas simples com poucos estados; proliferação de classes; pode ser confuso se transições forem muitas e complexas.
  • Relaciona-se com: Strategy (mesma estrutura, semântica diferente), Singleton (estados concretos frequentemente sem estado interno — podem ser singletons compartilhados).
  1. Observer vs. Mediator: ambos desacoplam objetos. Observer é descentralizado (publicadores não conhecem assinantes); Mediator centraliza (todos falam com o mediador). Construa um cenário — como um formulário de UI com campos interdependentes — e argumente qual padrão é mais adequado e por quê. O que acontece quando a lógica de coordenação cresce?
  2. Como você implementa undo/redo robusto com Command + Memento? Descreva a estrutura de dados necessária, como lidar com comandos que modificam estado externo (banco de dados, arquivos) e por que undo de operações assíncronas é fundamentalmente diferente do síncrono.
  3. Chain of Responsibility vs. Decorator: descreva com precisão onde cada um quebra a analogia com o outro — especialmente em relação a: (a) capacidade de interromper o fluxo, (b) quem monta a cadeia, (c) semântica das chamadas.
  4. Strategy vs. Template Method: dado um sistema de geração de relatórios com 3 formatos (PDF, CSV, HTML) onde o pipeline é idêntico mas a serialização varia, qual padrão você escolheria? Agora suponha que em 6 meses o pipeline também pode variar entre relatórios — como isso muda sua escolha?
  5. Como Observer se relaciona com Event Sourcing e CQRS? Quando o Observer não é suficiente e você precisa da semântica de Event Sourcing (eventos persistidos, replayáveis e imutáveis)? Quais garantias o Observer não fornece que Event Sourcing fornece?

The Twelve-Factor App

  • I — Codebase: um repositório versionado; múltiplos deploys de um só código.
  • II — Dependencies: declare e isole dependências explicitamente (package.json, requirements.txt).

Autenticação & Autorização

JWT — JSON Web Token RFC 7519 · RFC 7515 (JWS) · RFC 7518 (JWA)

JWT é um contêiner compacto e URL-safe para transmitir claims entre partes. Por si só define apenas a estrutura — a segurança vem de JWS (assinatura) ou JWE (criptografia). Um JWT assinado é tecnicamente um JWS Compact Serialization.

  • Formato Compact: BASE64URL(Header) . BASE64URL(Payload) . BASE64URL(Signature)
    Cada parte é codificada em Base64URL (sem padding =, sem +/) — seguro para URLs e cookies.
  • Header (JOSE Header): objeto JSON com metadados de processamento. Claims mais relevantes:
      alg — algoritmo de assinatura (ex.: RS256, ES256)
      typ — tipo do token; para Access Tokens use at+JWT (RFC 9068), não JWT genérico
      kid — Key ID que identifica qual chave no JWKS deve ser usada para validação, essencial para key rotation
      cty — Content Type; em Nested JWTs vale JWT
  • Payload: conjunto de claims (RFC 7519 §4). Não criptografado — qualquer um com o token pode decodificar o Base64URL e ler o conteúdo. Nunca coloque segredos, senhas ou dados altamente sensíveis no payload de um JWS.
  • Signature (JWS): sign(alg, BASE64URL(header) + "." + BASE64URL(payload), key)
    Garante integridade (payload não foi modificado) e autenticidade (foi emitido por quem detém a chave privada). Não garante confidencialidade.
  • Serialização JSON completa (alternativa à Compact): permite múltiplas assinaturas por documento — usado em contextos não-HTTP como JOSE em documentos assinados, mas raro em autenticação web.

Claims são pares chave-valor no payload. Dividem-se em: Registered (RFC 7519), Public (IANA JWT Registry) e Private (acordadas entre partes).

  • iss (Issuer) — URI do emissor. O validador deve conferir que corresponde ao AS esperado. Previne tokens cross-issuer.
  • sub (Subject) — identificador único do principal (usuário ou serviço). Localmente único ou globalmente único por iss.
  • aud (Audience) — string ou array de strings identificando os destinatários. O RS deve rejeitar tokens onde seu identificador não esteja em aud — previne token confusion entre serviços.
  • exp (Expiration Time) — Unix timestamp (segundos). O validador deve rejeitar tokens expirados. Pequena tolerância de clock skew (~30–60 s) é aceitável.
  • nbf (Not Before) — Unix timestamp a partir do qual o token é válido. Útil para tokens pré-emitidos.
  • iat (Issued At) — timestamp de emissão. Permite calcular a idade do token.
  • jti (JWT ID) — identificador único do token (UUID ou valor opaco). Essencial para: (1) implementar denylist de revogação por token individual, (2) prevenir replay attacks em fluxos críticos. Armazene jti revogados no Redis com TTL = exp - now.
  • scope — espaço-separado, define permissões concedidas (padrão OAuth 2.0 / RFC 9068).
  • client_id — identifica o OAuth client que solicitou o token (RFC 9068 — obrigatório em at+JWT).
  • Claims OIDC no ID Token: nonce (replay protection — vinculado ao request original), auth_time (momento da autenticação do usuário), acr (Authentication Context Class Reference — nível de garantia, ex.: MFA), amr (Authentication Methods References — como autenticou: pwd, otp, hwk).
  • Claims privadas — evite nomes curtos sem namespace para não colidir com claims futuras. Use URIs ("https://myapi.com/roles") ou prefixo de domínio reverso.
  • HS256 / HS384 / HS512 (HMAC-SHA-2) — simétrico: mesma chave para assinar e verificar. Problema crítico em ambientes distribuídos: todo serviço que valida o token precisa conhecer o segredo compartilhado — vazamento em qualquer nó compromete toda a infraestrutura. Use apenas quando emissor = validador.
  • RS256 / RS384 / RS512 (RSASSA-PKCS1-v1_5 + SHA-2) — assimétrico RSA. O AS assina com a chave privada; qualquer RS valida com a chave pública (JWKS). PKCS1-v1_5 tem propriedades de maleabilidade; prefira PSS para novos sistemas. Chaves RSA ≥ 2048 bits (recomendado 4096 para longa duração).
  • PS256 / PS384 / PS512 (RSASSA-PSS) — RSA com padding probabilístico. Mais seguro que PKCS1-v1_5; sem as vulnerabilidades de maleabilidade. Preferido em sistemas de alta segurança.
  • ES256 / ES384 / ES512 (ECDSA, curvas P-256/P-384/P-521) — assinaturas menores e verificação mais rápida que RSA. ES256 gera assinaturas de 64 bytes vs. ~256 bytes do RS256. Atenção: ECDSA requer geração de nonce (k) verdadeiramente aleatório a cada assinatura — k reutilizado vaza a chave privada (falha clássica em implementações fracas).
  • EdDSA / Ed25519 (RFC 8037) — mais moderno, resistente a falhas de RNG, assinaturas determinísticas, alta performance. Ainda com adoção crescente em libs — verifique suporte antes de adotar.
  • alg: "none" — remove a assinatura completamente. Jamais aceite em produção. Sempre fixe os algoritmos aceitos na validação (allowlist, nunca extraia do header sem restrição).
  • JWKS (JSON Web Key Set — RFC 7517) — endpoint público (/.well-known/jwks.json) onde o AS publica suas chaves públicas. Cada JWK contém: kty (tipo de chave), use (sig ou enc), kid, e os parâmetros da chave (n/e para RSA; crv/x/y para EC).
    Key Rotation: o AS publica o novo par de chaves com um novo kid. O RS, ao encontrar um kid desconhecido, re-busca o JWKS (com cache + TTL). Tokens emitidos antes da rotação continuam válidos enquanto a chave antiga permanece no JWKS — remova apenas após o último token com aquela kid expirar.

A ordem correta de validação importa. Um erro em qualquer etapa deve resultar em rejeição imediata — nunca processe claims de um token não validado.

  • 1. Estrutura: confirme que o token tem exatamente 3 segmentos Base64URL separados por pontos.
  • 2. Header decodificado: leia alg e kid. Verifique que alg está na allowlist configurada (nunca confie no valor do header sem restrição).
  • 3. Chave correta: resolva a chave via kid no JWKS local (com cache). Se kid ausente e há uma única chave, use-a — mas prefira sempre kid explícito.
  • 4. Assinatura: verifique a assinatura criptográfica sobre header.payload. Qualquer falha → rejeite.
  • 5. exp: now < exp (com tolerância mínima de clock skew, ex.: 30 s).
  • 6. nbf: se presente, now ≥ nbf.
  • 7. iss: compare com o issuer esperado (string exact match ou allowlist).
  • 8. aud: confirme que o identificador do RS está presente no claim aud. Etapa frequentemente omitida — leva a token confusion attacks.
  • 9. jti (se revogação imediata for requisito): consulte a denylist. Hit → rejeite.
  • 10. Claims de negócio: valide scope, roles, tenant e outros claims relevantes para a operação.
  • Revogação de JWTs stateless:
      • TTL curto + Refresh Token: AT expira em minutos; RT permite re-emissão — melhor para a maioria dos casos.
      • Denylist por jti: armazena JTIs inválidos no Redis com TTL = exp - now. Overhead de consulta a cada request.
      • Key rotation imediata: remove a chave de assinatura do JWKS — invalida todos os tokens emitidos com ela. Opção nuclear: use apenas em emergências (comprometimento de chave).
  • Algorithm Confusion (alg switching) — ataque clássico: o atacante modifica o header de "alg":"RS256" para "alg":"HS256" e assina o token com a chave pública RSA do AS (que é pública e conhecida) como segredo HMAC. Bibliotecas vulneráveis aceitam o token porque a verificação HMAC com a chave pública passa. Defesa: sempre fixe o algoritmo esperado no código do validador — nunca leia alg do header para decidir como verificar.
  • alg: "none" — token sem assinatura. O atacante remove a assinatura e seta alg como none. Bibliotecas que aceitam tokens "não assinados" para flexibilidade são vulneráveis. Defesa: rejeite alg: none categoricamente.
  • JWK Header Injection — o atacante inclui um parâmetro jwk no próprio header do JWT com sua chave pública. Bibliotecas que usam o JWK embutido para validação aceitam qualquer token auto-assinado. Defesa: ignore completamente o parâmetro jwk no header; use apenas chaves do JWKS pré-configurado.
  • kid Injection — se o kid do header é usado em SQL query ou path de arquivo sem sanitização para buscar a chave, o atacante pode injetar ../../../dev/null (path traversal) ou SQL para que a chave retornada seja vazia ou previsível. Defesa: trate kid como dado não-confiável; use allowlist de KIDs válidos ou busque apenas no JWKS pré-carregado.
  • Missing aud Validation — um AT emitido para o serviço A é aceito pelo serviço B porque B não verifica aud. Defesa: sempre valide que o aud contém o identificador do serviço receptor — obrigatório, não opcional.
  • Weak HS256 Secret — segredos HMAC curtos ou previsíveis são vulneráveis a offline brute-force (ferramentas como hashcat conseguem testar bilhões de combinações/segundo). Defesa: use segredos HMAC com entropia mínima de 256 bits; prefira algoritmos assimétricos em sistemas distribuídos.
  • ECDSA Nonce Reutilizado (k) — ECDSA (ES256/ES384/ES512) requer um nonce aleatório único por assinatura. Reutilizar k — mesmo em duas assinaturas — vaza matematicamente a chave privada. Defesa: use bibliotecas auditadas; prefira Ed25519 (nonce determinístico).
  • Confusão ID Token / Access Token — usar o ID Token como credencial de acesso a APIs ou o Access Token para autenticar o usuário. O ID Token pode ter claims falsas de escopo; o AT não garante identidade do usuário. Defesa: use typ: "at+JWT" em ATs (RFC 9068); valide typ no receptor; APIs devem aceitar apenas ATs, não ID Tokens.

JWE — JSON Web Encryption RFC 7516 · RFC 7518 (JWA) · RFC 7517 (JWK)

JWE criptografa o payload — ao contrário do JWS (JWT assinado), o conteúdo fica ilegível para quem não possui a chave. Usa duas camadas de criptografia: uma para a chave (key management) e uma para o conteúdo (content encryption).

  • Compact Serialization (5 partes):
    BASE64URL(Header) . BASE64URL(Encrypted Key) . BASE64URL(IV) . BASE64URL(Ciphertext) . BASE64URL(Auth Tag)
  • Parte 1 — JWE Protected Header: metadados em JSON. Claims principais: alg (algoritmo de key management), enc (algoritmo de content encryption), kid, typ, cty (para Nested JWT: "JWT"), epk (Ephemeral Public Key — usado em ECDH-ES).
  • Parte 2 — JWE Encrypted Key: a Content Encryption Key (CEK) cifrada com a chave do destinatário usando o algoritmo alg. Em ECDH-ES puro ("alg":"ECDH-ES"), esta parte é vazia — a CEK é derivada diretamente pelo ECDH, não transmitida.
  • Parte 3 — Initialization Vector (IV): nonce aleatório gerado pelo emissor para o algoritmo de content encryption. Deve ser único por operação de ciframento com a mesma CEK.
  • Parte 4 — Ciphertext: plaintext cifrado com a CEK usando o algoritmo enc. Se zip: "DEF" estiver no header, o plaintext é comprimido antes de ser cifrado (veja riscos em Vulnerabilidades).
  • Parte 5 — Authentication Tag: tag de autenticação AEAD que cobre o Header + IV + Ciphertext. Garante integridade do ciphertext e do header — qualquer modificação no header ou no conteúdo é detectada antes da decriptação.
  • Processo de encriptação resumido:
    ① Gere uma CEK aleatória adequada para o algoritmo enc.
    ② Cifre a CEK com a chave pública do destinatário usando alg.
    ③ Gere um IV aleatório.
    ④ Cifre o plaintext com CEK + IV + header como AAD (Additional Authenticated Data).
    ⑤ Obtenha o Authentication Tag do passo ④.
    ⑥ Serialize as 5 partes em Base64URL separadas por pontos.

Key Management (alg) — como a CEK é protegida

  • RSA-OAEP (recomendado) / RSA-OAEP-256 — assimétrico; cifra a CEK com a chave pública RSA do destinatário usando padding OAEP (Optimal Asymmetric Encryption Padding). Seguro contra ataques de padding oracle modernos. Chave RSA ≥ 2048 bits.
  • RSA1_5 (RSA-PKCS1-v1_5) — NUNCA use. Vulnerável ao ataque de Bleichenbacher (1998): um oráculo de padding que, com ~milhões de queries, permite recuperar a CEK sem a chave privada. Ainda listado no RFC por compatibilidade; remova de qualquer allowlist.
  • ECDH-ES — Key Agreement efêmero. O emissor gera um par EC efêmero, executa ECDH com a chave pública EC do destinatário, deriva a CEK via HKDF-Concat. A chave pública efêmera (epk) é incluída no header. A CEK nunca é transmitida — parte 2 do JWE fica vazia. Forward secrecy por design: comprometer a chave privada do destinatário não decripta JWEs passados (o par efêmero é descartado após uso).
  • ECDH-ES+A128KW / ECDH-ES+A256KW — igual ao ECDH-ES mas usa o resultado do ECDH como key wrapping key (AES Key Wrap). A CEK é separada e encapsulada — permite multi-destinatários na serialização JSON.
  • A128KW / A256KW — AES Key Wrap simétrico. Simples e eficiente para cenários onde emissor e destinatário compartilham um segredo.
  • dir — a chave compartilhada é diretamente a CEK. Mais eficiente (sem key wrapping), mas sem separação de camadas — comprometer a chave expõe todos os JWEs cifrados com ela.
  • PBES2-HS256+A128KW / PBES2-HS512+A256KW — derivação de CEK a partir de senha (PBKDF2). Adequado para JWEs derivados de senhas humanas (ex.: exportação de dados cifrada por senha).

Content Encryption (enc) — como o payload é cifrado

  • A128GCM / A192GCM / A256GCM (recomendados) — AES no modo GCM (Galois/Counter Mode). Fornece AEAD nativo: ciframento + autenticação em uma operação. Altamente eficiente em hardware com instrução AES-NI. A256GCM é o padrão para sistemas modernos.
  • A128CBC-HS256 / A192CBC-HS384 / A256CBC-HS512 — AES-CBC + HMAC-SHA-2. A autenticação é computada separadamente via HMAC. Atenção: a ordem correta é verify-then-decrypt (MAC First); implementar decrypt-then-verify é vulnerável a padding oracle attacks em AES-CBC. Prefira AES-GCM em implementações novas.

Um Nested JWT combina assinatura (JWS) e criptografia (JWE) num único token. A ordem sign-then-encrypt é mandatória — invertê-la cria vulnerabilidades sutis mas sérias.

  • Estrutura: o payload do JWE é um JWS completo (JWT assinado). O header do JWE deve ter "cty": "JWT" para indicar que o plaintext é um JWT. Após decriptação, o receptor processa o JWS interno normalmente.
  • Por que sign-then-encrypt?
    Se você encrypt-then-sign: o assinante prova que assinou o ciphertext, mas não que conhecia o plaintext. Um atacante pode remover a assinatura original e re-assinar com sua própria chave — o receptor decripta e vê conteúdo legítimo mas acredita que o atacante foi o autor. Com sign-then-encrypt, a assinatura está dentro do envelope cifrado — somente quem decriptou pode verificar quem assinou.
  • Casos de uso:
      • Tokens com PII (dados pessoais sensíveis) que transitam por sistemas de logging ou intermediários que não devem ver o conteúdo.
      • Claims de autorização sensíveis (salário, diagnóstico médico) em contextos regulatórios (HIPAA, LGPD).
      • Tokens armazenados em cookies do lado do cliente onde o usuário não deve ter visibilidade das claims internas (ex.: feature flags, preços internos).
  • Custo: dois passes criptográficos (assinar + criptografar). Na maioria dos casos, confidencialidade por TLS é suficiente — use Nested JWT apenas quando há necessidade documentada de confidencialidade além da camada de transporte.

TLS protege o canal de transporte. JWE protege o objeto — o payload permanece cifrado independentemente de como ou por onde trafega.

  • TLS termina nos terminadores de TLS (load balancers, proxies reversos, API gateways). O payload viaja em plaintext dentro da infraestrutura interna. Sistemas de logging, APM, SIEM e outros intermediários veem o conteúdo.
  • JWE protege end-to-end por objeto — o payload está cifrado desde a criação até a decriptação pelo destinatário final. Load balancers, proxies, logs e eventuais vazamentos de HTTP history não expõem o conteúdo.
  • Quando JWE supera TLS isolado:
      • Tokens persistidos em bancos, caches (Redis), filas de mensagens ou object storage — TLS não cobre dados em repouso.
      • Tokens entregues ao browser/client que não devem ser lidos pelo JavaScript (ex.: cookies HttpOnly não bastam se o servidor pode ser comprometido — JWE garante que nem o servidor intermediário lê).
      • Comunicações multi-hop onde o token passa por terceiros (federação entre organizações, parceiros externos).
      • Requisitos regulatórios que exigem proteção de PII no nível de objeto, não apenas em trânsito.
  • Quando TLS é suficiente: a grande maioria dos sistemas internos. JWE adiciona complexidade operacional (gestão de chaves de criptografia, rotação de chaves EC/RSA de enc, separação de chaves de sig vs. enc). Não aplique sem necessidade real.
  • Separação de chaves: nunca use a mesma chave para assinatura (JWS) e criptografia (JWE). O claim "use": "sig" vs. "use": "enc" no JWK marca a finalidade — bibliotecas corretas devem rejeitar uso cruzado.
  • Bleichenbacher / Padding Oracle (RSA-PKCS1-v1_5) — o ataque mais devastador em JWE. Um oráculo de padding (serviço que responde diferente para ciphertexts inválidos vs. válidos) permite recuperar a CEK com ~milhões de requisições. Completamente evitado usando RSA-OAEP ou ECDH-ES.
  • Invalid Curve Attack (ECDH) — se o servidor aceita pontos EC sem validar que pertencem à curva correta, o atacante pode enviar pontos de curvas de pequena ordem e, por múltiplas tentativas, recuperar bits da chave privada. Defesa: use bibliotecas que validam pontos EC; nunca implemente ECDH manualmente.
  • CRIME/BREACH em zip: "DEF" — compressão antes de criptografia é vulnerável quando parte do plaintext é controlada pelo atacante: a variação no tamanho do ciphertext vaza informação sobre o conteúdo comprimido (análogo ao ataque BREACH em TLS com compressão HTTP). Defesa: evite zip quando o plaintext contém dados do atacante e dados secretos juntos.
  • IV/Nonce Reutilizado em AES-GCM — reutilizar o mesmo IV com a mesma CEK em AES-GCM é catastrófico: vaza a chave de autenticação e permite recuperar o plaintext. Defesa: gere IV aleatório de 96 bits por operação; rotacione a CEK antes de atingir o limite de operações (~2^32 para IVs aleatórios com AES-GCM 96-bit).
  • Decrypt-then-Verify em AES-CBC — em implementações com AES-CBC+HMAC, verificar o MAC após decriptação expõe um oracle de padding CBC ao atacante. Defesa: sempre verifique o HMAC antes de decriptar (constant-time comparison); prefira AES-GCM que torna isso impossível por design.
  • Uso de Chave sig para enc — usar a mesma chave RSA/EC para assinar JWTs e criptografar JWEs cria riscos de cross-context attacks. Defesa: separe sempre as chaves por finalidade; publique no JWKS com "use" correto.

Um JWT assinado (JWS) é como uma carta entregue em envelope transparente com lacre oficial. Qualquer entregador vê o conteúdo, mas sabe que é autêntico — o lacre não foi violado.

Um JWE é essa mesma carta dentro de um cofre portátil com combinação. O entregador (proxy, load balancer, log system) carrega o cofre sem conseguir ler o conteúdo. Apenas o destinatário que tem a chave (alg) consegue abrir o cofre e ler a carta. O cadeado do cofre é o key management (alg); a proteção do interior é a content encryption (enc).

Um Nested JWT (JWS dentro de JWE) é a carta com lacre oficial dentro do cofre. O destinatário abre o cofre (decripta), encontra a carta lacrada, verifica o lacre (valida a assinatura) e só então lê. Isso garante que quem assinou a carta conhecia seu conteúdo — e que ninguém no caminho leu nem adulterou nada.

Session-Based Auth

  • Fluxo: login → servidor cria sessão no store (Redis/DB) → devolve Session ID via cookie.
  • Stateful: servidor mantém estado — fácil revogar (delete session), mas escala horizontal requer session store compartilhado.
  • Cookie flags: HttpOnly (sem JS), Secure (só HTTPS), SameSite (CSRF), Path, Domain.
  • Session fixation: regenerar Session ID após login para evitar que atacante pré-fixe ID.
  • Session hijacking: roubo via XSS, MITM — mitigado por HttpOnly + Secure + HTTPS.

OAuth 2.0 RFC 6749 · RFC 7636 · OIDC Core 1.0

OAuth 2.0 é um framework de autorização (RFC 6749), não de autenticação. Delega acesso a recursos sem expor credenciais do Resource Owner ao Client. OpenID Connect (OIDC) adiciona a camada de identidade sobre ele.

  • Resource Owner — entidade que possui os dados (geralmente o usuário humano). É quem concede ou nega o acesso via consent screen.
  • Client — a aplicação que solicita acesso em nome do Resource Owner. Pode ser confidential (armazena segredo com segurança, ex: backend) ou public (não pode guardar segredo, ex: SPA, mobile).
  • Authorization Server (AS) — emite tokens após autenticar o Resource Owner e obter consentimento. Expõe os endpoints /authorize e /token. É a fonte de verdade sobre permissões.
  • Resource Server (RS) — hospeda a API protegida. Valida o Access Token a cada requisição (via introspecção local — JWT — ou chamada ao AS via RFC 7662).
  • OpenID Connect (OIDC) — extensão sobre OAuth 2.0 que padroniza autenticação. Introduz o ID Token (JWT com claims de identidade), o endpoint /userinfo e discovery via /.well-known/openid-configuration.
  • Scopes — strings que delimitam o nível de acesso solicitado (read:profile, write:orders). O AS pode conceder um subconjunto; o RS deve verificar o scope antes de servir cada recurso.
  • Authorization Code (RFC 6749 §4.1) — fluxo padrão para web apps com backend. O AS retorna um authorization code de uso único via redirect; o backend troca o code + client_secret por tokens no canal de back-channel (server-to-server). Tokens nunca ficam expostos no browser.
  • Authorization Code + PKCE (RFC 7636) — obrigatório para clientes públicos (SPAs, mobile, desktop). Substitui o client_secret por um par criptográfico efêmero:
      ① code_verifier — string aleatória URL-safe (43–128 chars) gerada pelo client, armazenada localmente.
      ② code_challenge = BASE64URL(SHA-256(code_verifier)) enviado no request inicial (code_challenge_method=S256).
      ③ No token request, o client envia o code_verifier original; o AS recomputa o hash e compara — um interceptador do code não consegue completar o tradeoff sem o verifier.
  • Client Credentials (RFC 6749 §4.4) — machine-to-machine (M2M). O client autentica-se como si mesmo (sem contexto de usuário) usando client_id + client_secret ou mTLS/private_key_jwt. Ideal para serviços backend, workers e CLIs. Não emite Refresh Token.
  • Device Authorization (RFC 8628) — dispositivos com entrada limitada (smart TVs, CLIs, IoT). O device exibe um user_code e uma URL; o usuário autoriza em outro device; o device faz polling no token endpoint até aprovação ou expiração do device_code.
  • Resource Owner Password Credentials (ROPC) — depreciado. O client coleta credenciais diretamente e troca por tokens. Incompatível com MFA, SSO e federação. Nunca usar em novos sistemas.
  • Implicit Flow — depreciado (OAuth 2.1 remove formalmente). Retornava tokens diretamente no fragment da URL de redirect, expondo-os ao browser history, logs de proxy e referer headers. Substituído por Authorization Code + PKCE.
  • Access Token (AT) — credencial de acesso ao RS, geralmente um JWT assinado (RS256/ES256). Contém sub, aud, exp, iat, scope. Deve ser de vida curta (padrão: 3600 s). O RS valida a assinatura localmente via JWKS (/.well-known/jwks.json) sem chamada ao AS — stateless e escalável. Não deve ser usado para autenticar usuários — ele prova autorização, não identidade.
  • Refresh Token (RT) — credencial de longa duração para obter novos ATs sem re-autenticar o usuário. Deve ser armazenado com segurança (HttpOnly cookie ou secure storage nativo). Suporte a Refresh Token Rotation (cada uso emite um novo RT e invalida o anterior) mitiga roubo de RT. Não emitido no fluxo Client Credentials.
  • ID Token (OIDC) — JWT assinado pelo AS que representa a identidade do usuário autenticado. Claims obrigatórios: iss, sub, aud, exp, iat. Claims opcionais: name, email, nonce (replay protection). Deve ser validado pelo client, nunca pelo RS.
  • Endpoint /authorize — inicia o fluxo Authorization Code. Parâmetros principais: response_type=code, client_id, redirect_uri, scope, state (CSRF token), nonce (OIDC), code_challenge + code_challenge_method (PKCE).
  • Endpoint /token — troca code por tokens. Para Authorization Code: grant_type=authorization_code, code, redirect_uri, client_id, client_secret (confidential) ou code_verifier (PKCE). Resposta: access_token, token_type=Bearer, expires_in, refresh_token, id_token.
  • Token Introspection (RFC 7662) — endpoint /introspect permite ao RS verificar validade e metadados de tokens opacos consultando o AS. Adiciona latência; use JWT local para alta frequência e introspecção para revogação imediata crítica.
  • Token Revocation (RFC 7009) — endpoint /revoke invalida AT ou RT imediatamente (ex: logout, compromisso de credencial). Com JWTs stateless, revogação exige token blocklist ou redução do TTL do AT.
  • Authorization Code Interception — atacante intercepta o code via custom URL scheme malicioso (mobile) ou open redirect. PKCE elimina o problema: sem o code_verifier original, o code é inútil.
  • CSRF no Authorization Endpoint — sempre inclua o parâmetro state (valor opaco vinculado à sessão do browser) e valide-o no callback. Em OIDC, use também nonce vinculado ao ID Token para prevenir replay attacks.
  • Open Redirect — o AS deve validar o redirect_uri contra a lista de URIs pré-registrados exatamente (sem wildcard). Um redirect_uri aberto permite que um atacante redirecione o code para seu servidor.
  • Clientes Públicos vs. Confidenciais — SPAs não podem guardar client_secret (o código-fonte é público). Use PKCE sem secret. Para mobile nativo, same principle: use PKCE + custom URI scheme ou claimed HTTPS schemes (universal links / app links).
  • Token Storage (SPAs) — Access Token em memória (variável JS) é mais seguro que localStorage (vulnerável a XSS). Refresh Token em HttpOnly cookie com SameSite=Strict e Secure. BFF (Backend for Frontend) pattern delega toda a gestão de tokens ao backend.
  • mTLS / DPoP como alternativas ao Bearer — Bearer tokens são "holder-of-token": quem tiver o token pode usá-lo. DPoP (RFC 9449) vincula o AT a uma chave assimétrica controlada pelo client (proof-of-possession), tornando tokens roubados inúteis sem a chave privada correspondente.
  • Logout Seguro em OIDC — RP-Initiated Logout: client redireciona para /logout do AS com id_token_hint e post_logout_redirect_uri. Front-Channel Logout: AS carrega iframes das RPs para invalidar sessões. Back-Channel Logout (mais confiável): AS envia POST com logout_token JWT diretamente para os endpoints das RPs via servidor — não depende do browser estar aberto.
  • Least Privilege / Minimal Scopes — solicite apenas os scopes necessários para a operação em curso. Tokens com escopo amplo aumentam o impacto de um compromisso. Prefira tokens de curta duração com refresh a tokens longos com escopo amplo.

Call Stack

  • Estrutura LIFO (Last In, First Out) que rastreia onde a execução está no programa.
  • Cada chamada de função empilha um "stack frame" — contém: parâmetros, variáveis locais, endereço de retorno.
  • Stack Overflow: recursão infinita sem base case esgota o stack.
  • JavaScript Event Loop: Call Stack + Web APIs + Callback Queue + Microtask Queue.
  • Microtasks (Promises, queueMicrotask): processadas antes da próxima iteração do event loop — esvaziam a microtask queue completamente.
  • Macrotasks (setTimeout, setInterval, I/O): processadas uma por vez a cada "tick".
  • Blocking the stack: operação síncrona longa congela a UI — use Web Workers para CPU-bound tasks.
📚 Uma pilha de livros: você lê o topo, e quando abre um capítulo que referencia outro livro, coloca esse novo livro por cima. Só quando termina de ler ele, volta ao de baixo. Cache — HTTP · Frontend · Browser · Arquitetura · Segurança

HTTP Cache RFC 9111 · Cache-Control · ETag · 304

Cache-Control é o header primário de controle de cache HTTP (RFC 9111). Pode aparecer na resposta (servidor → cache → cliente) e na requisição (cliente → cache). Substitui Expires e Pragma (legados).

  • no-store — nunca armazena a resposta em nenhum cache. Cada request vai obrigatoriamente ao origin. Use para dados altamente sensíveis (banking, medical). Sem cópia em disco, sem cópia em CDN.
  • no-cache — nome enganoso: a resposta pode ser armazenada, mas o cache deve revalidar com o origin antes de servir (via ETag/If-None-Match). Se origin retorna 304, a cópia local é servida sem re-download do body.
  • max-age=N — a resposta é "fresh" por N segundos a partir da emissão. Aplica-se a browser e caches compartilhados. Ex.: max-age=3600 = 1 hora de freshness.
  • s-maxage=N — como max-age mas somente para caches compartilhados (CDN, proxies). Sobrescreve max-age nesses contextos. Padrão para CDN longa + browser curta: Cache-Control: public, max-age=60, s-maxage=86400.
  • public — qualquer cache (browser, CDN, proxy) pode armazenar a resposta, mesmo que normalmente não pudesse (ex: responses a requests com Authorization).
  • private — somente o browser do usuário pode armazenar. CDNs e proxies compartilhados são proibidos. Padrão para conteúdo personalizado.
  • immutable — o browser não deve revalidar enquanto a resposta for fresh, nem mesmo no reload manual (F5). Elimina requisições condicionais desnecessárias para assets com hash no nome (main.a3f9.js). Sem efeito em caches compartilhados. Suporte: Chrome, Firefox, Safari modernos.
  • stale-while-revalidate=N — serve a resposta stale imediatamente e atualiza em background. N segundos de janela após o max-age expirar. Latência zero + dados quase atuais. Ex.: max-age=60, stale-while-revalidate=300.
  • stale-if-error=N — serve o dado stale por N segundos se o origin retornar 5xx ou timeout. Resiliência a falhas sem degradação de UX.
  • must-revalidate — quando stale, o cache deve revalidar antes de servir. Proíbe servir stale mesmo em falhas de rede (oposto de stale-if-error). Aplica-se a todos os caches.
  • proxy-revalidate — igual a must-revalidate mas só para caches compartilhados.
  • no-transform — proíbe que proxies/CDNs transformem o payload (ex: não comprimir imagens, não minificar JS).
  • Diretivas de requisição: no-cache (força revalidação), no-store (não armazene), max-age=0 (trate como stale), max-stale=N (aceita respostas stale por até N segundos), min-fresh=N (exige ao menos N segundos de freshness restantes), only-if-cached (só serve do cache; 504 se não disponível).
  • Estratégia recomendada por tipo:
      • HTML: no-cache (revalida sempre via ETag)
      • Assets com hash: public, max-age=31536000, immutable
      • API REST dados: private, no-cache ou private, max-age=30, stale-while-revalidate=60
      • Fonte/imagem sem hash: public, max-age=86400, stale-while-revalidate=604800

Validação condicional permite que o cache confirme se sua cópia ainda é válida sem re-baixar o body — economizando apenas latência (no caso de 304) ou banda + latência (em 200).

  • ETag (Entity Tag): identificador opaco do estado atual do recurso — geralmente hash do conteúdo, versão ou timestamp. Servidor envia: ETag: "abc123". Cliente revalida com: If-None-Match: "abc123". Se inalterado → 304; se mudou → 200 com novo body e novo ETag.
  • ETag Forte vs. Fraco:
      • Forte — "abc123": identidade byte-a-byte exata. Qualquer alteração, mesmo metadata, muda o ETag.
      • Fraco — W/"abc123": equivalência semântica; mesmo conteúdo com encoding diferente pode ter mesmo ETag fraco. Não pode ser usado com If-Range (downloads parciais).
  • Last-Modified + If-Modified-Since: alternativa baseada em data (Last-Modified: Wed, 15 Apr 2026 10:00:00 GMT). Cliente revalida com If-Modified-Since: Wed, 15 Apr 2026 10:00:00 GMT. Limitações: granularidade de 1 segundo; problemático em clusters com clocks divergentes; arquivos gerados dinamicamente sempre têm Last-Modified no momento atual mesmo sem mudanças de conteúdo.
  • Precedência: quando o servidor envia ambos (ETag + Last-Modified), o cliente envia ambos na revalidação. O servidor usa ETag como critério primário (RFC 9111 §13.1.2) — mais preciso.
  • ETag para Controle de Concorrência (OCC): cliente lê o recurso e armazena seu ETag. Na escrita, envia If-Match: "etag-original" (PUT/PATCH/DELETE). Se outro cliente modificou o recurso enquanto isso, o ETag mudou → servidor retorna 412 Precondition Failed. Implementação de "lost update prevention" sem locks pessimistas. Base do padrão de optimistic concurrency em REST.
  • Headers condicionais completos:
      • If-None-Match: GET/HEAD — retorna 304 se ETag bater (cache hit)
      • If-Match: PUT/DELETE — executa só se ETag bater (escrita condicional)
      • If-Modified-Since: GET/HEAD — retorna 304 se não modificado
      • If-Unmodified-Since: PUT/PATCH — executa só se não modificado desde a data
      • If-Range: GET com Range — retorna range se ETag bater, caso contrário retorna recurso completo
  • Geração de ETag no servidor:
      • Hash do conteúdo (MD5/SHA1 do body): forte, correto, mas caro para conteúdo dinâmico.
      • Versão do registro no banco: "v42" — barato, preciso.
      • Combinação inode + tamanho + mtime: estratégia padrão de servidores web estáticos (Nginx, Apache). Problemática em NFS e clusters sem inode sincronizado.
  • Vary header: expande a cache key incluindo o valor de headers de requisição. Vary: Accept-Encoding cacheia gzip e sem compressão separadamente. Vary: Accept-Language cacheia por idioma. Armadilha: Vary: Cookie ou Vary: Authorization cria uma variante por usuário → hit rate CDN = 0%.

304 Not Modified é a resposta do servidor indicando que o recurso não mudou desde a última versão que o cliente possui em cache. Permite revalidação sem re-download do body — economizando banda e reduzindo TTFB.

  • Fluxo completo:
    ① Primeira requisição: GET /style.css → servidor responde 200 OK com body + ETag: "v2" + Cache-Control: no-cache.
    ② Browser armazena o arquivo e o ETag.
    ③ Segunda requisição (cache stale ou no-cache): GET /style.css com header If-None-Match: "v2".
    ④ Servidor compara ETag: se não mudou → 304 Not Modified sem body. Se mudou → 200 OK com novo conteúdo.
    ⑤ Browser atualiza os headers da entrada em cache (novo ETag, novo max-age) e usa a cópia local.
  • 304 não tem body: a RFC proíbe explicitamente body em 304. Os headers da resposta 304 devem ser idênticos ao que seria retornado em uma resposta 200 (ETag, Cache-Control, Content-Location, Expires, Vary) para que o cache atualize seus metadados.
  • Impacto de performance: uma resposta 304 trafega tipicamente 200–400 bytes (somente headers) versus kilobytes ou megabytes do body. O benefício é maior para: recursos grandes (imagens, scripts), conexões lentas, mobile. Para recursos pequenos (<1KB), o overhead de um round-trip pode exceder o ganho — considere max-age com inline em vez de revalidação.
  • 304 com Last-Modified: fluxo paralelo usando datas. Requisição envia If-Modified-Since: [data]. Servidor retorna 304 se o recurso não foi modificado após essa data.
  • Quando 304 não é retornado:
      • Cache-Control: no-store — o cliente não armazena, então não envia condicionais.
      • Hard reload (Ctrl+Shift+R) — browser envia Cache-Control: no-cache e Pragma: no-cache sem If-None-Match.
      • Primeira visita / cache limpo — sem ETag armazenado, sem conditional request.
      • Response com max-age ainda fresh — browser não revalida, usa direto do cache (não há request nenhuma).
  • 304 vs. 200 com max-age:
      • max-age fresh: zero requests — resposta direta do cache local. Mais rápido.
      • no-cache + 304: um round-trip sem body. Garante que o conteúdo está atualizado com mínimo de banda.
      • no-store: um round-trip com body completo. Mais lento, mais banda.
  • Condicional request em APIs REST: clientes HTTP que implementam cache corretamente (ex: Axios com adaptadores de cache, bibliotecas de HTTP Java/Go) enviam condicionais automaticamente se a resposta anterior continha ETag. Isso reduz drasticamente a carga em APIs com dados que mudam raramente.

Frontend Cache React Query · SWR · staleTime · Revalidation

React Query (TanStack Query) e SWR são bibliotecas de server-state management para React. Gerenciam fetching, caching, sincronização e revalidação de dados remotos — substituindo o padrão manual de useEffect + useState + loading/error.

  • React Query — conceitos centrais:
      • useQuery(queryKey, queryFn, options) — hook de leitura. queryKey é a cache key (array serializado); queryFn é a função de fetch.
      • useMutation(mutationFn, options) — para escrita (POST/PUT/DELETE). Permite callbacks onSuccess, onError, onMutate.
      • QueryClient — instância central do cache. Compartilhada via QueryClientProvider.
      • queryClient.invalidateQueries(key) — marca queries como stale e re-fetcha se houver subscriber ativo.
      • queryClient.prefetchQuery(key, fn) — pré-carrega no cache sem subscriber (ex: hover em link).
      • queryClient.setQueryData(key, data) — atualização manual do cache (optimistic update).
  • SWR — conceitos centrais:
      • useSWR(key, fetcher, config) — hook único para leitura.
      • mutate(key, data, options) — atualização manual e revalidação. mutate(key) sem dados força re-fetch.
      • useSWRMutation(key, fetcher) — para mutations (escrita).
      • Deduplicação nativa: múltiplos componentes com a mesma key compartilham uma única request em voo.
      • Middleware: pipeline de plugins para logging, mock, autenticação, etc.
  • React Query vs. SWR — diferenças práticas:
      • RQ tem API mais rica: Infinite Query, Suspense mode nativo, devtools avançado, SSR/RSC primeiro-classe.
      • SWR é mais leve (~4KB gz) e mais simples — ideal para projetos menores ou quando SWR basta.
      • RQ tem gcTime (garbage collection) separado do staleTime; SWR mantém o cache indefinidamente enquanto houver subscriber.
      • RQ v5 renomeou cacheTime → gcTime para clareza semântica.
      • Ambos suportam: SSR com hydration, optimistic updates, background sync, window focus refetch, retry exponential backoff.
  • Optimistic Updates (React Query):
    ① onMutate: atualiza o cache imediatamente com o valor esperado (setQueryData); salva o snapshot do valor anterior.
    ② onError: rollback para o snapshot salvo.
    ③ onSettled: invalida a query para sincronizar com o servidor independente do resultado. UX imediata, consistência garantida.

Em React Query, dois timers independentes controlam o ciclo de vida de um dado em cache: staleTime (quando re-fetchar?) e gcTime/cacheTime (quando deletar do cache?).

  • staleTime (padrão: 0ms): quanto tempo os dados são considerados "fresh" (frescos). Enquanto fresh, React Query não fará re-fetch, mesmo que o componente remonte, a janela ganhe foco ou a conexão seja restabelecida. Após o staleTime, os dados ficam "stale" — o cache ainda os tem, mas um re-fetch em background será disparado na próxima oportunidade (mount, focus, reconnect).
  • gcTime / cacheTime (padrão: 5 min): quanto tempo dados de uma query sem subscribers ativos permanecem no cache antes de serem removidos pelo garbage collector. Se o componente que usa a query desmontar, o timer do gcTime começa. Se o componente remontar antes do timer expirar, o usuário vê os dados em cache imediatamente (possivelmente stale) enquanto o re-fetch ocorre em background.
  • Relação entre os dois:
      • staleTime=0, gcTime=5min (padrão): dados ficam stale imediatamente, mas permanecem no cache 5 min após desmontar. Re-fetch em todo mount/focus.
      • staleTime=60000, gcTime=5min: dados fresh por 1 min — sem re-fetch nesse período. Excelente para dados que mudam raramente.
      • staleTime=Infinity: dados nunca ficam stale automaticamente. Só re-fetcha via invalidateQueries manual. Use para dados estáticos (configurações, enums).
      • gcTime=0: cache removido imediatamente ao desmontar. Sem experiência de cache entre navegações — raramente desejável.
      • staleTime > gcTime: tecnicamente possível mas incoerente — o cache seria removido antes dos dados ficarem stale.
  • Visualização do ciclo de vida:
    fetch → dados chegam → [fresh: staleTime] → dados ficam stale → [stale: em cache, re-fetch no próximo trigger] → componente desmonta → [inactive: gcTime contando] → gcTime expira → [removed]
  • SWR equivalentes:
      • dedupingInterval (1000ms padrão) — janela de deduplicação. Múltiplas chamadas dentro desse intervalo compartilham a mesma promise em voo.
      • revalidateIfStale (true padrão) — equivale a staleTime=0 em React Query.
      • SWR não tem gcTime — mantém o cache enquanto houver subscriber ou até a chave ser sobrescrita.
  • Escolhas por domínio:
      • Dados do usuário logado (perfil): staleTime: 60_000 (1 min)
      • Listas de produtos: staleTime: 30_000
      • Configurações estáticas: staleTime: Infinity
      • Dados de tempo real (chat, notificações): staleTime: 0 + refetchInterval: 5000

Revalidação é o processo de re-fetchar dados que estão stale para mantê-los atualizados. React Query e SWR implementam revalidação automática em múltiplos triggers.

  • refetchOnMount (RQ) / revalidateOnMount (SWR): quando um componente monta e os dados estão stale, dispara re-fetch imediatamente. Com staleTime > 0, só re-fetcha se os dados estiverem stale no momento do mount. Opções RQ: true (se stale), 'always' (sempre), false (nunca).
  • refetchOnWindowFocus (RQ) / revalidateOnFocus (SWR): quando o usuário retorna à aba/janela, re-fetcha dados stale. Simula o comportamento de "puxar para atualizar" em mobile. Padrão: true em ambos. Desabilite para dados que raramente mudam ou quando o fetch é caro: refetchOnWindowFocus: false.
  • refetchOnReconnect (RQ) / revalidateOnReconnect (SWR): ao reconectar à internet após offline, re-fetcha dados stale. Essencial para apps PWA / mobile.
  • refetchInterval (RQ) / refreshInterval (SWR): polling — re-fetcha a cada N ms. Útil para: dashboards em tempo real, status de jobs, notificações. Opção avançada (RQ): refetchIntervalInBackground controla se o polling continua quando a aba está em background.
  • Revalidação manual — invalidateQueries (RQ):
    queryClient.invalidateQueries({ queryKey: ['todos'] }) — marca todas as queries com chave ['todos'] (ou prefixo) como stale e re-fetcha se houver subscriber. O padrão pós-mutation: invalidar no onSuccess da useMutation.
    queryClient.refetchQueries({ queryKey: ['todos'] }) — força re-fetch independente do staleTime.
  • Revalidação manual — mutate (SWR):
    mutate('/api/todos') — revalida a key globalmente (todas as instâncias).
    mutate('/api/todos', newData, { revalidate: false }) — atualização pessimista (sem re-fetch, confia no dado local).
    mutate('/api/todos', updateFn, { optimisticData }) — optimistic update com rollback automático em erro.
  • Stale-While-Revalidate no contexto frontend: quando um componente monta com dados stale em cache, React Query/SWR servem os dados em cache imediatamente (sem loading spinner) enquanto o re-fetch acontece em background. Quando os novos dados chegam, a UI atualiza suavemente. Isso elimina "flash of loading" em navegações de volta para uma rota já visitada — experiência percebida como instantânea.
  • Deduplicação de requests: se 10 componentes na mesma árvore chamam useQuery(['user', id]) simultaneamente, apenas 1 request HTTP é feita. Todos os componentes compartilham o resultado quando ele chegar. Evita o problema clássico de "N componentes = N fetches".
  • Suspense mode: React Query v5 e SWR suportam Suspense — a query integra com o mecanismo de Suspense do React. O componente "suspende" durante o fetch e o boundary mais próximo exibe o fallback. Elimina estados de loading explícitos no componente.

Browser Cache Cache API · Service Workers · Estratégias

A Cache API é uma interface de armazenamento de pares Request/Response, disponível em Service Workers e em qualquer contexto de página segura (HTTPS). Diferente do HTTP cache, ela é completamente controlada pelo desenvolvedor — sem expiração automática.

  • Interface CacheStorage (caches):
      • caches.open(name) → Promise<Cache> — abre ou cria um cache com o nome dado.
      • caches.has(name) → boolean — verifica se existe.
      • caches.keys() → string[] — lista todos os caches (útil para limpeza de versões antigas).
      • caches.delete(name) → boolean — remove um cache inteiro.
  • Interface Cache — operações:
      • cache.put(request, response) — armazena par explicitamente. Response pode ser criada (new Response(body, init)) ou clonada (response.clone()). Atenção: uma Response só pode ser consumida uma vez — sempre clone antes de armazenar: cache.put(req, res.clone()).
      • cache.add(url) — faz fetch da URL + armazena no cache.
      • cache.addAll([urls]) — fetch + cache de múltiplas URLs atomicamente. Falha se qualquer uma falhar — use para precaching do app shell.
      • cache.match(request, options) → Response | undefined — lookup. options.ignoreSearch ignora query string na comparação; options.ignoreVary ignora o header Vary.
      • cache.matchAll(request) → Response[] — retorna todas as variantes.
      • cache.delete(request) → boolean — remove uma entrada.
  • Sem expiração automática: diferente do HTTP cache, a Cache API não expira entradas automaticamente. O desenvolvedor é 100% responsável por: limpeza de versões antigas (no evento activate do SW), invalidação após atualizações, e gestão de quota. Use versionamento de cache name: 'app-v2'.
  • Quota e Storage: navigator.storage.estimate() retorna { quota, usage } em bytes. Em Chrome, a quota é tipicamente 60% do disco disponível por origem. Quando esgotada, escrita na Cache API falha. Use navigator.storage.persist() para pedir quota persistente (não evictada sob pressão).
  • Cache API vs. IndexedDB: Cache API armazena pares Request/Response (HTTP) — ideal para assets e respostas de API serializadas. IndexedDB armazena dados estruturados arbitrários — ideal para estado de aplicação, dados offline, listas de itens. Ambos são acessíveis em SW e na página.
  • Acesso fora do Service Worker: a Cache API é acessível em qualquer contexto seguro (HTTPS + localhost). Permite que o código da página popule o cache diretamente, útil para SSR/SSG com precaching manual sem Service Worker.

Service Workers são proxies programáticos entre o browser e a rede — scripts JavaScript que interceptam requisições de rede, gerenciam o cache e habilitam funcionalidades offline. Rodam em thread separada (sem acesso ao DOM).

  • Registro: navigator.serviceWorker.register('/sw.js', { scope: '/' }) — o scope define quais URLs o SW controla (padrão: diretório do arquivo sw.js). O SW deve estar no root para controlar toda a origem.
  • Ciclo de vida:
      • installing: SW baixado e parseado. Evento install dispara — ideal para precaching do app shell com cache.addAll([...]). Chame self.skipWaiting() aqui para pular a espera e ativar imediatamente.
      • installed / waiting: SW instalado mas aguardando que as abas com o SW antigo fechem. Um SW novo fica "esperando" até que não haja mais clientes com a versão antiga.
      • activating: SW antigo está sendo substituído. Evento activate dispara — ideal para limpar caches de versões anteriores. Chame self.clients.claim() para tomar controle de abas abertas imediatamente (sem esperar reload).
      • activated: SW ativo, controlando todas as requisições no scope.
      • idle / terminated: SW pode ser encerrado pelo browser a qualquer momento quando não há eventos. É recriado automaticamente quando necessário.
  • Evento fetch: intercepta toda requisição de rede dentro do scope. event.respondWith(promise<Response>) permite servir qualquer Response — do cache, da rede, ou sintetizada. Se respondWith não for chamado, a requisição prossegue normalmente.
  • Background Sync API: permite adiar operações que falharam por falta de conexão. O SW registra um sync tag (registration.sync.register('send-message')); quando a conexão é restabelecida, o browser dispara o evento sync no SW, que tenta a operação novamente. Ideal para formulários offline, analytics, likes sem conexão.
  • Push API: recebe notificações push mesmo com o site fechado. O servidor envia via Push Service (Google FCM, Mozilla AutoPush); o SW recebe o evento push e exibe a notificação via registration.showNotification(). Requer permissão explícita do usuário.
  • skipWaiting + clients.claim: padrão para atualizações imediatas de SW. Sem eles, o novo SW pode ficar "esperando" por horas se o usuário tiver a aba aberta. Com eles, a atualização é instantânea mas pode causar inconsistência se assets com cache antigo ainda estiverem carregados. Avalie o trade-off para cada app.
  • Workbox (Google): biblioteca sobre Service Workers com estratégias pré-construídas, precaching declarativo, routing, background sync e broadcast update. Elimina boilerplate de SW manual. CLI workbox-cli e integração com webpack/Vite.

As estratégias de cache em Service Worker definem o comportamento do evento fetch: quando usar o cache, quando usar a rede, e o que fazer quando ambos falham.

  • Cache First (Cache Falling Back to Network):
    Tenta o cache → se miss, vai à rede → armazena no cache → retorna.
    Use para: assets versionados (JS/CSS/imagens com hash), fontes, ícones. TTL implícito via versionamento da cache key.
    Risco: conteúdo stale se o arquivo for atualizado sem mudar a URL.
  • Network First (Network Falling Back to Cache):
    Tenta a rede → se falhar (timeout/offline), usa o cache.
    Use para: HTML, dados de API onde freshness é importante, conteúdo frequentemente atualizado.
    Risco: lento em conexões instáveis (aguarda o timeout da rede antes de usar o cache). Configure networkTimeoutSeconds para limitar a espera.
  • Stale While Revalidate:
    Retorna o cache imediatamente (sem latência) + dispara re-fetch da rede em background + atualiza o cache com a resposta da rede.
    Use para: recursos que devem carregar rápido mas que se beneficiam de atualizações frequentes — avatares, listas de artigos, conteúdo editorial.
    Detalhe: o usuário vê o dado do cache nesta visita; na próxima visita, verá o dado atualizado. Aceita 1 visita de defasagem.
  • Cache Only:
    Serve exclusivamente do cache; falha se não estiver no cache.
    Use para: recursos do app shell que você garante estar no cache no install event. Útil para modo offline estrito.
  • Network Only:
    Vai sempre à rede; nunca usa cache.
    Use para: analytics, pagamentos, requests que não devem ser respondidos offline, dados em tempo real críticos.
  • Combinações e roteamento: em produção, você aplica estratégias diferentes por padrão de URL:
    /static/ → Cache First (immutable assets)
    /api/ → Network First (com fallback)
    /*.html → Network First ou SWR
    /images/ → Stale While Revalidate
    Workbox permite declarar isso com registerRoute(matcher, strategy).
  • Precaching vs. Runtime Caching:
      • Precache: lista explícita de URLs cacheadas no install. Garante offline funcionando de imediato. Workbox gera a lista via CLI/build.
      • Runtime cache: URLs são cacheadas dinamicamente conforme são requisitadas, com a estratégia configurada. Não garante offline na primeira visita.

Network Only: "sempre peço ao fornecedor, nunca uso o estoque."

Arquitetura de Cache CDN · BFF Caching · Cache Layers

CDNs são redes de caches geograficamente distribuídos (Points of Presence — PoPs) que servem conteúdo próximo ao usuário, reduzindo latência e carga no origin.

  • Fluxo básico: usuário → PoP da CDN mais próximo → cache hit? serve diretamente (latência ~5–30ms) : request ao origin (latência +100–300ms) → armazena + serve.
  • s-maxage vs. max-age na CDN: s-maxage controla TTL apenas para caches compartilhados; max-age controla browser. Estratégia assimétrica: Cache-Control: public, max-age=60, s-maxage=604800 — browser revalida a cada minuto, CDN mantém por 7 dias. Ideal para conteúdo público com alta frequência de acesso.
  • Cache Key: por padrão, URL completa (método + scheme + host + path + query). CDNs permitem customizar — ex: ignorar certos parâmetros de tracking (utm_*) para evitar fragmentação de cache. Cloudflare: Cache Rules; Fastly: VCL custom hash.
  • Surrogate Keys / Cache Tags: associe respostas a tags de domínio via header Cache-Tag: product-42 (Cloudflare) ou Surrogate-Key: product-42 (Fastly/Varnish). Invalidação em massa por tag: PURGE /tags/product-42 invalida todas as URLs tagueadas — granular, eficiente, sem conhecer cada URL. Essencial para conteúdo editorial e e-commerce.
  • Origin Shield (Shield PoP): PoP intermediário entre os edge nodes e o origin. Quando múltiplos edges têm cache miss simultâneo (ex: após invalidação), apenas o Shield consulta o origin — os edges consultam o Shield. Reduz drasticamente o tráfego ao origin (request collapse).
  • Stale-while-revalidate na CDN: serve a versão stale com zero latência enquanto busca a atualização do origin em background. O usuário seguinte recebe a versão atualizada. Altamente eficaz em CDNs com alta concorrência — evita stampede no origin após expiração.
  • Estratégias de Purge/Invalidação:
      • Por URL — simples, O(n) para muitas páginas.
      • Por Surrogate Key/Tag — O(1) para n páginas, a mais eficiente.
      • Wildcard (/api/products/*) — suporte varia por CDN.
      • Global purge — opção nuclear; use só em emergências (deploy com bug crítico).
  • CDN Hit Rate e diagnóstico: hit rate = cache_hits / (cache_hits + cache_misses). Alvo: >90% para assets estáticos, >70% para conteúdo dinâmico. Causas de baixo hit rate: Vary: Cookie/Authorization, query strings únicas por usuário, TTLs curtos, alta proporção de conteúdo personalizado.
  • Edge Compute (Cloudflare Workers, Fastly Compute@Edge): código JavaScript/WASM rodando no PoP da CDN. Permite lógica de cache customizada: A/B testing no edge, geolocalização, personalização sem ir ao origin, respostas sintéticas.

O padrão BFF (Backend for Frontend) introduz uma camada de servidor específica para o cliente entre o frontend e os microserviços downstream. Além de agregação, é um ponto natural de cache — o BFF pode cachear dados que o cliente precisaria buscar em múltiplas APIs.

  • Por que o BFF é estratégico para cache:
      • Consolida N chamadas de API numa única response para o cliente — reduz round trips e oportunidades de cache miss.
      • Sabe exatamente o que o cliente precisa — pode cachear apenas os campos relevantes, sem overfetching.
      • Controla os headers de Cache-Control da resposta ao client — pode impor políticas que os microserviços não expõem diretamente.
      • Gerencia tokens de autenticação (OAuth access/refresh tokens em HttpOnly cookies) — o client nunca toca nos tokens, e o BFF pode cachear dados usando os tokens internamente.
  • Cache compartilhado vs. per-user no BFF:
      • Compartilhado (Redis): dados que são iguais para todos os usuários — catálogos de produtos, configurações, dados públicos. Cache key = rota + parâmetros. Cacheável também na CDN.
      • Per-user (Redis com session key): dados personalizados — carrinho, feed, notificações. Cache key = user_id + rota. Não cacheável na CDN (Cache-Control: private). TTL curto (30–300s).
  • BFF como cache de agregação: a página de dashboard precisa de dados de 5 microserviços. Sem BFF, o cliente faz 5 requests em paralelo. Com BFF, 1 request ao BFF que: (1) verifica cache → hit: retorna tudo; miss: faz 5 requests em paralelo aos serviços, agrega, armazena no cache e retorna. Para a segunda requisição de qualquer usuário, é um cache hit.
  • Invalidação no BFF: ao contrário do browser cache, o BFF pode subscrever eventos de domínio (Kafka, webhooks dos serviços) para invalidar proativamente entradas relevantes. Ex: evento product.updated → BFF invalida todas as chaves de cache relacionadas ao produto no Redis.
  • Rate limiting e colapso de requisições: o BFF pode implementar request coalescing — se 100 usuários requisitam o mesmo dado simultaneamente e é um cache miss, apenas 1 requisição vai ao serviço downstream. As outras 99 aguardam o resultado. Padrão de "singleflight" — evita cache stampede no nível de BFF.
  • Riscos do BFF como cache:
      • Ponto único de falha se o BFF cair — use HA e circuit breaker.
      • Cache stale se a invalidação falhar — monitore janela de inconsistência.
      • BFF "gordo" pode virar gargalo se cachear dados muito pesados — use paginação.
      • Complexidade de invalidação com múltiplos serviços upstream.

Sistemas de alto tráfego usam cache em múltiplas camadas — cada uma com latência, escopo e custo diferentes. A chave é entender o que cada camada pode e não pode cachear.

  • L1 — In-Process (aplicação):
    Latência: ~ns. Escopo: por instância da aplicação. Volátil (perdido no restart). Sem overhead de rede.
    Libs: Caffeine (Java), node-lru-cache (Node.js), cachetools (Python), groupcache (Go).
    Use para: dados que não mudam entre requests (configurações, feature flags, enums). TTL muito curto (30–60s) para não servir stale demais em múltiplas instâncias.
    Problema: inconsistência entre instâncias — instância A tem cache quente com dado v1 enquanto instância B já recebeu o v2. Mitigar com TTL curto ou invalidação via pub/sub (Redis Pub/Sub notifica todas as instâncias para limpar L1).
  • L2 — Distributed Cache (Redis/Memcached):
    Latência: ~0.1–1ms. Escopo: compartilhado entre todas as instâncias. Sobrevive a restarts da aplicação.
    Cache-Aside, Read-Through, Write-Through/Behind.
    Use para: sessões, tokens, dados de domínio computados, rate limiting state.
  • L3 — CDN / Edge:
    Latência: ~5–50ms. Escopo: global por PoP. Apenas conteúdo público (sem Authorization, sem private).
    Use para: assets estáticos, respostas de API públicas, HTML de páginas indexáveis.
  • L4 — Browser / Client:
    Latência: ~0ms (memory cache) a ~1ms (disk cache). Escopo: por usuário/dispositivo. Controlado por Cache-Control.
    Use para: assets com hash (immutable), dados de sessão do app (React Query / SWR), app shell via Service Worker.
  • Estratégia de leitura (cache lookup cascade):
    Request → L4 (browser) → miss → L3 (CDN) → miss → L2 (Redis) → miss → L1 (in-process) → miss → Origin (banco/serviço). Na prática, a ordem é determinada pela arquitetura: o browser consulta o CDN (não o Redis diretamente).
  • Consistência entre camadas: o maior desafio é garantir que quando o dado muda na origem, todas as camadas são invalidadas na ordem correta:
    ① Atualizar banco → ② Invalidar L2 (Redis) → ③ Purge CDN (via API) → ④ Browser invalida automaticamente no próximo request condicional (ETag).
    Se L3 não for invalidado junto com L2, usuários que acertam o CDN recebem dados stale mesmo após atualização do banco.
  • Cache Warming: após um deploy ou falha, as camadas de cache estão frias (cold start). Estratégias:
      • Pré-aquecimento via script que gera requests aos endpoints mais críticos antes do tráfego chegar.
      • Gradual rollout (canary) — uma instância aquece o cache antes do traffic shift completo.
      • Read-through com fallback gracioso — primeira request é lenta, mas não quebra.

Segurança de Cache Cache Leakage · XSS + Storage · Cache Poisoning

Cache Leakage ocorre quando dados privados ou sensíveis de um usuário são armazenados em caches compartilhados e servidos para outros usuários. Uma das classes de vulnerabilidades mais comuns e subestimadas em produção.

  • Causa raiz mais comum: resposta autenticada (dados do usuário A) armazenada em cache CDN ou proxy e servida ao usuário B. Basta um Cache-Control: max-age=300 em uma rota autenticada sem private para que a CDN armazene a resposta do primeiro request e a sirva para todos os requests subsequentes àquela URL.
  • Regra de ouro: qualquer resposta que varia por usuário (contém dados personalizados, sessão, informações de conta) deve ter Cache-Control: private ou no-store. A ausência dessas diretivas em respostas com dados pessoais é uma vulnerabilidade de privacidade (LGPD, GDPR).
  • Responses com Authorization header: por padrão (RFC 9111 §3.5), caches compartilhados não devem armazenar responses a requests que contêm Authorization, a menos que a resposta inclua explicitamente Cache-Control: public ou s-maxage. Muitas CDNs seguem essa regra por padrão, mas nem todos os proxies internos seguem.
  • Cookie-based auth e CDN: se a autenticação usa cookies (não Authorization header), a proteção RFC acima não se aplica. A CDN pode cachear a resposta a não ser que private esteja explícito. Sempre inclua Cache-Control: private, no-store em qualquer rota que leia o cookie de sessão.
  • Vary: Cookie — armadilha: Vary: Cookie cria uma variante de cache por valor de cookie. Embora correto semanticamente, (a) aumenta o tamanho do cache CDN, (b) em CDNs que usam o valor do cookie como parte da cache key, um cookie com valor único por usuário = cache miss para todos. Não é segurança; é disfuncionalidade. Use private.
  • CDN com múltiplos tenants: CDNs de cloud servem múltiplos clientes na mesma infraestrutura. Certifique-se de que a configuração de cache key isola sua origem das demais. Hosts virtuais distintos têm namespaces separados na maioria das CDNs, mas misconfigurações de CNAME podem criar colisões. Valide com pentest de CDN isolation.
  • ETag como fingerprint (histórico): browsers usavam ETags para rastrear usuários entre sessões incógnitas — o ETag persistia no cache mesmo após limpar cookies. Browsers modernos limpam ETags junto com cookies no modo privado. Mas ETags em subresources (fontes, scripts) podem ainda ser usados para correlação.
  • X-Cache / X-Cache-Status headers: headers de CDN que indicam cache hit/miss e o PoP. Vazar esses headers em respostas públicas pode revelar topologia de infraestrutura. Remova em produção ou restrinja a IPs internos.
  • Diagnóstico de leakage: use ferramentas como Burp Suite ou curl para verificar: (1) faça request autenticado, copie a URL, faça o mesmo request sem autenticação ou em sessão diferente. Se receber o mesmo conteúdo, há leakage. Em CDN: compare Age: N — se N > 0 em request sem auth, é cache hit de uma resposta privada.

XSS (Cross-Site Scripting) e storage do browser interagem diretamente: onde você armazena tokens e dados sensíveis determina o que um atacante pode roubar via injeção de JavaScript.

  • localStorage e sessionStorage:
    Acessíveis a qualquer JavaScript da mesma origem — incluindo scripts injetados via XSS.
    localStorage.getItem('access_token') é trivial para um atacante com XSS.
    Nunca armazene: JWT access tokens, OAuth tokens, session IDs, PII.
    Use para: preferências de UI, dados não sensíveis, estado de app que não é segredo.
    Diferença entre os dois: localStorage persiste entre sessões; sessionStorage é limpo ao fechar a aba — mas ambos são igualmente vulneráveis a XSS enquanto a página está aberta.
  • Cookies com HttpOnly:
    HttpOnly — inacessível ao JavaScript; protege contra XSS que tenta ler o token.
    Secure — só enviado em HTTPS.
    SameSite=Strict — cookie não enviado em nenhum request cross-site (proteção CSRF forte).
    SameSite=Lax — enviado em navegações GET de top-level (ex: clicar num link), mas não em requests Ajax cross-site. Padrão em browsers modernos.
    SameSite=None; Secure — enviado cross-site. Necessário para cookies em iframes embeddados (OAuth redirects), mas remove proteção CSRF — exige CSRF token adicional.
    Conclusão: HttpOnly cookie é o storage mais seguro para tokens de autenticação.
  • IndexedDB: storage estruturado de alta capacidade. Acessível a qualquer JS da origem. Vulnerável a XSS exatamente como localStorage. Use para: dados offline de app, não para tokens.
  • Cache API (Service Worker): acessível ao SW e às páginas da origem via caches. Menos trivialmente explorada via XSS (requer código específico). Mas se um atacante pode registrar um SW malicioso (via XSS persistente + SW scope adequado), ele pode interceptar todas as requisições da origem — ataque mais severo que roubar um token.
  • Memory (variável JS — padrão para SPAs): mais seguro para access tokens — não persiste em storage, não acessível via DevTools de forma trivial, limpo ao recarregar.
    Fluxo recomendado (RFC 9700 / Auth0 recommendation):
    ① Access Token em memória (curto prazo, ex: 15 min).
    ② Refresh Token em HttpOnly cookie (não acessível ao JS).
    ③ Quando o AT expira, o app faz request ao endpoint de refresh — o RT é enviado automaticamente pelo browser via cookie.
    ④ Servidor valida RT e retorna novo AT.
    Um XSS não consegue extrair o RT (HttpOnly) nem o AT (memória — só existe enquanto a página está aberta).
  • Content Security Policy (CSP) como defesa em profundidade: mesmo que XSS seja injetado, um CSP restritivo (script-src 'self') impede a execução de scripts inline e externos não autorizados. Camada adicional que não substitui o storage seguro, mas limita o raio de explosão.
  • Subresource Integrity (SRI): garante que scripts externos (CDN) não foram adulterados. <script src="cdn.js" integrity="sha384-...">. Se o arquivo da CDN for substituído por um atacante, o browser rejeita. Relevante para o cenário de CDN comprometida que entrega XSS em escala.

Cache Poisoning é um ataque onde o atacante manipula o que é armazenado no cache, fazendo com que o cache sirva conteúdo malicioso ou incorreto para outros usuários.

  • HTTP Cache Poisoning: o atacante encontra um header de requisição que (1) influencia a resposta do servidor mas (2) não faz parte da cache key da CDN. Ele envia uma requisição com esse header malicioso, o servidor reflete o valor na resposta, a CDN armazena a resposta envenenada e a serve para todos os usuários sem esse header.
    Exemplo clássico: servidor usa X-Forwarded-Host para construir links (script src="https://[X-Forwarded-Host]/app.js"). Atacante envia X-Forwarded-Host: evil.com. CDN não inclui esse header na cache key. CDN armazena resposta com script src="https://evil.com/app.js" e serve para todos.
    Defesa: normalize e valide todos os headers que influenciam respostas; adicione headers críticos à cache key da CDN; não construa URLs a partir de headers de request sem allowlist estrita.
  • Web Cache Deception: atacante induz a vítima a acessar uma URL "disfarçada" de asset estático, ex: /account/profile/photo.jpg. O servidor ignora a extensão e retorna a página de perfil autenticada da vítima. A CDN, configurada para cachear .jpg, armazena o conteúdo privado. O atacante acessa a mesma URL sem autenticação e obtém os dados privados da vítima.
    Defesa: nunca sirva conteúdo dinâmico/autenticado em URLs com extensões de asset estático; use Content-Type real como parte da cache key; configure CDN para cachear apenas respostas com Content-Type de asset estático (image/*, text/css, application/javascript).
  • Response Splitting / Header Injection: se valores de input do usuário são refletidos em headers HTTP sem sanitização, o atacante pode injetar newlines (\r\n) para inserir headers adicionais ou até split a resposta em duas — a segunda sendo uma resposta completamente fabricada que o cache armazena. Defesa: sanitize toda saída em headers; use frameworks que escapam headers automaticamente.
  • DNS Cache Poisoning: atacante injeta registros DNS falsos no cache de um resolver. Usuários que consultam aquele resolver são direcionados a IPs maliciosos. Defesa: DNSSEC (registros DNS assinados criptograficamente); DoH (DNS over HTTPS); DoT (DNS over TLS); validação DNSSEC no resolver.
  • Cache Key Injection: se a cache key é construída a partir de input do usuário sem sanitização (ex: cache_key = tenant_id + ":" + resource_id), o atacante pode forjar chaves com separadores: tenant_id = "A:B" → colide com chave do tenant B. Defesa: use hashing dos componentes da cache key; não use delimitadores que podem aparecer nos valores; sanitize todos os componentes.
  • Redis sem autenticação: Redis exposto sem requirepass + ACLs em rede acessível = trivialmente explorado. Atacante pode sobrescrever qualquer chave de cache, injetar dados maliciosos, comprometer sessões. Defesa: Redis nunca exposto à internet; VPC privada; requirepass forte + ACL por usuário; TLS (tls-port); firewall de porta.
  • CDN Origin Header Manipulation: CDNs encaminham requests ao origin com headers adicionais (True-Client-IP, CF-Connecting-IP). Se o origin confia nesses headers para autorização ou logging sem validar a origem da requisição, atacantes que acessam o origin diretamente (bypass CDN) podem falsificar esses headers. Defesa: origin deve aceitar requests apenas do IP range da CDN; valide o shared secret entre CDN e origin.

Cache Leakage é como um hotel que usa a mesma fotocópia do quarto 301 (com os pertences do hóspede visíveis) para todos os recém-chegados do mesmo andar. O hóspede original deixou dados privados expostos porque o recepcionista (CDN) não percebeu que o documento era privado (Cache-Control: private ausente).

Cache Poisoning é um cliente que convence o balconista a colocar uma instrução falsa no mural interno da loja. Todo funcionário que consulta o mural segue a instrução envenenada — sem saber que foi adulterada. Os clientes legítimos sofrem as consequências sem nenhum sinal de alerta.

localStorage vs. cookie HttpOnly é a diferença entre deixar a chave do cofre em cima da mesa (qualquer visita maliciosa [XSS] vê e copia) vs. soldar a chave dentro do cofre (o gerente [browser] a usa automaticamente, mas nem você pode vê-la com código — apenas o servidor pode abrir o cofre).

  1. Você está fazendo um pentest de CDN cache leakage. Descreva o procedimento de teste passo a passo: quais URLs testar primeiro, quais headers enviar, como interpretar o header Age na resposta, e como confirmar que uma resposta autenticada está sendo servida a usuários não autenticados.
  2. Um atacante descobre que o servidor usa o header X-Forwarded-Proto para decidir se links gerados usam http:// ou https://, e que a CDN não inclui esse header na cache key. Descreva o ataque de HTTP Cache Poisoning completo — request do atacante, o que a CDN armazena, e o impacto para usuários legítimos. Como você corrige na CDN e no servidor?
  3. Argumente: "tokens de autenticação em memory (variável JS) são mais seguros que em localStorage, mas menos seguros que em HttpOnly cookie". Para cada comparação, descreva o cenário de ataque específico que justifica a afirmação — não apenas a regra geral.
  4. Um Service Worker com vulnerabilidade de XSS persistente é registrado em seu domínio. O que exatamente o atacante pode fazer com acesso ao SW? Como isso é mais perigoso do que XSS sem SW? Quais mecanismos do browser (CSP, Feature Policy, SW scope) limitam o dano?
  5. Sua API de BFF usa Redis para cache. Um log de segurança mostra que o Redis recebeu um comando FLUSHALL inesperado. Descreva: (a) como o atacante pode ter chegado ao Redis, (b) o impacto imediato no sistema (Cache Avalanche), (c) as ações imediatas de contenção, e (d) as mudanças arquiteturais para prevenir recorrência.
