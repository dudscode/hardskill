Segurança — CORS, CSP e CSRF

CORS Cross-Origin Resource Sharing

  • Política Same-Origin (SOP): por padrão o browser bloqueia requisições de uma origem para outra diferente (protocolo+host+porta).
  • CORS relaxa essa política via cabeçalhos HTTP que o servidor envia.
  • Preflight (OPTIONS): para métodos não-simples (PUT, DELETE, cabeçalhos customizados), o browser envia uma request OPTIONS antes da real.
  • Cabeçalhos chave: Access-Control-Allow-Origin, Access-Control-Allow-Methods, Access-Control-Allow-Headers, Access-Control-Allow-Credentials.
  • Wildcard * e credenciais não podem coexistir — ao usar cookies/tokens, a origem deve ser explícita.
  • Erro comum: configurar CORS no frontend — ele só existe no servidor.
🏘️ Imagina que você mora numa rua e quer pegar uma bola no quintal do vizinho. O vizinho tem uma placa na porta dizendo quem pode entrar. Se sua casa está na lista, tudo bem. Se não está, você precisa bater, perguntar se pode entrar (preflight), e só aí ele diz "pode" ou "não pode". O browser é o guarda que lê essa placa antes de você atravessar.
  1. Qual a diferença entre uma requisição CORS "simples" e uma que dispara preflight? Quais critérios definem cada uma?
  2. Por que Access-Control-Allow-Origin: * é insuficiente quando a request inclui withCredentials: true? O que acontece internamente no browser?
  3. Como você implementaria uma whitelist dinâmica de origens sem introduzir header injection ou refletir origens arbitrárias de forma insegura?
  4. O que é "CORS bypass via subdomain takeover" e como uma configuração incorreta de CORS torna um sistema vulnerável a esse vetor?
  5. Explique o comportamento de CORS em Service Workers — eles herdam as políticas da página ou operam de forma independente?

CSP Content Security Policy

  • Defesa contra XSS: informa ao browser quais fontes de conteúdo são confiáveis.
  • Diretivas principais: default-src, script-src, style-src, img-src, connect-src, frame-ancestors.
  • Nonces e hashes permitem scripts inline sem abrir 'unsafe-inline' — cada request gera um nonce único.
  • Report-Only: modo de auditoria — viola mas não bloqueia; envia reports para um endpoint via report-uri / report-to.
  • Strict CSP: usa nonce + 'strict-dynamic' para eliminar listas de hosts (que podem ser bypassadas via CDN comprometido).
  • Entregue via header HTTP ou meta tag (limitado — sem frame-ancestors).
🎒 Pensa num lanche escolar: a escola manda uma lista do que pode entrar na lancheira. Só comida da lista é liberada para comer. Se alguém tentar colocar doce que não está na lista, a professora descarta. O CSP é essa lista — ela diz ao browser "só execute código que veio daqui, ignore o resto".
  1. Por que script-src 'unsafe-inline' anula praticamente toda a proteção do CSP contra XSS?
  2. Explique como 'strict-dynamic' muda o modelo de confiança e por que ele é superior a uma whitelist de domínios para scripts.
  3. Como um atacante pode bypassar um CSP que permite script-src https://cdn.example.com se o CDN hospedar conteúdo enviado por usuários (ex.: JSONP endpoints, Angular/React CDN)?
  4. Qual é a diferença entre frame-ancestors no CSP e o header X-Frame-Options? Em qual cenário um pode falhar mas o outro não?
  5. Descreva uma estratégia de rollout de CSP em produção sem causar regressão — incluindo fases de auditoria, tratamento de violações legítimas e gestão de nonces em SSR.

CSRF Cross-Site Request Forgery

  • Ataque: força o browser autenticado de uma vítima a enviar requisições involuntárias para outro site.
  • Explora cookies automáticos — o browser os envia em toda request sem que o JS precise tocá-los.
  • Defesas principais: CSRF tokens (synchronizer token pattern), Double Submit Cookie, SameSite cookies.
  • SameSite=Strict/Lax é a mitigação mais moderna — impede o browser de enviar cookies em requisições cross-site.
  • APIs que usam apenas Bearer tokens (Authorization header) não são vulneráveis a CSRF pois o JS precisa setar explicitamente o header.
  • Verificar Origin/Referer como defesa complementar mas não suficiente isolada.
🎭 Você está logado no banco pela internet. Um vilão manda um e-mail com um botão que, quando clicado, abre uma janela invisível e faz seu browser pedir ao banco para transferir dinheiro — usando seu cookie de login, que o browser manda sozinho. O banco não sabe que não foi você. O CSRF token é como um segredo extra que só o site do banco conhece e que o formulário precisa incluir.

  4. Como o Double Submit Cookie difere do Synchronizer Token Pattern em termos de estado server-side e quais são as fraquezas de cada um?
  5. Em uma arquitetura de microserviços onde um BFF (Backend for Frontend) faz chamadas internas, como você projeta a estratégia CSRF de ponta a ponta?

Arquitetura

Clean Architecture

  • Regra da dependência: camadas internas não conhecem camadas externas — dependências só apontam para dentro.
  • Camadas (de dentro pra fora): Entities → Use Cases → Interface Adapters → Frameworks/Drivers.
  • Entities: regras de negócio enterprise — independentes de tudo.
  • Use Cases: orquestram entities; definem Ports (interfaces) que serão implementados por adapters.
  • Testabilidade: core testável em isolamento total sem banco, framework ou HTTP.
  • Inversão de dependência (DIP): o use case define a interface do repositório; o adapter implementa.
🧅 Uma cebola: o miolo (regras do negócio) não sabe que existe a casca de fora (banco de dados, tela, framework). As camadas de fora dependem das de dentro, nunca o contrário. Trocar o banco de dados é como trocar a casca — o coração da cebola não muda nada.

  2. Qual a diferença entre Clean Architecture e Hexagonal Architecture (Ports & Adapters)? São equivalentes ou há nuances importantes?
  3. Quando Clean Architecture se torna over-engineering? Defina os sinais de que o custo supera o benefício para um determinado projeto.
  4. Como você lida com "cross-cutting concerns" como logging e tracing sem vazar infraestrutura para os Use Cases?
  5. Explique como você modelaria um domínio rico com agregados DDD dentro da camada de Entities sem que ela dependa de ORMs.

MVC Model-View-Controller

  • Model: dados e regras de negócio; notifica a View quando muda.
  • View: apresentação — renderiza o estado do Model.
  • Controller: recebe input do usuário, manipula Model, seleciona View.
  • Fluxo clássico: User → Controller → Model → View → User.
  • Problemas comuns: Massive View Controller (iOS), lógica de negócio vazando para Controller.
  • Variações: MVP (passive view), MVVM (data binding), MVT (Django), VIPER (iOS).
🍽️ Num restaurante: o garçom (Controller) recebe o pedido do cliente, fala com a cozinha (Model) que prepara o prato, e o prato vai para a mesa (View) para o cliente ver e comer. O cliente não vai até a cozinha, e a cozinha não precisa saber como a mesa está decorada.

  2. Como o padrão MVC se manifesta diferentemente em frameworks server-side (Rails, Laravel) vs. client-side (Backbone.js)?
  3. Por que Controllers "gordos" surgem na prática e quais refatorações (ex.: Service Objects, Form Objects) você aplicaria?
  4. Em um SPA com React, onde MVC se encaixa — ou ele foi substituído por outro modelo? Argumente.
  5. Como você testaria cada camada do MVC de forma isolada sem acoplamento entre elas?

MVVM Model-View-ViewModel

  • ViewModel: expõe estado observável e comandos; não referencia a View.
  • Data Binding: a View se sincroniza automaticamente com o ViewModel — bidirecional ou unidirecional.
  • Separação de responsabilidades: lógica de apresentação fica no ViewModel, testável sem UI.
  • Popularizado pelo WPF/Silverlight, amplamente adotado em iOS (Combine), Android (Jetpack), Angular, SwiftUI.
  • Reatividade: ViewModel usa Observables/Streams; View "reage" a mudanças sem polling.
📟 Pense num painel de carro: o ViewModel é o computador de bordo que recebe dados do motor (Model) e converte em "velocidade: 80 km/h", "combustível: 40%". O painel (View) só exibe o que o computador já preparou. Se você trocar o painel, o computador de bordo continua igual.
  1. Qual o problema de ter um ViewModel que referencia diretamente elementos de UI (ex.: TextView no Android)? Como isso violaria os princípios do MVVM?
  2. Explique one-way data flow (Redux/Unidirectional) vs. two-way data binding — quando cada um é preferível?
  3. Como você lida com "ViewModel Explosion" em aplicações grandes onde cada tela tem um ViewModel complexo?
  4. No contexto de Kotlin Coroutines + StateFlow no Android, como você garante que o ViewModel sobrevive a rotação de tela mas não a navegação para outra stack?
  5. Como você testaria um ViewModel que depende de um UseCase assíncrono sem acionar efeitos colaterais reais?

SOLID

  • S — Single Responsibility: uma classe tem apenas um motivo para mudar.
  • O — Open/Closed: aberto para extensão, fechado para modificação.
  • L — Liskov Substitution: subtipos devem poder substituir seus tipos base sem quebrar o programa.
  • I — Interface Segregation: clientes não devem depender de métodos que não usam — interfaces menores e coesas.
  • D — Dependency Inversion: dependa de abstrações, não de implementações concretas.
  • Objetivo: código manutenível, extensível e testável.
🔧 S: Um canivete suíço é útil, mas um chef prefere facas especializadas. O: Adicione uma nova lâmina sem destruir as outras. L: Qualquer pássaro que herda de "Pássaro" deve voar — não coloque Avestruz como pássaro voador se ela não voa. I: Não obrigue o Avestruz a implementar voar(). D: A tomada não depende da marca do carregador — ela usa um padrão (interface).

  2. SOLID é um conjunto de princípios, não regras — quando aplicar todos aumenta a complexidade desnecessariamente? Como você decide quando parar?

Design Patterns

Criacionais
  • Singleton: uma única instância global controlada.
  • Factory Method: subclasse decide qual objeto criar.
  • Abstract Factory: famílias de objetos sem especificar classes concretas.
  • Builder: constrói objetos complexos passo a passo.
  • Prototype: clona objetos existentes.
Estruturais
  • Adapter: converte interface incompatível.
  • Decorator: adiciona comportamento sem alterar a classe.
  • Facade: simplifica subsistema complexo com interface unificada.
  • Proxy: controla acesso a um objeto (lazy, segurança, cache).
  • Composite: trata objetos e composições uniformemente (árvores).
Comportamentais
  • Observer: notifica dependentes de mudanças de estado.
  • Strategy: encapsula algoritmos intercambiáveis.
  • Command: encapsula ações como objetos (undo, queue).
  • Chain of Responsibility: passa request por cadeia de handlers.
  • Template Method: define esqueleto de algoritmo, subclasses preenchem.
  • State: altera comportamento quando estado interno muda.
  • Iterator: percorre coleção sem expor estrutura interna.
🏗️ Criacionais: como você pede uma pizza — pelo cardápio (Factory), personalizada item a item (Builder), ou copia uma já pronta (Prototype).

Estruturais: um adaptador de tomada (Adapter), uma embalagem de presente que adiciona algo ao produto (Decorator), um gerente que faz ligações pra você (Facade).

Comportamentais: um feed de notícias que avisa seguidores (Observer), uma mochila que adapta transporte dependendo do meio (Strategy), um controle remoto que grava comandos para executar depois (Command).
  1. Qual a diferença fundamental entre Decorator e Proxy? Dê um exemplo onde trocar um pelo outro causaria problemas.
  2. Singleton em ambientes multi-thread ou multi-processo (ex.: serverless) — quais problemas surgem e como você os mitiga?
  3. Abstract Factory vs. Factory Method: quando a escolha de um sobre o outro é arquiteturalmente significativa?
  4. Como Observer se relaciona com Event Sourcing e CQRS? Onde os padrões convergem e onde divergem?
  5. Cite um cenário em que usar um Design Pattern introduz mais complexidade do que resolve — e como você justificaria não usar o pattern nesse caso.

The Twelve-Factor App

  • I — Codebase: um repositório versionado; múltiplos deploys de um só código.
  • II — Dependencies: declare e isole dependências explicitamente (package.json, requirements.txt).
  • III — Config: toda configuração que varia por ambiente fica em variáveis de ambiente.
  • IV — Backing Services: trate banco, cache, fila como recursos anexados — intercambiáveis.
  • V — Build/Release/Run: separação rígida entre build (artefato), release (artefato + config) e run.
  • VI — Processes: execute a app como processos stateless e share-nothing.
  • VII — Port Binding: exporte serviços via bind de porta — a app é auto-suficiente.
  • VIII — Concurrency: escale horizontalmente via model de processos (process types).
  • IX — Disposability: startup rápido e shutdown gracioso — resista a falhas inesperadas.
  • X — Dev/Prod Parity: mantenha dev, staging e prod o mais similares possível.
  • XI — Logs: trate logs como streams de eventos — escreva para stdout, não para arquivos.
  • XII — Admin Processes: rode tarefas administrativas como processos pontuais no mesmo ambiente.
🧱 Pensa num bloco de LEGO: cada peça (fator) é padronizada para encaixar em qualquer construção. Se você guardar a memória do brinquedo dentro da peça, ela para de funcionar quando você a encaixar em outra coisa. Processos stateless são como peças de LEGO — combinam com qualquer ambiente porque não carregam estado próprio.
  1. O fator VI (Processos stateless) é frequentemente violado com sessões em memória. Quais são as alternativas e seus trade-offs de latência e complexidade?
  2. Como o fator III (Config) se relaciona com gerenciamento de segredos? Variáveis de ambiente são suficientes ou há riscos? (ex.: leakage via /proc, environment dumps)
  3. O fator X (Dev/Prod Parity) — como você lida com serviços gerenciados em produção (ex.: RDS) que não têm equivalente local exato?
  4. O fator VIII (Concurrency) assume horizontal scaling — quais workloads resistem a esse modelo e como você os aborda em uma arquitetura 12-factor?
  5. Há tensão entre o fator XI (Logs como streams) e a necessidade de correlação de tracing distribuído. Como você reconcilia os dois?

OWASP Top 10

  • A01 — Broken Access Control: usuário acessa recursos além de sua permissão (IDOR, escalada de privilégio).
  • A02 — Cryptographic Failures: dados sensíveis expostos por criptografia fraca ou ausente (MD5, TLS 1.0, dados em texto claro).
  • A03 — Injection: SQL, NoSQL, OS, LDAP injection — input não sanitizado executado como código.
  • A04 — Insecure Design: falhas arquiteturais — ausência de threat modeling, controles de segurança por design.
  • A05 — Security Misconfiguration: permissões abertas, headers ausentes, serviços desnecessários ativos, senhas padrão.
  • A06 — Vulnerable Components: dependências com CVEs conhecidas não atualizadas.
  • A07 — Auth Failures: senhas fracas, brute force, credential stuffing, sessões não invalidadas.
  • A08 — Software/Data Integrity Failures: CI/CD sem verificação, deserialização insegura, sem assinatura de artefatos.
  • A09 — Logging/Monitoring Failures: incidentes não detectados por falta de logs ou alertas adequados.
  • A10 — SSRF: aplicação faz requests para URLs controladas pelo atacante — acessa rede interna, metadados de cloud.

  2. SSRF (A10) em ambientes cloud — como o atacante explora o metadata endpoint do EC2/GCP e quais controles de rede e aplicação você implementaria em camadas?
  3. A08 cobre deserialização insegura. Explique o vetor de ataque em Java com gadget chains e como mitigações como allowlists de classes funcionam.
  4. Como você implementaria um programa de gestão de vulnerabilidades em dependências (A06) num pipeline CI/CD sem bloquear entregas legítimas por falsos positivos?
  5. "Insecure Design" (A04) não tem correção técnica direta. Como você incorpora threat modeling no processo de desenvolvimento ágil sem torná-lo um gargalo?

4 Golden Signals

  • Latency (Latência): tempo para atender uma request. Meça p50, p95, p99 — a média esconde tail latencies.
  • Traffic (Tráfego): demanda sobre o sistema — RPS, conexões ativas, eventos/s. Define a escala de todos os outros sinais.
  • Errors (Erros): taxa de requests que falham — explícitas (5xx) e implícitas (200 com body errado).
  • Saturation (Saturação): quão "cheio" está o sistema — CPU, memória, fila, I/O. Prediz degradação futura.
  • Origem: SRE Book do Google — base de SLI/SLO/SLA e alertas acionáveis.
  • Complemento: RED Method (Rate, Errors, Duration) para serviços; USE Method (Utilization, Saturation, Errors) para recursos.
🏥 São os sinais vitais de um paciente: latência é o tempo de resposta (frequência cardíaca), tráfego é quantas consultas estão chegando (fluxo de pacientes), erros são diagnósticos incorretos ou falhas no tratamento, e saturação é o hospital cheio — quando a sala de espera transborda, tudo piora antes mesmo de alguém reclamar.
  1. Por que alertar em médias de latência é perigoso? Como você configuraria alertas baseados em percentis para evitar "alertas que não acordam ninguém de noite"?
  2. Explique a relação entre os 4 Golden Signals e a definição de SLOs. Como você traduziria "99,9% das requests em menos de 200ms" em alertas acionáveis sem gerar ruído?
  3. Saturation pode ser difícil de medir diretamente. Quais métricas proxy você usaria para detectar saturação antecipada em uma aplicação Node.js single-threaded?
  4. Como os 4 Golden Signals se comportam em arquiteturas serverless (Lambda/Cloud Functions) onde infraestrutura é efêmera e você não controla a saturação diretamente?
  5. Descreva como você construiria um runbook de on-call baseado nos 4 Golden Signals que permita a um engenheiro junior diagnosticar e agir em incidentes de produção.

Fluxos Git

Git Flow

  • Branches permanentes: main (produção) e develop (integração).
  • Feature branches: saem de develop, voltam para develop.
  • Release branches: stabilização antes de ir para main.
  • Hotfix branches: saem de main e voltam para main e develop.
  • Indicado para: projetos com ciclos de release definidos, multiple versões em produção.
  • Problema: overhead de merge, long-lived branches que geram conflitos.
🏭 Uma fábrica com esteira de produção: as peças são feitas em bancadas separadas (feature branches), passam por controle de qualidade (develop), depois por inspeção final (release branch) e só então chegam ao cliente (main). Um defeito no produto entregue vai para um reparo expresso (hotfix) sem parar a linha.
  1. Git Flow foi projetado para releases versionadas. Quais são os problemas concretos ao tentar usá-lo para entrega contínua (CD)?
  2. Como você lida com cherry-picks entre main e develop após um hotfix? Quais riscos de divergência surgem?
  3. Em que cenários o overhead do Git Flow compensa em relação ao Trunk-Based Development?

Trunk-Based Development

  • Um branch principal (trunk/main) — todos commitam diretamente ou via branches curtíssimas (<1 dia).
  • Feature Flags: código incompleto é commitado protegido por flags — desacoplam deploy de release.
  • CI obrigatório: build + testes verdes antes de qualquer merge.
  • Habilita CD: qualquer commit no trunk pode ir para produção.
  • Indicado para: times com alta maturidade de testes, pipelines rápidos, entrega contínua.
  • Risco: requer disciplina e feature flags bem gerenciadas.
🚄 Um trem bala com uma única trilha principal. Todo mundo embarca na mesma trilha, e o trem parte frequentemente. Se uma peça não está pronta, ela viaja coberta (feature flag) — visível mas inativa. Não há estação secundária esperando — ou embarca agora ou perde o trem.
  1. Como você evita que feature flags virem dívida técnica permanente? Qual é o lifecycle adequado de uma flag?
  2. TBD exige que testes rodem rápido. Qual estratégia você usaria para manter um suite de testes sob 10 minutos em um monorepo grande?
  3. Como você lida com breaking changes de API em TBD quando múltiplos serviços precisam migrar em conjunto (expand-contract pattern)?

Pirâmide de Testes

  • Base — Unit Tests: testam unidades isoladas (funções, classes). Rápidos, baratos, muitos. Usam mocks/stubs.
  • Meio — Integration Tests: testam a colaboração entre componentes — banco real, serviços internos. Mais lentos, menos quantidade.
  • Topo — E2E / UI Tests: testam o fluxo completo pelo browser/app. Lentos, frágeis, poucos — somente happy paths críticos.
  • Antipadrão "Ice Cream Cone": pirâmide invertida — muitos E2E, poucos unitários.
  • Testing Trophy (Kent C. Dodds): eleva integration tests como base — mais valor por esforço em aplicações modernas.
  • Contract Tests: entre serviços — garantem que produtor e consumidor mantêm o contrato (Pact).
🏗️ Uma pirâmide de construção: a base larga (testes unitários) sustenta tudo e é construída rapidamente com tijolos pequenos. O meio (integração) são as paredes — menos peças mas mais estruturais. O topo (E2E) é a ponta — existe para mostrar que a pirâmide ficou de pé, mas não é onde você constrói a estabilidade.
  1. Quando testes unitários com muitos mocks dão falsa confiança? Como você decide o que deve ser mockado vs. testado com dependência real?
  2. Explique o conceito de "Testing Trophy" de Kent C. Dodds e em quais tipos de projeto ele faz mais sentido do que a pirâmide clássica.
  3. Como você estruturaria uma estratégia de testes para uma arquitetura de microserviços, incluindo contract tests (Pact), component tests e testes de caos?
  4. Testes E2E são frágeis por natureza. Quais técnicas você usa para torná-los determinísticos (sem flakiness) em CI?
  5. Como você mensura "test coverage" de forma útil — por que cobertura de linha não é suficiente e o que você mediria em vez disso?

Autenticação & Autorização

JWT JSON Web Token

  • Estrutura: Header.Payload.Signature — Base64URL encoded, ponto-separados.
  • Header: algoritmo (alg) e tipo (typ: JWT).
  • Payload: claims — sub, exp, iat, iss, aud e claims customizadas.
  • Signature: HMAC-SHA256 (chave simétrica) ou RSA/ECDSA (assimétrica).
  • Stateless: o servidor não precisa de banco para validar — verifica apenas a assinatura.
  • Limitação: não há revogação nativa — tokens válidos vivem até expirar (use refresh + denylist ou rotação curta).
  • Vulnerabilidade clássica: alg: none — sempre valide o algoritmo explicitamente.
🪪 Um RG assinado pelo governo: qualquer porteiro pode verificar que o documento é autêntico pela assinatura, sem precisar ligar para o cartório. Se o RG expirou, não vale mais — mas revogar antes de vencer exige uma lista negra.
  1. Explique o ataque de "algorithm confusion" (RS256 → HS256) em implementações JWT vulneráveis e como a biblioteca deve se defender.
  2. Como você implementa revogação de JWT sem estado server-side? Quais são os trade-offs entre denylist, rotação de chaves e tokens de curta duração?
  3. JWT no localStorage vs. HttpOnly cookie — analise os vetores de ataque XSS e CSRF para cada abordagem.
  4. O que são "nested JWTs" e quando você os usaria vs. uma abordagem mais simples?
  5. Como você projeta um sistema de refresh tokens seguro com rotação automática e detecção de reutilização (refresh token rotation + reuse detection)?

JWE JSON Web Encryption

  • Diferença do JWT: JWT assina (garante integridade/autenticidade) mas não criptografa — payload é visível. JWE criptografa o payload.
  • Estrutura: 5 partes Base64URL: Header · Encrypted Key · IV · Ciphertext · Authentication Tag.
  • Algoritmos: RSA-OAEP ou ECDH-ES para key wrapping; AES-GCM ou AES-CBC+HMAC para conteúdo.
  • Uso típico: tokens com dados sensíveis que passam por terceiros, tokens opaque para clientes.
  • Nested JWT: JWS dentro de JWE — primeiro assina, depois criptografa (sign-then-encrypt).
🔐 JWT é uma carta com assinatura reconhecida em cartório — qualquer um pode ler o conteúdo, mas sabe que é autêntica. JWE é essa mesma carta dentro de um envelope lacrado com criptografia — só quem tem a chave consegue abrir e ler o conteúdo.
  1. Por que é importante assinar antes de criptografar (sign-then-encrypt) em vez de criptografar antes de assinar?
  2. Qual vulnerabilidade existe em algoritmos de JWE com padding (ex.: RSA-PKCS1v1.5) e por que RSA-OAEP é preferido?
  3. Em que cenários práticos JWE supera o uso de TLS para confidencialidade do payload?

Session-Based Auth

  • Fluxo: login → servidor cria sessão no store (Redis/DB) → devolve Session ID via cookie.
  • Stateful: servidor mantém estado — fácil revogar (delete session), mas escala horizontal requer session store compartilhado.
  • Cookie flags: HttpOnly (sem JS), Secure (só HTTPS), SameSite (CSRF), Path, Domain.
  • Session fixation: regenerar Session ID após login para evitar que atacante pré-fixe ID.
  • Session hijacking: roubo via XSS, MITM — mitigado por HttpOnly + Secure + HTTPS.
🎟️ É como um vestiário de clube: você entrega seus pertences (credenciais), recebe um número de armário (Session ID) num pulseira (cookie), e toda vez que apresenta a pulseira na porta, o atendente verifica no caderno (session store) o que está liberado pra você. Se a pulseira for perdida, o clube pode cancelar aquele número imediatamente.
  1. Como você implementa sticky sessions vs. session store centralizado em um cluster? Quais são os pontos de falha de cada abordagem?
  2. O que é "session fixation"? Como a regeneração de ID após autenticação bem-sucedida mitiga esse ataque?
  3. Compare session-based auth e JWT em termos de: revogação, latência, escalabilidade horizontal e segurança em dispositivos móveis.

OAuth 2.0 + PKCE + OpenID Connect

  • Authorization Framework (não autenticação): delega acesso sem compartilhar credenciais.
  • Grants (fluxos): Authorization Code (web apps), Implicit (depreciado), Client Credentials (M2M), Device Code (TV/CLI).
  • PKCE (Proof Key for Code Exchange): obrigatório para SPAs e mobile — evita code interception com code_verifier/challenge.
  • Tokens: Access Token (curto prazo, acesso a recurso), Refresh Token (longo prazo, obtém novo AT), ID Token (OIDC — identidade).
  • OpenID Connect (OIDC): camada de identidade sobre OAuth 2.0 — adiciona ID Token (JWT) com claims do usuário.
  • Scopes: granularidade de permissões concedidas ao client.
🏨 Você vai a um hotel (Resource Server). Em vez de dar a chave mestre do seu cofre de casa (senha), você pede ao cartório (Authorization Server) uma chave temporária para o quarto específico (access token com scope). O hotel aceita essa chave sem precisar saber sua senha real. Se a chave perder a validade, você volta ao cartório com seu refresh token para pegar outra.
  1. Por que o fluxo Implicit foi depreciado e como PKCE resolve o problema de confidencialidade em SPAs e aplicações mobile?
  2. Explique o ataque de "authorization code interception" e como code_verifier + code_challenge impedem sua exploração mesmo com acesso ao código.
  3. Qual a diferença entre OAuth 2.0 e OpenID Connect? O que o ID Token garante que o Access Token não garante — e por que você não deve usar o AT para autenticar usuários?
  4. Como você implementa logout seguro em OIDC — incluindo front-channel e back-channel logout — e quais são os desafios práticos?
  5. Em um cenário de microserviços, como você propaga identidade e autorização entre serviços downstream? Compare token introspection, JWT localmente verificado e mTLS.

Performance Web

Lazy Loading

  • Carrega recursos somente quando necessário — imagens abaixo do fold, rotas não visitadas.
  • Imagens: atributo loading="lazy" (nativo) ou Intersection Observer API.
  • Rotas: dynamic import() — React.lazy(), code splitting por rota.
  • Melhora TTI e reduz bytes iniciais; requer Suspense/fallback para UX adequado.
📦 Pedir comida num buffet só quando chegar perto do prato — você não carrega o prato inteiro de volta pra mesa antes de ver o que tem.
  1. Lazy loading de imagens com loading="lazy" pode causar Layout Shift (CLS). Como você previne isso sem carregar a imagem antecipadamente?
  2. Em quais cenários lazy loading de rotas piora a experiência em vez de melhorar? Como você decide o threshold de split?

Code Splitting

  • Divide o bundle em chunks menores carregados sob demanda.
  • Estratégias: por rota, por componente, por vendor (vendor chunk), por feature.
  • Tree Shaking: remove código não utilizado (dead code) na build.
  • Controle de cache: vendor chunk estável → cache longo; app chunk muda com cada deploy.
📚 Em vez de entregar a enciclopédia inteira, você entrega o volume do assunto que o leitor está procurando agora — os outros ficam na prateleira até serem pedidos.
  1. Como você evita "chunk waterfall" — múltiplos chunks carregados sequencialmente — em uma SPA com code splitting por rota?
  2. Explique como prefetching e preloading de chunks diferem e quando cada estratégia de hint (<link rel="prefetch"> vs. preload) deve ser usada.

Gzip / Brotli

  • Compressão de transferência HTTP — reduz bytes na rede, não no disco.
  • Browser anuncia suporte via Accept-Encoding: gzip, br; servidor responde com Content-Encoding: gzip.
  • Brotli (br): 20-30% melhor que gzip para texto; suporte moderno universal.
  • Compressão dinâmica (por request) vs. estática (pré-comprimida no CI — melhor para assets grandes).
  • Não comprima imagens JPEG/PNG/WebP — já são binários comprimidos; overhead negativo.
🗜️ Como empacotar roupas com saco a vácuo na mala: o volume reduz na viagem, mas você descompacta igual no destino. A mala (browser) sabe descompactar porque o etiqueteiro (Content-Encoding header) avisou o que estava dentro.
  1. Por que você não deve habilitar compressão para payloads muito pequenos (<1KB)? Qual é o threshold prático?
  2. Explique o ataque CRIME/BREACH contra HTTPS+gzip. Como a mitigação funciona e por que ele é relevante para tokens em cookies?

Core Web Vitals

  • LCP (Largest Contentful Paint): <2.5s — velocidade de carregamento do maior elemento visível.
  • INP (Interaction to Next Paint): <200ms — responsividade a interações do usuário (substituiu FID em 2024).
  • CLS (Cumulative Layout Shift): <0.1 — estabilidade visual — elementos que pulam durante carregamento.
  • Fator de rankeamento Google Search desde 2021.
  • Ferramentas: Lighthouse, Chrome DevTools, CrUX (dados reais), PageSpeed Insights.
⏱️ LCP: quanto tempo até o garçom trazer o prato principal. INP: quanto tempo o garçom demora para responder quando você acena. CLS: a mesa que sacode e derruba o copo quando alguém passa — quanto mais estável, melhor.
  1. Quais são as principais causas de LCP alto e como você as diagnostica e corrige sistematicamente — incluindo render-blocking resources, TTFB e resource prioritization?
  2. INP substituiu FID em 2024 — qual a diferença fundamental de medição e por que INP é mais representativo da experiência do usuário?
  3. Como você previne CLS causado por fontes web (FOUT/FOIT) e por anúncios/embeds de terceiros sem prejudicar o design?
  4. Como você monitora Core Web Vitals em produção com dados reais (field data) e como eles diferem dos dados de lab (Lighthouse)?

Call Stack

  • Estrutura LIFO (Last In, First Out) que rastreia onde a execução está no programa.
  • Cada chamada de função empilha um "stack frame" — contém: parâmetros, variáveis locais, endereço de retorno.
  • Stack Overflow: recursão infinita sem base case esgota o stack.
  • JavaScript Event Loop: Call Stack + Web APIs + Callback Queue + Microtask Queue.
  • Microtasks (Promises, queueMicrotask): processadas antes da próxima iteração do event loop — esvaziam a microtask queue completamente.
  • Macrotasks (setTimeout, setInterval, I/O): processadas uma por vez a cada "tick".
  • Blocking the stack: operação síncrona longa congela a UI — use Web Workers para CPU-bound tasks.
📚 Uma pilha de livros: você lê o topo, e quando abre um capítulo que referencia outro livro, coloca esse novo livro por cima. Só quando termina de ler ele, volta ao de baixo. Se você empilhar livros infinitamente, a pilha cai (stack overflow). O event loop é um bibliotecário que, quando a pilha está vazia, busca o próximo livro na fila de espera.
  1. Explique a ordem de execução: setTimeout(fn, 0), Promise.resolve().then(fn), queueMicrotask(fn) e código síncrono — por quê?
  2. O que é "stack frame" e quais informações ele contém? Como o motor JS usa isso para gerar stack traces?
  3. Por que async/await não bloqueia o event loop mesmo fazendo código async parecer síncrono? Explique o mecanismo por baixo.
  4. O que acontece quando a microtask queue nunca esvazia (ex.: Promise que resolve outra Promise recursivamente)? Como isso difere de um stack overflow?
  5. Em Node.js, como o event loop difere do browser? Explique as fases: timers, pending callbacks, poll, check (setImmediate), close callbacks.

Como Funcionam os Navegadores

1. Navegação & Rede
  • URL → DNS lookup → TCP Handshake → TLS Handshake → HTTP Request/Response.
  • HTTP/2 multiplexing, HTTP/3 QUIC — múltiplos streams sem head-of-line blocking.
  • Cache: memory cache, disk cache, Service Worker cache, Push cache.
2. Parsing & DOM
  • HTML Parser: tokenização → tree construction → DOM tree (incremental, tolerante a erros).
  • CSS Parser: CSSOM tree — regras de cascata, especificidade, herança.
  • <script> bloqueia o parser (exceto defer/async).
3. Render Pipeline
  • Style: DOM + CSSOM → Computed Styles.
  • Layout (Reflow): calcula posições e tamanhos de cada elemento — custoso.
  • Paint: preenche pixels por layer — cores, bordas, sombras.
  • Composite: GPU combina camadas — transform e opacity ficam só nessa etapa (jank-free).
  • Layers: will-change, transform: translateZ(0) promovem elemento a layer separada.
4. JavaScript Engine (V8)
  • Parsing → AST → Ignition (bytecode interpreter) → TurboFan (JIT compiler para código quente).
  • Garbage Collection: Scavenge (young gen) + Mark-Sweep/Compact (old gen).
  • Hidden Classes e Inline Caches para otimização de propriedades de objetos.
5. Processo & Segurança
  • Multi-processo: Browser Process, Renderer Process (por tab/site), GPU Process, Network Process.
  • Sandbox: Renderer Process não acessa SO diretamente — IPC com Browser Process.
  • Site Isolation: cross-site iframes em processos separados — mitigação Spectre/Meltdown.
🏙️ O browser é uma cidade: o DNS é o GPS (converte nome em endereço), o TCP/TLS é a estrada segura, o HTML Parser é o arquiteto que lê o projeto e monta o esqueleto (DOM), o CSS é o decorador de interiores (CSSOM), o Layout é a medição de cada cômodo, o Paint é pintar as paredes, e o Composite é tirar a foto final que você vê na tela. O V8 é a fábrica que converte ordens escritas (JS) em ação imediata.

  3. Como o V8 usa "Hidden Classes" para otimizar acesso a propriedades de objetos? Como a criação de propriedades fora de ordem deoptimiza esse mecanismo?
  4. O que é "Spectre" e como Site Isolation e SharedArrayBuffer restrictions no browser mitigam o ataque de timing side-channel?
