Autenticação & Autorização

JWT, Session Token & OAuth

Mecanismos de autenticação usados em aplicações modernas — quando usar cada um e como funcionam.

🔑 JWT — JSON Web Token

Token stateless composto por 3 partes separadas por ponto: header.payload.signature.

  • Header: algoritmo (HS256, RS256) e tipo
  • Payload: claims — dados do usuário (sub, email, roles, exp)
  • Signature: garante que o token não foi alterado

Vantagens: sem necessidade de consultar banco a cada request, escalável, ideal para microsserviços.
Desvantagens: não dá para invalidar antes de expirar (necessita blacklist ou refresh token curto).

🍪 Session Token (Cookie)

O servidor gera um ID de sessão, guarda no banco/redis, e envia ao browser via cookie. A cada request o servidor consulta a sessão.

  • Stateful: sessão vive no servidor
  • Revogação fácil: delete a sessão → usuário deslogado imediatamente
  • Problemas: não escala bem horizontalmente sem store compartilhada (Redis)

🔐 OAuth 2.0 & OpenID Connect

OAuth 2.0 é um protocolo de autorização (delegar acesso a recursos). OIDC adiciona camada de autenticação em cima do OAuth.

  • Authorization Code Flow: mais seguro, usado em SPAs com PKCE
  • Client Credentials: machine-to-machine (sem usuário)
  • Access Token + Refresh Token: access curto (15min), refresh longo (7d)

⚖️ Quando usar cada um

JWTMicrosserviços / APIs stateless
SessionMonólito web tradicional, SSR
OAuth/OIDCLogin social, SSO, B2B

Decodificando um JWT

▼

Um JWT tem a forma xxxxx.yyyyy.zzzzz. Cada parte é base64url-encoded.

// Header decodificado
{ "alg": "HS256", "typ": "JWT" }

// Payload decodificado
{
  "sub": "user_123",
  "email": "edu@example.com",
  "roles": ["frontend-dev"],
  "iat": 1713000000,
  "exp": 1713003600   // expira em 1h
}

// Verificando no Node.js
import jwt from 'jsonwebtoken'
const payload = jwt.verify(token, process.env.JWT_SECRET)
// lança JsonWebTokenError se inválido ou expirado

Refresh Token Flow

▼ 
// 1. Login — servidor emite par de tokens
POST /auth/login → {
  access_token:  "eyJ...", // expira em 15min
  refresh_token: "opq...", // expira em 7d (httpOnly cookie)
}

// 2. Quando access expirar (401), frontend chama:
POST /auth/refresh
Cookie: refresh_token=opq...
→ { access_token: "eyJ..." } // novo access

// 3. Logout — revoga o refresh_token no banco

Authorization Code + PKCE (SPA)

▼ 
// 1. Gera code_verifier aleatório e code_challenge = SHA256(verifier)
const verifier  = crypto.randomBytes(32).toString('base64url')
const challenge = crypto.createHash('sha256').update(verifier).digest('base64url')

// 2. Redireciona o usuário para o IdP
GET /authorize?
  client_id=...
  &redirect_uri=...
  &response_type=code
  &code_challenge=challenge
  &code_challenge_method=S256

// 3. IdP retorna com code; troca pelo token:
POST /token { code, code_verifier: verifier, ... }
→ { access_token, id_token, refresh_token }
Performance

Monitoramento, Logs, Métricas & Cache

Como medir, observar e otimizar o desempenho de aplicações frontend e backend.

📊 Core Web Vitals

  • LCP (Largest Contentful Paint): tempo para o maior elemento visível aparecer. Meta: < 2.5s
  • INP (Interaction to Next Paint): latência de interação. Meta: < 200ms
  • CLS (Cumulative Layout Shift): estabilidade visual. Meta: < 0.1
  • FID/FCP/TTFB: First Input Delay, First Contentful Paint, Time To First Byte

🪵 Logs & Observabilidade

Os 3 pilares: Logs (eventos), Métricas (números ao longo do tempo), Traces (rastreamento de uma request através de serviços).

  • Structured logging: JSON com campos consistentes (level, timestamp, requestId, userId)
  • Níveis: ERROR > WARN > INFO > DEBUG
  • Ferramentas: Datadog, New Relic, Grafana + Loki, ELK Stack

⚡ Cache

  • HTTP Cache: Cache-Control: max-age, stale-while-revalidate
  • CDN: edge cache — arquivos estáticos próximos ao usuário
  • Redis/Memcached: cache de dados no servidor (queries caras, sessões)
  • Memoização: cache em memória de resultados de funções puras
  • Cache invalidation: o problema mais difícil — estratégias: TTL, event-driven, versioning na URL

🔁 Callback & Async

Padrões de performance em JS: evitar blocking no event loop, usar Web Workers para CPU-heavy, lazy loading de módulos.

  • Debounce: adia execução enquanto evento continuar disparando (ex: input search)
  • Throttle: limita execuções por intervalo de tempo (ex: scroll handler)
  • requestAnimationFrame: animações sincronizadas com refresh do browser

Debounce vs Throttle

▼ 
// Debounce — só executa após N ms de silêncio
function debounce(fn, delay) {
  let timer
  return (...args) => {
    clearTimeout(timer)
    timer = setTimeout(() => fn(...args), delay)
  }
}
const onSearch = debounce((q) => fetchResults(q), 300)

// Throttle — executa no máximo 1x a cada N ms
function throttle(fn, limit) {
  let last = 0
  return (...args) => {
    const now = Date.now()
    if (now - last >= limit) { last = now; fn(...args) }
  }
}
window.addEventListener('scroll', throttle(handleScroll, 100))

Cache-Control Headers

▼ 
// Arquivo estático com hash no nome → cache longo
Cache-Control: public, max-age=31536000, immutable
// main.abc123.js nunca muda → pode cachear 1 ano

// HTML da aplicação → sem cache (ou curto)
Cache-Control: no-cache
// Browser revalida sempre, mas usa cache se 304

// API que pode ficar stale por 1min enquanto revalida
Cache-Control: public, max-age=60, stale-while-revalidate=300

Segurança

OWASP Top 10 & Boas Práticas

Principais vulnerabilidades web e como preveni-las no frontend e backend.

🛡️ OWASP Top 10 (2021)

  1. Broken Access Control — usuário acessa recurso que não deveria
  2. Cryptographic Failures — dados sensíveis sem criptografia, MD5/SHA1 obsoleto
  3. Injection — SQL, NoSQL, LDAP, OS command injection
  4. Insecure Design — falhas arquiteturais, sem threat modeling
  5. Security Misconfiguration — configs default, headers ausentes, CORS aberto
  6. Vulnerable & Outdated Components — deps com CVEs conhecidas
  7. Authentication Failures — senhas fracas, sem MFA, brute-force permitido
  8. Software & Data Integrity Failures — sem verificação de integridade (ex: supply chain)
  9. Security Logging & Monitoring Failures — sem alertas de ataques
  10. SSRF — servidor faz request para URL controlada pelo atacante

💉 XSS & Injeção

  • XSS Reflected: script injetado via URL refletido na página
  • XSS Stored: script salvo no banco, executa para todos os usuários
  • Prevenção: escapar output, CSP (Content Security Policy), DOMPurify para HTML dinâmico
  • SQL Injection: nunca concatenar input em query. Sempre parameterized queries

🔒 Headers de Segurança

  • Content-Security-Policy — controla quais recursos podem ser carregados
  • X-Frame-Options: DENY — previne clickjacking
  • Strict-Transport-Security — força HTTPS
  • X-Content-Type-Options: nosniff
  • Referrer-Policy: strict-origin-when-cross-origin

🌐 CORS

Cross-Origin Resource Sharing. Browser bloqueia requests cross-origin por padrão. O servidor deve responder com headers Access-Control-Allow-Origin corretos.

  • Nunca use Access-Control-Allow-Origin: * com credentials: true
  • Whitelist explícita de origens permitidas
  • Preflight (OPTIONS) precisa responder corretamente

XSS — vulnerável vs seguro

▼ 
// ❌ Vulnerável — XSS stored
element.innerHTML = userComment  // se comment = "<script>alert(1)</script>"

// ✅ Seguro — textContent não executa HTML
element.textContent = userComment

// ✅ Se precisar HTML dinâmico: sanitizar
import DOMPurify from 'dompurify'
element.innerHTML = DOMPurify.sanitize(userComment)

SQL Injection — vulnerável vs seguro

▼ 
// ❌ Vulnerável
const q = `SELECT * FROM users WHERE email = '${email}'`
// email = "' OR '1'='1" → retorna todos os usuários

// ✅ Parameterized query (Node + pg)
const { rows } = await db.query(
  'SELECT * FROM users WHERE email = $1',
  [email]
)

Arquitetura

Microsserviços, Clean Arch & 12 Factors

Padrões arquiteturais para sistemas escaláveis e maintíveis.

🧩 Microsserviços vs Monólito

  • Monólito: deploy único, simples de desenvolver inicialmente, difícil de escalar partes individualmente
  • Microsserviços: serviços independentes, deploy e escala individuais, complexidade de rede e distribuição
  • Quando migrar: quando o time crescer ou quando precisar escalar partes diferentes de forma independente
  • Comunicação: REST, gRPC, eventos (Kafka/RabbitMQ)

🧅 Clean Architecture / Onion

Camadas concêntricas onde as dependências sempre apontam para dentro (para o domínio).

  • Domain/Entities: regras de negócio puras, sem frameworks
  • Use Cases/Application: orquestra entidades, casos de uso da aplicação
  • Interface Adapters: controllers, presenters, gateways
  • Frameworks & Drivers: Express, React, banco de dados — detalhes de infra

Regra de ouro: código do domínio não importa nada do lado de fora.

📋 12 Factor App

  1. Codebase: um repo, múltiplos deploys
  2. Dependencies: declare explicitamente (package.json)
  3. Config: variáveis de ambiente, não no código
  4. Backing Services: trate DB, cache como recursos attachables
  5. Build/Release/Run: fases separadas e imutáveis
  6. Processes: stateless, sem estado em memória entre requests
  7. Port Binding: app exporta serviço via porta
  8. Concurrency: escala via processos
  9. Disposability: startup rápido, shutdown gracioso
  10. Dev/Prod Parity: ambientes o mais parecidos possível
  11. Logs: trate como event streams (stdout)
  12. Admin Processes: tarefas admin como one-off processes

Clean Architecture — estrutura de pastas

▼ 
src/
├── domain/
│   ├── entities/          # User, Order (sem deps externas)
│   └── repositories/      # interfaces (contratos)
├── application/
│   └── use-cases/         # CreateUser, ProcessOrder
├── infrastructure/
│   ├── db/                # implementação do repositório (Prisma)
│   └── http/              # Express controllers
└── main/                  # composição / DI

SRE

4 Golden Signals

Os 4 indicadores fundamentais para monitorar a saúde de qualquer serviço (Google SRE Book).

⏱️ 1. Latência

Tempo para processar uma request. Meça latência de sucesso separado do erro — um erro 500 rápido não é "bom desempenho".

  • Monitore percentis: p50, p95, p99 (não só média)
  • Alertas úteis: p99 > 1s por mais de 5 minutos

📈 2. Tráfego (Traffic)

Demanda sobre o sistema. Para web: requests por segundo (RPS). Para streaming: bytes por segundo.

  • Referência para dimensionar capacidade
  • Picos inesperados de tráfego indicam possível ataque ou viralização

❌ 3. Erros

Taxa de requests que falham. Inclua erros explícitos (5xx) e erros implícitos (200 com conteúdo errado).

  • Error rate = erros / total de requests
  • Defina SLO: ex: < 0.1% de erros

🔋 4. Saturação

O quão "cheio" está o serviço. CPU, memória, disco, conexões de DB. Prevê degradação antes de acontecer.

  • Alta saturação → alta latência em breve
  • Alertas: CPU > 80% por 10min, connection pool > 90%

📌 USE Method (complementar)

Para recursos de infraestrutura: Utilization, Saturation, Errors. Enquanto os 4 signals focam no serviço, USE foca nos recursos de hardware.

Versionamento

Git Flow & Boas Práticas

Estratégias de branching, commits semânticos e colaboração com Git.

🌊 Git Flow

Modelo de branching com branches de vida longa e branches de feature de curta duração.

  • main/master: produção, sempre estável
  • develop: integração de features
  • feature/xxx: branch por feature, sai de develop
  • release/x.y.z: prep de release (bump de versão, bugfixes)
  • hotfix/xxx: correção urgente em produção, sai de main

🔀 Trunk-Based Development (alternativa)

Feature branches muito curtas (1-2 dias), merge frequente para main. Exige feature flags para código incompleto. Preferido por times com CI/CD maduro.

📝 Commits Semânticos (Conventional Commits)

  • feat: nova funcionalidade
  • fix: correção de bug
  • docs: apenas documentação
  • style: formatação (sem mudança de lógica)
  • refactor: refatoração (sem fix nem feat)
  • test: adição de testes
  • chore: build, deps, configs
  • perf: melhoria de performance
  • ci: mudanças de CI

Com ! indica breaking change: feat!: remove deprecated API

✅ Boas Práticas

  • Commits atômicos — uma mudança lógica por commit
  • Pull Requests pequenos — mais fácil de revisar
  • Code review obrigatório antes de merge
  • .gitignore — nunca commitar .env, node_modules, build
  • Protected branches + required checks no main
  • Rebase local antes de abrir PR (histórico limpo)

Qualidade

Paradigmas & Práticas de Teste

Pirâmide de testes, TDD, tipos e estratégias para garantir qualidade de software.

🔺 Pirâmide de Testes

  • Unit (base): rápidos, isolados, muitos. Testam uma função/classe.
  • Integration (meio): testam a colaboração entre módulos (ex: service + DB real)
  • E2E (topo): poucos, lentos, testam o sistema completo via browser (Playwright, Cypress)

Evite a "pirâmide invertida" — muitos E2E e poucos unit são frágeis e lentos.

🔴🟢🔵 TDD — Test Driven Development

  • Red: escreve o teste que falha
  • Green: escreve o código mínimo para passar
  • Refactor: melhora o código mantendo os testes verdes

Benefícios: design emergente, confiança no refactor, documentação viva.

🎭 Mocks, Stubs e Spies

  • Stub: substitui uma dependência com resposta pré-definida (sem verificação)
  • Mock: stub com verificação — você checa se foi chamado, como e quantas vezes
  • Spy: observa chamadas sem substituir a implementação real
  • Fake: implementação alternativa simplificada (ex: InMemoryRepository)

📏 Boas Práticas

  • Arrange, Act, Assert (AAA) — estrutura clara
  • Um assert lógico por teste
  • Nomes descritivos: should return 404 when user not found
  • Evite testes que dependem de ordem de execução
  • Coverage como guia, não como objetivo — 100% com maus testes é pior que 70% com bons

Unit test com Vitest/Jest

▼ 
// src/utils/formatCurrency.ts
export function formatCurrency(value: number, currency = 'BRL') {
  return new Intl.NumberFormat('pt-BR', {
    style: 'currency', currency
  }).format(value)
}

// src/utils/formatCurrency.test.ts
import { describe, it, expect } from 'vitest'
import { formatCurrency } from './formatCurrency'

describe('formatCurrency', () => {
  it('formats BRL correctly', () => {
    expect(formatCurrency(1234.5)).toBe('R$\u00a01.234,50')
  })
  it('handles zero', () => {
    expect(formatCurrency(0)).toBe('R$\u00a00,00')
  })
  it('supports other currencies', () => {
    expect(formatCurrency(10, 'USD')).toContain('10')
  })
})

DevOps

CI/CD — Pipeline & Workflow

Entrega contínua: automatizar build, test, deploy e garantir qualidade a cada mudança.

🔄 CI — Continuous Integration

Integrar código frequentemente e validar automaticamente. A cada push/PR:

  • Lint + type check
  • Testes unitários e de integração
  • Build da aplicação
  • Análise de segurança (SAST, dependency audit)
  • Code coverage report

🚀 CD — Continuous Delivery/Deployment

  • Delivery: código sempre pronto para deploy (aprovação manual para produção)
  • Deployment: deploy automático para produção a cada merge
  • Estratégias: Blue/Green, Canary, Feature Flags
  • Rollback: sempre ter plano de reversão — imagens Docker imutáveis ajudam

🔵🟢 Blue/Green & Canary

  • Blue/Green: dois ambientes idênticos, alterna o tráfego 100% para o novo (rollback instantâneo)
  • Canary: nova versão recebe % crescente de tráfego (ex: 1% → 10% → 50% → 100%)
  • Feature Flags: desacopla deploy de release — feature vai para prod mas fica desligada
